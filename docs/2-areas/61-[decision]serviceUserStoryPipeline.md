# Service User Story Pipeline — 2026-03-27

## 배경

인프라/라이브러리는 모듈=PRD 단위로 충분하지만, 서비스(CMS 등)는 사용자 체감 단위로 구현을 도출해야 한다. discuss→PRD 사이에 "어떤 구현 단위가 필요한가"를 도출하는 분해 레이어가 필요.

## 내용

### 4단 레이어 (안정성 순서)

1. **욕구** (가장 안정) — "사용자는 콘텐츠 편집을 하기 원한다". 서비스 전체 지도 역할
2. **스토리** (릴리즈 단위) — "사용자는 원하는 콘텐츠를 선택하고 내용을 편집할 수 있다". 크기 기준: "이것만 릴리즈 가능한가?"
3. **기능** (준안정) — "선택, 엔터로 편집모드 전환". 스토리를 가능하게 하는 수단
4. **화면/위젯** (가변) — 캔버스, 사이드바, 툴바. 기능을 배치하는 그릇. PRD 연결 지점

### 3가지 목적

1. **전체 조망** — 욕구 리스트 = 서비스의 전체 지도
2. **진척도/릴리즈 판단** — 스토리 상태(⬜/🟢) = 사용자 체감 단위의 완료 여부
3. **PRD 파편화 방지** — 스토리→기능→화면 참조로 PRD 출처 추적

### 원칙

- **사용자 체감 언어** — 구현 언어(axis, pattern, plugin) 배제, 사용자가 느끼는 것을 적는다
- **사업 기획 범위 밖** — Value/ROI/마켓핏은 다루지 않음. 구현을 위한 기획
- **기획↔구현 매핑은 PRD 담당** — 이 문서에서는 신경 안 씀

### 스킬 형태

- **인터뷰 방식** — /discuss처럼 갭 질문으로 빈칸을 채워감
- **누적 가능** — 여러 세션에 걸쳐 기존 맵을 로드하고 빈칸을 발견
- **산출물 = 파일** — 단일 파일, 4개 섹션(Needs/Stories/Features/Screens) 상호 참조
- **discuss 이후, PRD 이전**에 위치하는 파이프라인 단계

### 파이프라인 위치

```
discuss → [유저스토리 맵] → PRD → Plan → Go
```

### 스토리 크기 기준

"이것만 릴리즈 가능한가?" 판별 테스트:
- 너무 크면: "사용자가 콘텐츠를 관리할 수 있다" ← 이건 욕구
- 적정: "사용자가 콘텐츠를 선택하고 편집할 수 있다" ← 이것만 릴리즈 가능
- 너무 작으면: "사용자가 엔터를 눌러 편집모드에 진입한다" ← 이건 기능
