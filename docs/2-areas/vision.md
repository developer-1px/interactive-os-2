# Vision

> FE의 가치는 사라지지 않았다. 비용이 가치를 가렸을 뿐이다.

## Thesis

CLI + LLM이 GUI 조작을 대체하면서 프론트엔드는 위기에 처한 것처럼 보인다. 하지만 진짜 문제는 **GUI의 가치가 사라진 것이 아니라, 구축 비용이 가치를 가리고 있었다는 것**이다.

비용을 0에 가깝게 낮추면, nice-to-have는 기본값이 된다.

interactive-os는 그 비용을 제거하는 도구다.

## 왜 GUI인가 — 인지 인터페이스

```
CLI:  순차적(sequential) — 한 줄씩 읽는다
GUI:  동시적(simultaneous) — 구조를 한 번에 본다
```

LLM이 **조작(manipulation)**을 대행해도, **인지(cognition)**는 사람의 눈이 해야 한다.

- 트리 구조에서 부모-자식 관계를 **한눈에** 파악
- 칸반에서 상태 분포를 **시각적으로** 스캔
- 그리드에서 2차원 데이터의 패턴을 **즉시** 발견

텍스트 100줄보다 treegrid 한 화면이 구조를 빠르게 파악시킨다. 이건 인간의 시각 인지 구조가 바뀌지 않는 한 변하지 않는다.

> **"있으면 더 낫다. 그런데 비용이 높아서 없어도 됐다."**
>
> 비용이 사라지면? 없을 이유도 사라진다.

## 왜 CLI가 이겼나 — 가치가 아니라 비용

GUI의 조작 비용:

```
인터페이스 학습 → 위치 탐색 → 클릭/입력    (3단계)
```

CLI + LLM의 조작 비용:

```
의도 표현 → 실행                            (2단계)
```

중간의 **"어디를 눌러야 하지?"**가 통째로 사라졌다.

CLI가 이긴 이유는 "더 좋아서"가 아니다. **"더 싸서"**다. 조작 비용이 낮으니 ROI가 맞는 것이다. GUI는 구축 비용(키보드, 포커스, CRUD, 접근성)이 높아서 ROI가 안 맞았을 뿐이다.

**비용을 맞추면 게임이 달라진다.**

## interactive-os의 역할 — 비용 제거기

```tsx
// Before: treegrid를 만들려면 —
// 키보드 핸들링, 포커스 관리, 선택, 확장/축소, 접근성 속성,
// undo/redo, copy/paste, drag & drop...
// → 수백 줄, 수일의 작업

// After: interactive-os가 있으면 —
useAria({ behavior: treegrid, data, plugins, onChange })
// → 한 줄. 키보드, 포커스, CRUD, 접근성 모두 포함.
```

| 비용 항목 | Before | After |
|-----------|--------|-------|
| 키보드 인터랙션 | 매번 구현 | 축(Axis)으로 해결 |
| 포커스 관리 | 매번 구현 | 엔진이 처리 |
| CRUD + Undo/Redo | 매번 구현 | 플러그인으로 해결 |
| ARIA 접근성 | 매번 구현 | behavior가 바인딩 |
| DnD / Copy-Paste | 매번 구현 | 플러그인으로 해결 |
| **총 비용** | **수일** | **수분** |

GUI 생산 비용이 CLI 수준으로 떨어지면 — **"굳이 안 만들 이유"가 사라진다.**

## LLM도 레이어 위에서 일한다

LLM이 아무리 똑똑해져도 바닥부터 짜지 않는다. JavaScript 위에서, React 위에서, TypeScript 위에서 짠다. 그리고 그 레이어가 좋을수록 결과가 좋다.

```
JavaScript
  └─ React
       └─ interactive-os    ← 다음 레이어
            └─ LLM 생성 코드
```

interactive-os는 React 위의 **다음 결정적 레이어**가 된다.

그리고 LLM의 비용과 속도는 0이 아니다:

| 항목 | 블록 없이 | 블록 있으면 |
|------|-----------|-------------|
| 토큰 비용 | treegrid 500줄 생성 | `useAria("treegrid")` 1줄 |
| 검증 비용 | 매번 접근성/키보드 테스트 | 298개 테스트 이미 통과 |
| 일관성 | 매번 다르게 생성 | ARIA 표준 보장 |
| 시간 | 생성 + 디버그 | 즉시 동작 |

**블록이 있으면 LLM의 비용도, 사람의 비용도, 둘 다 줄어든다.**

## LLM DX 풀세트 — 채택 장벽 제거

라이브러리의 채택은 문서 품질에 달려 있다. LLM 시대에는 **LLM이 얼마나 잘 쓰는가**가 채택 기준이 된다.

| 산출물 | 대상 | 역할 |
|--------|------|------|
| `llms.txt` | LLM | 라이브러리 개요, API 요약, 컨텍스트 주입 |
| `skill` | Claude Code | 사용법을 세션에 주입, pit of success 유도 |
| `cli` | 개발자 + LLM | scaffolding, 코드 생성, 보일러플레이트 제거 |
| `test` | 모두 | 작동하는 코드 = 가장 정확한 문서 |

이 네 가지가 세트로 있으면 LLM이 interactive-os를 "학습"하는 게 아니라 **레일 위에 올라탄다.** 잘못 쓰기가 더 어려운 구조.

## 사정권 — 어디까지 커버하는가

```
┌─────────────────────────────────────────────┐
│              interactive-os 사정권             │
│                                             │
│  관리도구        조작도구        웹 콘텐츠      │
│  ─────────    ─────────    ─────────       │
│  CMS           에디터         뷰어           │
│  대시보드       설정 패널      문서           │
│  백오피스       폼 빌더       포트폴리오       │
│                                             │
│  = ARIA 패턴으로 매핑 가능한 구조적 UI         │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│              사정권 밖                        │
│                                             │
│  Figma 캔버스 · 3D 뷰어 · 지도 · 게임        │
│  = 자유 좌표계, ARIA 패턴에 매핑 불가          │
└─────────────────────────────────────────────┘
```

사정권은 좁지 않다. 웹의 대다수 서비스는 관리도구 + 조작도구 + 콘텐츠 뷰어로 구성된다.

## PoC — Visual CMS

Visual CMS는 thesis의 증명이다:

1. **CMS** = 가장 보편적인 관리도구. 모든 웹이 콘텐츠를 관리한다
2. **Visual** = "있으면 더 나은" GUI의 가치를 직접 증명한다
3. **nested array 편집** = treegrid, tabs, CRUD, dnd — interactive-os의 핵심 패턴이 전부 동원된다

> "interactive-os 덕분에 visual이 공짜"
>
> → "그러면 굳이 안 visual일 이유가 없다"

## 확산 전략 — 웹에서 증명, 플랫폼으로 확산

interactive-os의 핵심은 React 컴포넌트가 아니라 **모델**이다:

```
렌더러 독립 모델
├── Normalized Store     (엔티티 정규화)
├── Command Engine       (CRUD + undo/redo)
├── Plugin Architecture  (keyMap 소유)
└── Behavior Axes        (13개 범용 축)
```

이 모델은 렌더러에 종속되지 않는다. ARIA 패턴의 키보드 인터랙션은 웹이든 TUI든 동일하다 — Arrow로 이동, Enter로 진입, Escape로 복귀.

```
              웹에서 증명
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
     앱(RN)   Rust TUI    다른 FW
   Swift/Kt  ratatui    Vue/Svelte
```

웹은 가장 빠르게 증명할 수 있는 플랫폼이다. 모델이 증명되면 렌더러만 바꾸면 된다.

## 요약 — FE 개발자의 새 정체성

```
Before:  "GUI 조작 인터페이스 장인"
         버튼, 폼, 드래그 — LLM이 대행하면서 가치 하락

After:   "인지 인터페이스의 빌딩블록 설계자"
         보고 판단하는 구조를 설계하고,
         그 구축 비용을 0으로 만들어,
         GUI가 당연히 있는 세상을 만든다
```

레이어는 쌓이지, 대체되지 않는다. React가 사라지지 않듯, interactive-os도 사라지지 않는다.
