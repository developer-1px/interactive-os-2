# AI 협업 실전 팁과 인사이트

> 작성일: 2026-03-25
> 작성자: FE개발
> 대상: AI 코딩 도구를 사용하고 있거나 도입을 검토하는 개발자
> 성격: 3개월간 Claude Code 전면 사용에서 얻은 실전 교훈

---

## 1. 가장 비싼 실수: "잘못된 것을 빠르게 만드는 것"

AI 코딩 도구의 가장 큰 위험은 느린 것이 아니라 **너무 빠른 것**이다.

"이거 만들어줘"라고 하면 AI는 즉시 200줄을 작성한다. 문제는 그 200줄이 내가 원한 것과 **미묘하게 다른 방향**일 때 발생한다. 수정하려면 200줄을 이해하고 부분적으로 고쳐야 하는데, 처음부터 다시 만드는 게 빠른 경우가 대부분이다.

**해법: 코딩 전에 "왜"를 먼저 정렬한다.**

```
❌ "로그인 기능 만들어줘"
✅ "왜 로그인이 필요한지부터 정리하자. 현재 상태는 이렇고, 이상은 이런 건데..."
```

이 한 단계가 재작업률을 체감상 70% 이상 줄였다. AI가 빠르게 실행할 수 있다는 것은, **방향이 맞을 때만** 장점이다.

---

## 2. AI에게 "어떻게 할까요?"를 금지시켜라

AI의 기본 행동은 **판단을 회피하고 질문을 던지는 것**이다.

```
❌ AI: "A와 B 중 어느 것을 선호하시나요?"
✅ AI: "제 판단: A. 이유: [근거]. 다르게 보시면 말씀해주세요."
```

이 규칙 하나로 대화 턴 수가 절반으로 줄었다. AI가 판단을 먼저 밝히면, 내가 동의/반박만 하면 되므로 **인지 부하가 AI에게 이전**된다. AI를 "질문하는 비서"가 아니라 "판단하는 파트너"로 쓰는 것이 핵심이다.

실제로 적용한 프롬프트:
> "질문할 때 판단을 먼저 밝힌다. 모델이 판단을 회피하고 '어떻게 할까요?'만 던지면, 선택의 인지 부하를 고스란히 사용자에게 전가하게 된다."

---

## 3. "기억"을 시스템으로 관리한다

AI와 대화하면 훌륭한 결론에 도달하지만, 다음 세션에서는 **모든 것이 리셋**된다. 같은 교정을 반복하는 것은 가장 큰 시간 낭비다.

**해법: 피드백을 즉시 파일로 영속화한다.**

```markdown
# memory/feedback_no_mock.md
---
name: mock 호출 검증 금지
type: feedback
---
테스트에서 toHaveBeenCalled() 같은 mock 호출 검증을 사용하지 않는다.

**Why:** 테스트를 위한 테스트가 된다. 실제 DOM 상태로 검증해야 의미가 있다.
**How to apply:** 통합 테스트에서 user.keyboard() → DOM 결과로 검증.
```

이런 피드백 파일이 60+개 쌓여있다. 새 세션에서 AI가 이 파일들을 읽으면, 3개월간의 교정이 **한 번도 설명하지 않아도** 반영된다.

**팁: 성공한 접근도 기록한다.** 실패 교정만 기록하면 AI가 점점 소극적으로 변한다. "이 방식 좋았다"도 기록해야 균형이 잡힌다.

## 4. PRD가 애자일 스토리보다 낫다 (AI 협업에서)

인간 팀에서는 "사용자로서 ~하고 싶다" 한 줄이면 충분할 수 있다. 팀원들이 암묵적 맥락을 공유하기 때문이다.

AI에게는 이 암묵적 맥락이 없다. 그래서 **촘촘한 PRD**(Product Requirements Document)가 훨씬 효과적이다.

실제로 사용하는 PRD 구조:
```
1. 동기 — 왜 이걸 하는가
2. 산출물 — 완료되면 무엇이 존재하는가
3. 인터페이스 — 입력 X → 출력 Y (인과)
4. 경계 — 여기까지만 하고 여기는 안 한다 (인과)
5. 원칙 대조 — 기존 설계 원칙과 충돌하지 않는가
6. 부작용 — 이 변경이 만드는 새 문제는
7. 금지 — 절대 하지 않을 것
8. 검증 — 어떻게 확인할 것인가
```

**인사이트: 명세가 촘촘하면 구현이 최소로 수렴한다.** PRD가 정확하면 AI는 딱 필요한 만큼만 만든다. 이것은 버그가 아니라 정확성의 증거다.

---

## 5. "왜?"는 반대 의사가 아니다

AI에게 "왜 이렇게 했어?"라고 물으면, 대부분의 AI는 사과하고 다른 방향으로 바꾼다. 하지만 나는 진짜로 이유가 궁금해서 물은 것이다.

**실제 적용한 규칙:**
> "'왜'는 이유를 묻는 질문이다. 반대/철회 의사로 해석하지 않는다."

이 한 줄을 CLAUDE.md에 넣은 후, AI가 자신의 판단을 방어적으로 바꾸는 행동이 사라졌다. AI와의 대화 품질이 올라가려면, **AI가 자기 판단을 유지할 수 있는 환경**을 만들어야 한다.

---

## 6. 설계 의도를 이해하기 전에 수정하지 마라

테스트가 실패하면 AI는 본능적으로 "통과하도록 고치려 한다". 하지만 **테스트 통과 ≠ 올바른 수정**이다.

실제 사례: AI가 테스트를 통과시켰는데, 원래 설계의 불변 조건(invariant)을 깨뜨린 적이 있었다. 그 이후로 디버깅 프로토콜을 만들었다:

```
1. 재현 — 코드를 읽기 전에 테스트로 재현
2. 관측 도구 검증 — 디버깅 도구 자체의 버그를 먼저 배제
3. 설계 의도 파악 — 원래 이 코드가 왜 이렇게 되어있는지 이해
4. 수정 판단 — 설계에 맞는 수정인지 확인한 후 수정
```

**인사이트: 관측 도구도 틀릴 수 있다.** AI가 디버깅할 때, 관측 도구(테스트 환경, jsdom 등) 자체의 버그를 대상 버그로 오진하는 경우가 있었다. "도구를 먼저 검증"이 프로토콜에 들어간 이유다.

## 7. 자가 개선 루프: AI가 혼자 점수를 올린다

가장 놀라운 발견은 **점수 함수만 정의하면 AI가 혼자 개선할 수 있다**는 것이다.

```
1. 점수 측정: pnpm score:design → 49%
2. AI가 낮은 점수 항목 식별
3. 해당 항목 수정
4. 재측정 → 62%
5. 반복...
6. 최종: 96%
```

사람은 점수 함수를 정의하고 실행 명령만 내렸다. AI가 **측정→진단→수정→재측정**을 자율 반복하며 49%에서 96%까지 올렸다.

**핵심 조건: 점수 함수가 명확해야 한다.** "좋은 코드를 만들어줘"는 동작하지 않지만, "이 스크립트를 실행하면 숫자가 나오는데, 그 숫자를 올려줘"는 동작한다. **측정 가능한 것만 개선 가능하다.**

## 8. 문서는 부산물이 아니라 사고의 도구다

처음에는 문서를 "나중에 정리하는 것"으로 생각했다. 하지만 AI 협업에서 문서는 **사고의 매체**다.

- `/discuss`의 이해도 테이블 → 빈칸이 "내가 모르는 것"을 보여준다
- `/prd`의 8단계 체인 → 채우다 보면 놓친 것이 드러난다
- `/retrospect`의 역PRD → "실제로 만든 것"을 보면 원래 의도와의 갭이 보인다

**인사이트: 문서를 쓰는 행위 자체가 사고를 강제한다.** AI에게 "정리해줘"가 아니라 "같이 이 표를 채우자"로 접근하면, 문서 작성 과정이 의사결정 과정이 된다.

PARA 방법론(Projects, Areas, Resources, Archives)을 적용해서, 문서가 자연스럽게 **수명 주기를 따르도록** 했다. inbox에서 시작해서, 활성 프로젝트 → 영구 영역 → 아카이브로 흐른다.

## 9. mock 호출 검증을 버려라

테스트에서 `toHaveBeenCalled()`, `toHaveBeenCalledWith()` 같은 mock 호출 검증을 사용하면, 테스트가 구현에 결합된다. AI는 이런 테스트를 잘 통과시키지만, **실제 동작과 무관한 테스트**가 된다.

적용한 원칙:
```
계산 → unit test (순수 함수)
인터랙션 → 통합 test (userEvent → DOM 상태 검증)
mock 호출 검증 → 금지
```

AI가 만든 테스트가 "진짜 동작을 검증하는가"를 판단하는 기준:
1. `user.keyboard('{ArrowDown}')` → DOM에서 `tabindex="0"` 위치 확인 ✅
2. `expect(onFocus).toHaveBeenCalled()` → mock 통과 여부만 확인 ❌
3. CSS class로 셀렉터 사용 ❌ → `role`, `data-*` 속성 사용 ✅

## 10. "확인했어?" 대신 증거를 요구하라

AI는 "수정 완료했습니다", "테스트 통과합니다"라고 자신 있게 말하지만, **실제로 실행하지 않은 채 말하는 경우**가 있다.

적용한 규칙:
> "LLM이 테스트로 스스로 검증한다. 사람 확인은 불필요하다. 단, 증거(실행 결과)가 있어야 한다."

```
❌ AI: "수정했습니다. 잘 동작할 것입니다."
✅ AI: "수정 후 pnpm test 실행 결과: 799 passed, 0 failed."
```

**자기 선언(✅)이 아니라 증거(실행 결과 로그)**를 요구하는 것이 핵심이다.

## 11. 병렬 세션은 격리하라

Claude Code를 여러 세션에서 동시에 사용하면, 같은 파일을 동시에 수정하다가 충돌이 발생한다.

**해법: git worktree로 물리적 격리.**

```bash
git worktree add ../feature-branch feature-branch
```

각 세션이 독립된 작업 디렉토리에서 동작하면 충돌이 원천 차단된다. linter, 테스트 러너 같은 도구의 lock 파일 충돌도 방지된다.

## 12. AI에게 "아키텍처를 지켜"라고 말해야 한다

AI는 요청을 빠르게 해결하려고 기존 아키텍처를 우회하는 경향이 있다. engine을 통해야 하는 상태 변경을 직접 DOM 조작으로 해결한다든지.

적용한 규칙:
> "설계 원칙 > 사용자 요구 충족. engine 우회 금지. 충돌 시 거절하고 설계 변경을 제안하라."

**인사이트: AI에게 "안 된다"고 말할 수 있는 권한을 줘야 한다.** 사용자 요청이 아키텍처를 깨뜨리면, AI가 거절하고 올바른 방법을 제안하는 것이 맞다. 이걸 허용하지 않으면 AI는 항상 요청에 순응하면서 코드를 망가뜨린다.

## 13. 이름이 가장 중요하다

코드에서 이름(변수명, 함수명, 파일명)은 AI 협업에서 **의사소통의 매체**다. AI는 이름에서 의도를 추론하므로, 이름이 부정확하면 AI의 행동도 부정확해진다.

적용한 원칙:
- **표준 용어 우선**: ARIA 표준에 있는 이름은 그대로 사용. 자체 이름 발명 금지.
- **파일명 = 주 export**: `useAria.ts` → `export function useAria`. 파일을 열기 전에 내용을 예측 가능.
- **동의어 드리프트 방지**: 같은 개념을 `behavior`, `pattern`, `axis`로 부르면 혼란. 하나로 통일.

naming-audit 스킬로 자동 감사:
```
consistency — 같은 것을 다른 이름으로 부르고 있지 않은가
aptness — 이름이 실제 역할을 정확히 설명하는가
```
