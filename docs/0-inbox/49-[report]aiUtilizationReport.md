## 1. 개요

### 1.1 목적

본 보고서는 Claude Code(Anthropic CLI)를 활용한 AI 협업 개발 체계의 **구축 현황, 정량 성과, 운영 방법론, 향후 계획**을 공유한다. 단순 코드 생성 보조가 아닌, **사고→계획→실행→검증→개선의 전체 개발 사이클을 AI와 구조적으로 협업**하는 방식을 다룬다.

### 1.2 배경

2026년 1월부터 개인 프로젝트(interactive-os: 키보드 우선 ARIA 프레임워크)에서 Claude Code를 전면 도입하여 AI 협업 워크플로우를 실험·검증해왔다. 약 3개월간의 운영을 통해 **재현 가능한 방법론**으로 정립되었으며, 이를 사내 개발 프로세스에 적용 가능한지 검토하기 위해 본 보고서를 작성한다.

## 2. 구축한 시스템

### 2.1 AI 협업 워크플로우 (Discuss-First Pipeline)

AI와의 협업에서 가장 큰 리스크는 **"잘못된 것을 빠르게 만드는 것"**이다. LLM은 모호한 요청도 즉시 실행하므로, 의도가 정렬되지 않은 채 코드가 수백 줄 생성된 후에야 방향 오류를 발견하게 된다.

이를 해결하기 위해 **"실행 전 사고 정렬"**을 강제하는 5단계 파이프라인을 구축했다.

```
discuss ──→ prd ──→ go ──→ retrospect ──→ improve
(의도 정렬)  (명세)  (실행)   (회고)        (자가 개선)
```

#### 각 단계 상세

| 단계 | 역할 | 입력 | 출력 |
|------|------|------|------|
| **discuss** | 문제 구조화 + 의도 정렬 | 사용자의 모호한 요청 | 11개 요소 이해도 테이블 |
| **prd** | 구현 명세 작성 | 합의된 방향 | 8단계 의존 체인 PRD |
| **go** | 자율 실행 | PRD | 코드 + 테스트 |
| **retrospect** | 의도 vs 결과 비교 | git diff + 원본 PRD | 갭 분석 + 역PRD |
| **improve** | 점수 기반 자율 개선 | 점수 함수 | 반복 수정 |

#### 이론적 기반

이 파이프라인은 단일 이론의 적용이 아니라, **5개 독립 학문의 교차점**에서 설계되었다.

| 이론 | 출처 분야 | 파이프라인 역할 |
|------|----------|---------------|
| TOC Thinking Processes (Goldratt) | 경영/제조 | discuss의 11요소 구조화 |
| OODA Loop (Boyd) | 군사 전략 | discuss = Orient (정향) 단계 |
| Double-Loop Learning (Argyris) | 조직 학습 | "왜?"부터 시작 = 가정 자체 질문 |
| Deliberate Practice (Ericsson) | 인지 과학 | retrospect + improve = 피드백 루프 |
| PDCA (Deming) | 품질 관리 | 전체 순환 구조의 골격 |

> 5개 이론이 군사, 제조, 학습 과학, 조직 이론에서 **독립적으로** 동일 구조로 수렴한다는 것은, 이 구조가 인간 인지의 기본 패턴을 반영한다는 것을 시사한다.

### 2.2 스킬 에코시스템

Claude Code의 **커스텀 스킬** 기능을 활용하여, 워크플로우의 각 단계를 슬래시 커맨드(`/discuss`, `/prd`, `/go` 등)로 호출 가능하도록 자동화했다.

#### 스킬 분류 (26개)

| 분류 | 스킬 | 역할 |
|------|------|------|
| **사고 루프** (코어) | discuss, prd, go, retrospect, improve, close | 개발 사이클 전체를 순환 |
| **사고 도구** | conflict, doubt, ideal, explain, resource | 루프 내에서 필요 시 호출 |
| **품질 게이트** | naming-audit, demo-coverage, simplify | 실행 결과 자동 검증 |
| **문서 관리** | inbox, para, area, publish, backlog | 지식 누적 + 정보 흐름 관리 |
| **구현 도구** | debugging, design-extract, design-implement | 도메인 특화 자동화 |

#### 주요 스킬 상세

**discuss (의도 정렬)**
- TOC의 Thinking Processes를 기반으로 11개 요소(목적, 배경, 이상적 결과, 현실, 문제, 원인, 제약, 목표, 해결, 부작용, 장애물)의 이해도를 🟢/🟡/🔴로 추적
- 모든 요소가 🟢이 되면 자동으로 FRT(미래현실트리) 역검증 수행
- **"AI가 질문할 때 판단을 먼저 밝힌다"** 규칙으로 AI의 빈 질문을 방지

**retrospect (회고)**
- git diff에서 역PRD(실제 구현 명세)를 자동 생성
- 원본 PRD와 역PRD를 대조하여 갭을 자동 감지
- 갭이 발견되면 backlog에 추가하거나 다음 사이클로 연결

**improve (자가 개선 루프)**
- 프로젝트에 정의된 점수 함수(예: `pnpm score:design`)를 반복 실행
- 점수가 목표에 도달할 때까지 **측정→진단→수정→재측정**을 자율 반복
- 사람 개입 없이 점수를 올리는 "hill climbing" 패턴
- 실적: 디자인 점수 49% → 96% 달성 (자율 루프)

현재 60+건의 메모리가 누적되어 있으며, 새 세션에서도 이전 대화의 맥락과 피드백이 반영된다.

### 3.2 워크플로우 성과

| 성과 | 상세 |
|------|------|
| **의도 불일치 감소** | discuss 도입 후 "방향 틀어짐 → 재작업" 패턴이 대폭 감소 |
| **자가 개선 루프** | 디자인 점수 49% → 96% (사람 개입 없이 AI 자율 달성) |
| **지식 누적** | 70+ 문서가 PARA 체계로 누적, 세션 간 맥락 유지 |
| **CI 안정화** | 10+ 연속 failure → green 전환 + 병렬화 + Node 24 + 패키지 전수 최신화 |

### 3.3 핵심 교훈

1. **"실행 전 사고 정렬"이 가장 높은 ROI**: AI가 빠르게 실행할 수 있다는 것이 오히려 리스크. discuss로 의도를 먼저 정렬하면 재작업이 극적으로 줄어든다.
2. **PRD의 촘촘함이 품질을 결정**: 애자일 스토리보다 촘촘한 PRD가 AI 협업에서 더 효과적. 컨텍스트 오염 방지.
3. **피드백은 즉시 영속화**: 한 번 교정한 행동이 다음 세션에서도 유지되려면, memory에 즉시 저장해야 한다.
4. **최소 구현이 좋은 특성**: 명세가 촘촘하면 구현이 최소로 수렴한다. 이는 버그가 아니라 정확성의 증거.

## 4. 운영 방법론

### 4.1 일상 개발 플로우

```
1. /discuss로 시작 — 무엇을 왜 하는지 정렬
2. /prd로 구체화 — 인터페이스, 경계, 검증 기준 명세
3. /go로 실행 — AI가 자율적으로 Plan→Execute→Verify
4. /retrospect로 회고 — 의도 vs 결과 갭 감지
5. /improve로 개선 — 점수 함수 기반 자율 루프
6. /close로 마무리 — 상태 갱신, 문서 누적, 커밋
```

### 4.2 품질 게이트

| 시점 | 자동 실행 | 내용 |
|------|----------|------|
| 실행 완료 후 | /naming-audit | 네이밍 일관성/적합성 감사 |
| 커밋 전 | /simplify | 코드 품질/효율 리뷰 |
| 회고 시 | /retrospect | PRD 대비 갭 분석 |
| CI | lint + test + build | 자동화된 품질 관문 |

### 4.3 지식 관리 플로우

```
발생 → /inbox (0-inbox/)
  ├─ 구현 결정 → /prd (1-projects/)
  ├─ 영구 지식 → /area (2-areas/)
  ├─ 외부 조사 → /resource (3-resources/)
  ├─ 완료/폐기 → (4-archive/)
  └─ 보류 → /backlog (5-backlogs/)
```

### 4.4 디버깅 프로토콜

AI에게 "고쳐줘"라고 말하는 것이 아니라, **구조화된 4단계**를 강제한다:

1. **재현**: 코드를 읽기 전에 테스트로 재현
2. **관측 도구 검증**: 디버깅 도구 자체의 버그를 먼저 배제
3. **설계 의도 파악**: 수정 전에 원래 설계를 이해
4. **수정 판단**: 테스트 통과 ≠ 올바른 수정. 설계에 맞는지 확인

## 5. 향후 계획

### 5.1 범용 플러그인화 (진행 중)

현재 프로젝트에 종속된 스킬 에코시스템을 **범용 글로벌 플러그인**으로 분리하는 작업을 진행 중이다.

| 항목 | 현황 | 목표 |
|------|------|------|
| 범용 스킬 | 7개 (35%) | 전체 26개 범용화 |
| 프로젝트 설정 | 스킬에 하드코딩 | project.yml로 분리 |
| 배포 | 로컬 프로젝트 내 | 글로벌 플러그인 마켓플레이스 |

#### 핵심 설계 방향

- **프로젝트 프로필 (project.yml)**: CLAUDE.md(행동 지시), MEMORY.md(대화 기억)와 별도로, "이 프로젝트의 구조와 관심사"를 선언하는 제3의 설정 레이어
- **관심사별 3-layer**: 문서/네이밍/테스트/디자인/품질 각 관심사마다 철학(why) → 구조(what) → 도구(how)를 선언
- **프로필 없어도 동작**: 글로벌 스킬은 project.yml이 없는 프로젝트에서도 합리적 기본값으로 동작

### 5.2 사내 적용 검토 사항

| 검토 항목 | 상태 | 비고 |
|----------|------|------|
| 보안 | 검토 필요 | 코드가 외부 API로 전송되는 범위 확인 |
| 라이선스 | 검토 필요 | Anthropic 이용 약관 + 사내 정책 |
| 비용 | 검토 필요 | API 사용량 기반 과금 구조 |
| 팀 확산 | 단계적 | 코어 루프(discuss→go)부터 시범 적용 |
| 교육 | 필요 | /discuss 사용법 워크숍 → 자연 확산 |

### 5.3 기대 효과

| 영역 | 기대 효과 |
|------|----------|
| **의사결정 품질** | discuss-first로 "왜?"부터 정렬 → 방향 오류 감소 |
| **개발 속도** | PRD + go로 구조화된 실행 → 재작업 감소 |
| **지식 누적** | PARA + 자동 문서화 → 팀 컨텍스트 유지 |
| **코드 품질** | 자동 게이트(naming, simplify, test) → 일관성 향상 |
| **온보딩** | 누적된 memory + 문서 → 새 팀원/세션의 학습 곡선 단축 |
