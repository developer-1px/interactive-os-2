# AI한테 일 잘 시키는 법 — 3개월간 Claude Code 전면 사용 후기

> 작성일: 2026-03-25
> 작성자: FE개발

---

## 이 글의 목적

AI 코딩 도구를 이미 쓰고 있는 분들을 위한 글입니다. Copilot이든 Claude든 ChatGPT든, "AI가 코드를 만들어주는 건 아는데, 뭔가 아쉬운" 분들에게. 3개월간 Claude Code를 메인 도구로 사용하면서 발견한 것들을 공유합니다.

대단한 건 아닙니다. 보고 나면 "아 그렇게 하면 되는 거였어?" 싶은 것들입니다.

---

## 1. AI한테 "만들어줘"라고 하지 마세요

가장 큰 발견입니다.

AI 코딩 도구의 최대 장점이 "빠르게 만들어준다"인데, 이게 최대 함정이기도 합니다. "로그인 기능 만들어줘"라고 하면 AI는 즉시 200줄을 뽑아냅니다. 문제는, 그 200줄이 내가 원한 것과 **살짝 다른 방향**일 때 발생합니다.

200줄을 읽고, 이해하고, 부분 수정하는 데 걸리는 시간 ≥ 처음부터 방향 맞추고 만드는 시간.

그래서 저는 모든 작업을 **"왜 이걸 하는가?"라는 질문**으로 시작합니다.

```
❌ "로그인 기능 만들어줘"
✅ "우리 서비스에 인증이 왜 필요한지부터 정리하자"
```

이 한 단계를 추가한 후 재작업이 체감 70% 이상 줄었습니다. AI가 빠르다는 건, 방향이 맞을 때만 장점입니다.

### 왜 이게 동작하는가

OODA Loop(관찰→정향→결정→행동)을 만든 John Boyd는 4단계 중 **Orient(정향)**가 가장 중요하다고 했습니다. 빠르게 행동하는 것보다, 상황을 어떻게 해석하는지의 프레임을 먼저 맞추는 것이 의사결정 품질을 결정합니다.

"만들어줘"는 Orient를 건너뛰고 Act로 직행하는 것. "왜부터 정리하자"는 Orient를 먼저 하는 것. 군사 전략에서 검증된 원리가 AI 협업에도 그대로 적용됩니다.

### 실제로 하는 방법

저는 `/discuss`라는 커스텀 커맨드를 만들어서 씁니다. 하지만 커맨드가 없어도 됩니다. 핵심은 **AI한테 코딩을 시키기 전에 이 4개 질문을 먼저 던지는 것**입니다:

1. **왜** 이걸 하려는가?
2. 지금 **현실**은 어떤가?
3. 다 되면 **어떤 모습**인가?
4. 현실과 이상의 **갭**은 뭔가?

이 4개에 답하고 나면, AI한테 내릴 지시가 훨씬 명확해집니다. AI도 정확한 지시에는 정확한 결과를 냅니다.

## 2. AI한테 질문을 시키지 말고, 판단을 시키세요

AI의 기본 습성이 있습니다. 뭘 물어보면 **"A와 B 중 어떤 걸 원하시나요?"**라고 되묻는 겁니다. 겸손해 보이지만, 이건 결국 판단의 부담을 나한테 다시 넘기는 것입니다.

이걸 바꿨습니다:

```
❌ AI: "REST API와 GraphQL 중 어느 것을 선호하시나요?"
✅ AI: "제 판단: REST. 이유: 현재 팀 스택이 REST 기반이라 학습 비용이 낮습니다.
       다르게 생각하시면 말씀해주세요."
```

방법은 간단합니다. CLAUDE.md(프로젝트 설정 파일)에 한 줄 넣으면 됩니다:

```
질문할 때 판단을 먼저 밝힌다.
```

이것만으로 대화 턴 수가 절반으로 줄었습니다. AI가 판단을 제시하면, 나는 동의하거나 반박만 하면 됩니다. **생각하는 비용이 AI에게 이전**됩니다.

### 왜 이게 동작하는가

인지 부하(Cognitive Load) 이론에서, 의사결정의 가장 비싼 단계는 **선택지 생성**입니다. 선택지가 주어진 상태에서 고르는 건 비교적 쉽습니다. AI가 판단을 먼저 제시하면, 사용자는 "생성"이 아니라 "평가"만 하면 됩니다. 생성은 비싸고, 평가는 싸다.

## 3. 피드백을 파일로 저장하세요

AI한테 "그렇게 하지 마"라고 한 번 말하면, 그 세션에서는 고쳐집니다. 그런데 다음 날 새 세션을 열면? **처음부터 다시 같은 교정을 해야 합니다.**

이게 가장 큰 시간 낭비였습니다. 같은 말을 매일 반복하는 거.

해결법은 놀랍도록 단순합니다. **피드백을 파일로 저장**하면 됩니다.

```markdown
# .claude/memory/no-mock-verification.md
테스트에서 toHaveBeenCalled() 금지.
이유: 구현에 결합된 테스트가 된다. DOM 상태로 검증할 것.
```

이 파일이 있으면 AI가 새 세션에서도 자동으로 읽고 반영합니다. 교정을 한 번만 하면 **영구히 적용**됩니다.

저는 3개월간 60개 넘는 피드백을 이렇게 저장해왔습니다. 지금 새 세션을 열어도 3개월간의 모든 교정이 한 번도 말하지 않아도 반영됩니다.

**하나 더 중요한 팁: 잘한 것도 저장하세요.** 실수 교정만 쌓으면 AI가 점점 소극적으로 변합니다. "이 접근 좋았다, 계속 이렇게 해"도 기록해야 균형이 잡힙니다.

### 왜 이게 동작하는가

K. Anders Ericsson의 Deliberate Practice(의도적 연습) 이론의 핵심은 **즉각적 피드백 + 반복**입니다. 전문가와 비전문가를 가르는 건 재능이 아니라 "피드백을 받고 수정하는 사이클의 밀도"입니다.

피드백을 파일로 저장하는 건, 이 피드백 루프를 **세션을 넘어 영속화**하는 것입니다. 매 세션이 이전 피드백 위에 쌓이므로, AI의 행동이 deliberate practice처럼 점진적으로 정교해집니다.

## 4. "왜?"라고 물으면 AI가 사과부터 합니다. 이걸 고치세요.

AI한테 "왜 이렇게 했어?"라고 물으면, 대부분 이렇게 반응합니다:

> "죄송합니다, 다른 방식으로 변경하겠습니다."

근데 나는 진짜로 이유가 궁금해서 물은 겁니다. 그 방식이 맞을 수도 있는데 AI가 먼저 철회해버립니다.

CLAUDE.md에 한 줄 추가했습니다:

```
"왜"는 이유를 묻는 질문이다. 반대 의사로 해석하지 않는다.
```

이후로 AI가 자기 판단을 설명하고 방어하기 시작했습니다. 대화 품질이 확 올라갑니다. AI가 "네 말이 맞으니까 바꿀게"가 아니라 "내가 이렇게 한 이유는 이건데, 너는 왜 다르게 보는지 말해줘"로 바뀝니다.

### 왜 이게 동작하는가

Chris Argyris의 Double-Loop Learning 이론에서, 조직 학습의 최대 장벽은 **defensive reasoning**(방어적 추론)입니다. 사람이든 AI든, "틀렸을 수 있다"는 신호를 받으면 방어적으로 후퇴합니다. "왜?"를 반대 의사로 해석하는 건 전형적인 defensive reasoning입니다.

이 한 줄 규칙은 AI의 defensive reasoning을 해제하고, **가정 자체를 질문하는 double-loop**을 가능하게 합니다. Single-loop("행동만 수정")이 아니라 double-loop("왜 그렇게 생각했는지 검토")으로 전환되는 것.

## 5. AI한테 "고쳐줘" 대신 "설계부터 파악해"

테스트가 실패하면 AI는 반사적으로 "통과하도록" 고칩니다. 빨리 초록불을 보여주려는 건데, 여기서 사고가 납니다.

실제로 겪은 일: AI가 테스트를 통과시켰는데, 원래 아키텍처의 핵심 규칙을 깨뜨린 채로 통과시킨 겁니다. 테스트는 초록이지만 시스템은 망가진 상태.

그 이후로 디버깅 규칙을 하나 만들었습니다:

```
수정하기 전에 "원래 왜 이렇게 되어있는지"를 먼저 파악한다.
```

이것도 CLAUDE.md에 넣으면 됩니다. AI가 버그를 고치기 전에 먼저 "이 코드의 설계 의도는 이런 것 같은데, 맞나요?"라고 물어보게 됩니다. 5초가 더 걸리지만, 잘못된 수정을 원복하는 30분을 아낍니다.

### 왜 이게 동작하는가

Chesterton's Fence 원칙입니다. "울타리가 왜 있는지 모르면, 그 울타리를 치우지 마라." 코드의 모든 줄에는 이유가 있습니다(없을 수도 있지만, 확인 전에는 모릅니다). AI가 이유를 모른 채 수정하면, 의도치 않게 울타리를 치우는 꼴이 됩니다.

이건 소프트웨어 공학의 **Lindy Effect**와도 연결됩니다. 오래 살아남은 코드일수록 이유가 있을 확률이 높고, 그 이유를 이해하지 않은 수정은 regression 확률이 높습니다.

## 6. 점수를 매기면 AI가 혼자 개선합니다

이게 가장 콜롬버스의 달걀이었습니다.

"코드 품질을 올려줘"라고 하면 AI는 뭘 해야 할지 모릅니다. 하지만 **숫자를 주면** 행동합니다.

```bash
$ pnpm score:design
> 디자인 토큰 적용률: 49%
```

"이 숫자를 올려줘"라고 하면 AI가 스스로:
1. 낮은 점수 항목을 찾고
2. 수정하고
3. 다시 측정하고
4. 반복합니다

사람이 한 건 **점수 스크립트를 만든 것**뿐입니다. AI가 혼자 49%에서 96%까지 올렸습니다.

핵심 아이디어: **측정할 수 있으면 AI가 개선할 수 있다.** lint 규칙, 커버리지, 토큰 적용률, 번들 사이즈 — 숫자로 나오는 건 뭐든 이 패턴이 동작합니다.

### 왜 이게 동작하는가

Ericsson의 Deliberate Practice + Deming의 PDCA(Plan-Do-Check-Act) 순환의 결합입니다.

Deliberate Practice의 4조건: ①명확한 목표 ②즉각적 피드백 ③반복 기회 ④오류 활용. 점수 스크립트가 ①②를 제공하고, AI의 반복 실행이 ③④를 담당합니다. 사람은 목표(점수 함수)만 정의하고, 연습(반복 개선)은 AI가 합니다.

이건 사실상 **hill climbing 최적화**를 AI가 수동으로 하는 것인데, 코드라는 도메인에서는 gradient가 없어도 "점수가 낮은 항목을 식별하고 수정"하는 휴리스틱으로 충분히 수렴합니다.

적용하려면:
1. 관심 있는 품질 지표를 숫자로 출력하는 스크립트를 만든다
2. AI한테 "이 스크립트 실행해서 점수 확인하고, 낮은 항목 수정하고, 다시 측정해"라고 한다
3. 끝

## 7. CLAUDE.md가 진짜 무기입니다

위에서 계속 "CLAUDE.md에 넣으면 된다"고 했는데, 정리하면 이겁니다.

CLAUDE.md는 **AI의 행동 규칙을 선언하는 파일**입니다. 프로젝트 루트의 `.claude/CLAUDE.md`에 두면 모든 세션에서 자동 적용됩니다.

대부분 비워두거나 "이 프로젝트는 React입니다" 정도만 쓰는데, 여기에 **행동 규칙**을 넣으면 AI가 완전히 달라집니다.

제가 실제로 넣어둔 것들 중 범용적으로 쓸 수 있는 것들:

```markdown
# .claude/CLAUDE.md

- 질문할 때 판단을 먼저 밝힌다
- "왜"는 이유를 묻는 질문이다. 반대 의사로 해석하지 않는다
- 수정 전 설계 의도 이해 필수. 테스트 통과 ≠ 올바른 수정
- 설계 원칙이 사용자 요구와 충돌하면, 거절하고 올바른 방법을 제안한다
```

4줄입니다. 이 4줄만 넣어도 AI와의 협업 품질이 체감상 2배는 됩니다.

**핵심: AI한테 "잘해줘"가 아니라 "이렇게 행동해"라고 말하는 것.** 모호한 기대 대신 명시적 규칙.

### 왜 이게 동작하는가

경영학의 **Governing Variables**(Argyris)와 같은 개념입니다. 조직의 행동을 바꾸려면 개별 행동을 교정하는 게 아니라, 행동을 지배하는 변수(가치, 규범, 목표)를 바꿔야 합니다. CLAUDE.md는 AI의 governing variables를 선언하는 파일입니다.

"매번 이렇게 해줘"(single-loop)가 아니라 "이런 원칙으로 판단해"(double-loop)를 설정하는 것. 한 번 설정하면 모든 세션에 적용되므로, 교정 비용이 O(n)에서 O(1)로 줄어듭니다.

## 8. 커스텀 커맨드를 만드세요

Claude Code에서 `.claude/skills/` 폴더에 마크다운 파일을 넣으면 슬래시 커맨드가 됩니다. 이게 게임 체인저입니다.

반복하는 작업 패턴이 있으면 스킬로 만들 수 있습니다.

### 제일 쉽게 시작하는 방법

가장 자주 하는 3가지 유형의 작업을 생각하고, 각각에 대해 **"AI한테 매번 설명하는 말"**을 파일로 저장하면 됩니다.

예를 들어, 매번 코드 리뷰를 부탁할 때 이렇게 말하고 있다면:

```
이 PR 봐줘. 네이밍이 일관적인지, 불필요한 코드 없는지,
테스트 누락은 없는지 확인해줘. 의견만 주고 직접 수정은 하지 마.
```

이걸 `.claude/skills/review/SKILL.md`로 저장합니다:

```markdown
---
name: review
description: 코드 리뷰. "/review", "리뷰해줘" 등을 말할 때 사용.
---

이 PR을 리뷰한다.
1. 네이밍 일관성 확인
2. 불필요한 코드 감지
3. 테스트 누락 확인
의견만 제시하고 직접 수정은 하지 않는다.
```

이제 `/review`만 치면 됩니다. 매번 같은 말을 반복할 필요가 없어집니다.

### 왜 이게 동작하는가

소프트웨어 공학의 **DRY 원칙**(Don't Repeat Yourself)을 사람-AI 커뮤니케이션에 적용한 것입니다. 같은 지시를 반복하는 건 코드에서 같은 로직을 복붙하는 것과 같습니다. 한 곳에서 정의하고 이름으로 호출하는 것이 유지보수도 쉽고 일관성도 보장됩니다.

또한 인지 과학의 **chunking** 개념과 연결됩니다. 전문가는 복잡한 절차를 하나의 청크(이름)로 압축합니다. `/review`라는 한 단어가 5줄의 절차를 압축하고 있는 것. 이 압축이 작업 기억(working memory)의 부하를 줄여줍니다.

### 제가 만든 건 26개입니다

처음부터 26개를 만든 게 아닙니다. 반복 패턴이 보일 때마다 하나씩 추가한 겁니다. 3개월이 지나니 26개가 됐습니다.

처음에 만들면 좋은 것 3개를 추천하면:

1. **작업 시작 전 정리** — "왜 하는지, 뭘 만들지 정리해줘" (= discuss)
2. **작업 완료 후 체크** — "변경 코드 리뷰해줘" (= simplify)
3. **메모 저장** — "이거 파일로 남겨줘" (= inbox)

## 9. 문서를 "나중에 정리하는 것"으로 생각하지 마세요

이건 AI 협업에서 발견한 관점 전환입니다.

문서는 사고의 **결과물**이 아니라 사고의 **도구**입니다.

제가 사용하는 `/discuss`의 이해도 테이블을 예로 들면:

```
| 요소        | 내용         | 이해도 |
|------------|-------------|--------|
| 목적        | 인증 도입     | 🟢 90% |
| 현실        | 현재 미인증   | 🟢 90% |
| 이상적 결과  | ?            | 🔴 20% |
```

빈칸(🔴)이 **"내가 아직 모르는 것"**을 보여줍니다. 이 표를 채우는 과정이 곧 사고 과정입니다.

PRD도 마찬가지입니다. 8개 칸(동기, 산출물, 인터페이스, 경계, 원칙대조, 부작용, 금지, 검증)을 채우다 보면 놓친 것이 저절로 드러납니다.

**문서를 쓰는 행위 자체가 빠진 것을 발견하는 행위입니다.** AI한테 "정리해줘"가 아니라 "같이 이 표를 채우자"로 접근해보세요.

### 왜 이게 동작하는가

교육학의 **externalization**(외재화) 개념입니다. 머릿속에 있는 생각을 글, 표, 다이어그램으로 밖으로 꺼내면, 논리적 빈틈이 눈에 보이기 시작합니다. 생각만으로는 "대충 맞는 것 같은데"에 머물지만, 표의 빈칸은 "여기 모른다"를 강제로 드러냅니다.

Goldratt의 TOC(제약 이론)에서도 같은 원리를 씁니다. 현실 트리(CRT)를 그리면 "왜 이 문제가 존재하는지"의 인과 사슬이 보이고, 미래 트리(FRT)를 그리면 "이 해결이 진짜 문제를 푸는지"가 검증됩니다. 트리를 그리는 행위 자체가 사고입니다.

## 10. AI한테 "안 돼"라고 말할 권한을 주세요

이상하게 들리지만, 이게 중요합니다.

AI는 기본적으로 요청에 순응합니다. "이렇게 해줘"라고 하면 아키텍처를 깨뜨려서라도 요청대로 합니다. 빠른 해결이 기존 구조를 우회하는 경우가 많습니다.

```markdown
# CLAUDE.md에 추가
설계 원칙이 사용자 요구와 충돌하면, 거절하고 올바른 방법을 제안한다.
```

이 한 줄이 있으면 AI가 "그렇게 하면 기존 아키텍처와 충돌합니다. 대신 이런 방법은 어떤가요?"라고 말합니다. 처음엔 거슬리지만, 이게 아키텍처를 지켜줍니다.

**핵심: AI를 "시키는 대로 하는 도구"가 아니라 "설계를 지키는 파트너"로 포지셔닝하는 것.**

### 왜 이게 동작하는가

소프트웨어 아키텍처의 **Architectural Decision Records(ADR)** 사상과 연결됩니다. 아키텍처 결정은 한 번 내리면 이후의 모든 구현이 그 위에 쌓입니다. 개별 요청이 아키텍처를 침식하면 기술 부채가 누적됩니다.

AI에게 거절 권한을 주는 건, **guardrail을 코드가 아니라 대화 레벨에 설치**하는 것입니다. CI에서 lint가 규칙 위반을 잡아주듯이, AI가 설계 위반을 대화 단계에서 잡아줍니다. 비용은 "나중에 발견"보다 "지금 발견"이 항상 싸다.

## 정리: 내일 당장 할 수 있는 3가지

| 순서 | 할 일 | 소요 시간 |
|------|------|----------|
| **1** | `.claude/CLAUDE.md`에 위 4줄 규칙 넣기 | 1분 |
| **2** | AI한테 코딩 시키기 전에 "왜?" 4개 질문 먼저 던져보기 | 5분 |
| **3** | AI가 잘못한 거 교정했으면 `.claude/memory/`에 파일로 저장 | 2분 |

이 3가지만 해도 AI 협업의 체감 품질이 달라집니다. 커스텀 커맨드나 자가 개선 루프 같은 건 그다음입니다.
