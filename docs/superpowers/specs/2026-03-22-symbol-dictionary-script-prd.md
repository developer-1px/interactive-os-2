# Symbol Dictionary Script — PRD

> Discussion: 프로젝트 규모 증가에 대비하여, LLM 네이밍 분석의 시드 데이터를 기계적으로 수집하는 스크립트를 만든다. 모든 심볼을 AST 파싱 → camelCase split → fragment 그룹핑 → JSON 출력.

## ① 동기

| # | Given | When | Then | 역PRD |
|---|-------|------|------|-------|
| M1 | 프로젝트에 수백 개의 심볼이 존재한다 | LLM에게 네이밍 분석을 의뢰한다 | LLM이 소스 코드 전체를 읽어야 하므로 컨텍스트 초과 또는 비용 과다 | |
| M2 | naming-dictionary.md가 수동 관리된다 | 새 심볼이 추가된다 | 딕셔너리가 갱신되지 않아 누락이 쌓인다 | |
| M3 | LLM이 직접 심볼을 수집한다 | 같은 코드를 두 번 분석한다 | 비결정적 결과가 나온다 (수집 누락, 다른 그룹핑) | |

완성도: 🟢

## ② 산출물

> 구조, 관계, 이름 — 파일/컴포넌트/데이터 스키마

| 산출물 | 설명 | 역PRD |
|--------|------|-------|
| `scripts/collectSymbols.ts` | TS compiler API로 심볼 수집 → JSON 출력하는 CLI 스크립트 | |
| stdout JSON | fragment dictionary — fragment별 그룹핑된 식별자 목록 | |

**출력 JSON 스키마:**

```jsonc
{
  // fragment = camelCase/PascalCase split 후 소문자화한 단어
  "<fragment>": {
    "identifiers": [
      {
        "name": "createStore",       // 원본 식별자
        "file": "src/core/createStore.ts"  // 파일 경로 (프로젝트 루트 상대)
      }
    ]
  }
}
```

## ③ 인터페이스

| 입력 | 현재 상태 | 행동 | 왜 이 결과가 나는가 | 결과 상태 | 역PRD |
|------|----------|------|-------------------|----------|-------|
| `npx tsx scripts/collectSymbols.ts` | src/ 디렉토리에 TS/TSX 소스 존재 | TS compiler API로 AST 순회, 모든 식별자 선언(변수, 함수, 클래스, 타입, 인터페이스, enum, 파라미터, destructuring 바인딩) 수집 | AST는 모든 선언을 정확하게 캡처하므로 정규식 대비 누락 없음 | stdout에 JSON 출력 | |
| 수집된 식별자 `createBehaviorContext` | — | camelCase/PascalCase 경계에서 split → `["create", "behavior", "context"]` (소문자화) | 네이밍 컨벤션이 camelCase/PascalCase이므로 대소문자 경계 = 단어 경계 | 3개 fragment 각각에 이 식별자가 등록됨 | |
| 1글자 식별자 `i`, `x`, `_` 등 | — | 1글자 식별자는 제외 | 루프 변수 등 의미 없는 심볼. 2글자 이상은 LLM이 알아서 거름 | 출력에서 제외 | |
| UPPER_SNAKE_CASE `FOCUS_ID` | — | `_` 기준으로 split → `["focus", "id"]` (소문자화) | 상수는 UPPER_SNAKE_CASE 관례 | 2개 fragment에 등록 | |
| `__tests__/` 하위 파일 | — | 스킵 | 테스트 로컬 변수는 노이즈 | 수집하지 않음 | |
| `*.test.*`, `*.spec.*` 파일 | — | 스킵 | 동일 | 수집하지 않음 | |
| `dist/`, `node_modules/` | — | 스킵 | 번들/의존성은 분석 대상 아님 | 수집하지 않음 | |

완성도: 🟢
