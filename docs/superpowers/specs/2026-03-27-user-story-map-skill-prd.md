# User Story Map Skill — PRD

> Discussion: 서비스(CMS 등)에서 discuss→PRD 사이에 유저스토리 분해 레이어가 필요. 인터뷰로 4단 레이어(욕구→스토리→기능→화면)를 채워가는 스킬.

## ① 동기

### WHY (discuss FRT에서 이식)

- **Impact**: 서비스에서 discuss→PRD 직행하면 "어떤 구현 단위가 필요한가"를 도출할 수 없다. 전체 조망 불가, 진척도/릴리즈 판단 불가, PRD가 파편화됨.
- **Forces**: 인프라는 모듈=PRD 단위여서 이 레이어가 불필요했음. 서비스는 사용자 행동에서 구현 단위를 도출해야 함. 단, 사업 기획(Value/ROI)은 범위 밖.
- **Decision**: 4단 안정성 레이어(욕구→스토리→기능→화면) + 인터뷰 스킬. 기각 대안: (a) PRD에 유저스토리 섹션 추가 → 전체 조망 불가, (b) Area MDX를 스토리 기준 재조직 → 모듈 구조 문서와 역할 다름.
- **Non-Goals**: 비즈니스 가치/ROI 분석, 구현 언어(axis/pattern/plugin) 사용, 기획↔구현 매핑(PRD가 담당).

### 시나리오

| # | Given | When | Then | 역PRD |
|---|-------|------|------|-------|
| 1 | 새 서비스(CMS)의 discuss가 완료됨 | `/story` 실행 | 빈 유저스토리 맵 파일이 생성되고, 첫 인터뷰 질문(욕구)이 시작된다 | |
| 2 | 유저스토리 맵에 욕구 3개, 스토리 5개가 채워져 있음 | 다음 세션에서 `/story` 실행 | 기존 맵을 로드하고, 빈칸이 있는 첫 레이어를 찾아 인터뷰를 이어간다 | |
| 3 | 스토리가 전부 채워짐 | 스토리 하나를 선택하여 `/prd`로 전환 | 해당 스토리 + 연관 기능 + 화면 정보가 PRD ① 동기에 pre-fill된다 | |
| 4 | 인터뷰 중 스토리 크기 판단이 모호함 | AI가 "이것만 릴리즈 가능한가?" 테스트 제시 | 사용자가 스토리를 분할하거나 병합하여 릴리즈 단위로 조정한다 | |
| 5 | 서비스 전체 진척도를 확인하고 싶음 | `/story` 실행 후 맵 열람 | 욕구별 스토리 상태(⬜/🟢)가 한눈에 보여 릴리즈 가능 범위를 판단할 수 있다 | |

완성도: 🟡

### 유저스토리 맵 문서 구조

```markdown
# <Service Name> — User Story Map

## Needs (욕구)

| ID | 욕구 | 설명 |
|----|------|------|
| N1 | 사용자는 콘텐츠 편집을 하기 원한다 | ... |

## Stories (스토리 = 릴리즈 단위)

| ID | 욕구 | 스토리 | 수용조건 | PRD | 상태 |
|----|------|--------|---------|-----|------|
| S1 | N1 | 사용자는 원하는 콘텐츠를 선택하고 내용을 편집할 수 있다 | 디자인 변경 불가, 텍스트/이미지만 교체 | — | ⬜ |

## Features (기능)

| ID | 스토리 | 기능 | 화면 |
|----|--------|------|------|
| F1 | S1 | 선택 | W2 |
| F2 | S1 | 엔터로 편집모드 전환 | W2 |

## Screens (화면/위젯)

| ID | 화면 | 기능 | PRD |
|----|------|------|-----|
| W1 | 섹션 사이드바 | F5, F6 | — |
| W2 | 캔버스 | F1, F2, F3 | — |
```

### 레이어 간 참조 규칙

- 욕구 → 스토리: 1:N (욕구 하나에 여러 스토리)
- 스토리 → 기능: 1:N (스토리 하나에 여러 기능)
- 기능 → 화면: N:M (기능이 여러 화면에, 화면이 여러 기능을)
- 화면 → PRD: 1:1 (화면 단위로 PRD 작성)
- 스토리 상태: ⬜(미구현) / 🟢(릴리즈 완료) — 하위 기능이 전부 구현되면 🟢

### 안정성 레이어

```
욕구 (가장 안정)  ← 서비스 존재 이유, 거의 바뀌지 않음
  ↓
스토리 (릴리즈 단위) ← 스토리가 바뀌면 기능이 바뀜
  ↓
기능 (준안정)     ← 기능 조합이 바뀌면 화면이 재배치
  ↓
화면 (가변)       ← PRD 연결 지점, 가장 자주 바뀜
```

완성도: 🟡 ← ① 🟡이므로 초안

## ③ 인터페이스

> 스킬의 인터랙션 — 사용자가 스킬과 어떻게 대화하는가

| 입력 | 현재 상태 | 행동 | 왜 이 결과가 나는가 | 결과 상태 | 역PRD |
|------|----------|------|-------------------|----------|-------|
| `/story <service>` | 맵 파일 없음 | 빈 맵 파일 생성 + 욕구 인터뷰 시작 | 안정한 레이어부터 채워야 하위가 흔들리지 않음 | 맵 파일 생성됨, 욕구 인터뷰 중 | |
| `/story <service>` | 맵 파일 있음, 빈칸 있음 | 기존 맵 로드 + 빈칸 있는 첫 레이어로 인터뷰 재개 | 누적 가능해야 여러 세션에 걸쳐 작업 가능 | 인터뷰 재개 | |
| `/story <service>` | 맵 파일 있음, 전부 채워짐 | 맵 표시 + 진척도 요약 + PRD 전환 제안 | 전체 조망과 릴리즈 판단이 목적 | 열람 모드 | |
| 사용자가 욕구를 말함 | 욕구 인터뷰 중 | Needs 테이블에 행 추가 + 다음 욕구 질문 또는 스토리 전환 | 욕구가 충분하면 다음 레이어로 | 욕구 추가됨 | |
| 사용자가 스토리를 말함 | 스토리 인터뷰 중 | Stories 테이블에 행 추가 + 릴리즈 단위 검증("이것만 릴리즈 가능?") | 스토리 크기가 릴리즈 단위여야 PRD와 1:1 대응 | 스토리 추가됨 | |
| 사용자가 기능을 말함 | 기능 인터뷰 중 | Features 테이블에 행 추가 + 화면 배치 질문 | 기능이 어떤 화면에 속하는지가 PRD 단위 결정 | 기능 추가됨 | |
| "S1을 PRD로" | 스토리 채워짐 | 해당 스토리 + 연관 기능/화면 정보를 /prd에 전달 | 화면 단위 PRD 작성이 다음 단계 | /prd 전환 | |

완성도: 🟡 ← ② 🟡이므로 초안

## ④ 경계
