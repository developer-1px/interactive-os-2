# CMS Page Selection — PRD

> Story S1: 관리자는 편집할 페이지를 선택할 수 있다. 현재 CMS는 단일 페이지만 지원. 멀티 페이지 서비스가 되려면 페이지 목록/선택이 필요.

## ① 동기

### WHY

- **Impact**: CMS가 단일 페이지만 편집 가능. 관리자가 여러 랜딩페이지를 관리할 수 없어 서비스로 기능하지 못함.
- **Forces**: 기존 섹션 편집 엔진은 잘 동작함. 페이지 레이어를 추가하면서 기존 섹션 CRUD/네비게이션을 깨뜨리면 안 됨. 현재 데이터 모델은 ROOT → sections 직통이라 중간에 page 레이어가 없음.
- **Decision**: 데이터 모델에 `page` 엔티티를 추가하고, 사이드바에서 페이지 선택 → 해당 페이지의 섹션만 표시. 기각: (a) URL 파라미터만으로 페이지 구분 → 데이터 모델 변경 없이는 한 store에 여러 페이지 공존 불가, (b) 별도 라우트로 페이지별 CMS 인스턴스 → "하나의 앱 = 하나의 store" 원칙 위반.
- **Non-Goals**: 백엔드 API 연동, 배포 파이프라인, 페이지 생성/삭제 UI (이번 범위는 선택만).

### 시나리오

| # | Given | When | Then | 역PRD |
|---|-------|------|------|-------|
| 1 | CMS에 3개 페이지(Home, Docs, Blog)가 있음 | CMS에 접속 | 사이드바에 3개 페이지 목록이 보이고, 첫 번째 페이지가 자동 선택됨 | |
| 2 | Home 페이지를 편집 중 | 사이드바에서 Docs 페이지 클릭 | 캔버스에 Docs 페이지의 섹션들이 표시됨. Home 편집 내용은 유지됨 | |
| 3 | Docs 페이지를 편집 중 | 브라우저 URL이 /docs를 반영 | 새로고침해도 Docs 페이지가 선택된 상태로 복원됨 | |

완성도: 🟡

## ④ 경계

| 극단 조건 | 현재 상태 | 왜 이렇게 동작해야 하는가 | 예상 동작 | 결과 상태 | 역PRD |
|----------|----------|------------------------|----------|----------|-------|
| 페이지가 1개만 있음 | 단일 페이지 | 페이지 목록이 1개여도 구조는 동일해야 함 (미래 확장) | 페이지 목록에 1개 표시, 자동 선택 | 단일 페이지 선택됨 | |
| URL의 pageSlug가 존재하지 않는 페이지 | 잘못된 URL 접속 | 에러 대신 안전한 폴백 | 첫 번째 페이지로 리다이렉트 | 첫 페이지 선택됨 | |
| 페이지 전환 중 이전 페이지에 미저장 편집이 있음 | 페이지 A 편집 중 | 하나의 store이므로 모든 변경은 즉시 반영됨. 별도 저장 개념 없음 | 전환해도 A의 편집 내용 유지 | 페이지 B 선택, A 편집 유지 | |
| 섹션이 0개인 페이지 | 빈 페이지 | 새로 만든 페이지일 수 있음 | 빈 캔버스 + 섹션 추가 유도 | 빈 캔버스, [+] 활성 | |
