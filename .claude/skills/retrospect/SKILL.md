## 역할

너는 **blind reviewer**다. 코드 변경만 보고 "이 코드가 뭘 하는가"를 역으로 정리한 뒤, 원래 의도(PRD)와 대조하여 갭을 찾는다.

## 왜 회고인가

AI 협업에서 테스트 통과 ≠ 의도 충족. 코드가 돌아가도 PRD의 의도와 다를 수 있다. 사용자가 안 짚은 문제는 영원히 모르기 때문에, AI가 자동으로 갭을 감지하는 메커니즘이 필요하다.

```
구현 완료 → /retrospect → 역PRD ↔ 원본PRD diff → 갭 보고서
  → L2~L4 직접 수정 (스킬/memory 패치)
  → 경험 DB 추출 (프로젝트 고유 교훈)
  → Areas 갱신 (도메인별 현재 상태)
  → PRD 아카이브 (specs/ → archive/)
```

## Step 0: 입력 확인

1. **PRD 파일 경로** — `docs/superpowers/specs/*-prd.md` 중 가장 최근, 또는 사용자 지정
2. **git diff 범위** — Plan 파일 생성 커밋부터 현재 HEAD까지. Plan이 없으면 사용자에게 범위 요청
3. PRD와 diff가 모두 있으면 다음 단계로

## Step 1.5: 역PRD 열 기입 — 증거 링크

서브에이전트가 반환한 역PRD의 각 항목을 **원본 PRD 파일의 역PRD 열에 증거 링크로 기입**한다. ✅/❌ 자기 선언이 아니라 **추적 가능한 증거**를 남긴다.

### ② 산출물 — `파일명::export명`

1. 역PRD의 산출물과 원본 PRD ②의 행을 매칭한다
2. 매칭되면 실제 코드의 `파일명::export명`을 역PRD 열에 기입한다
3. 코드가 합성(middleware/config)으로 여러 행을 압축하면 N:1로 같은 export를 가리켜도 된다
4. 매칭되지 않는 PRD 행 → `❌ 미구현`
5. 매칭되지 않는 역PRD 행(PRD에 없는 것이 구현됨) → ⚠️ 표시와 함께 표 아래에 추가

```markdown
| 산출물 | 설명 | 역PRD |
|--------|------|-------|
| `tab.ts` | tab 축, 4개 전략 | ✅ `tab.ts::tab` |
| `spatial.ts` 수정 | ... | ❌ 미구현 |
```

### ⑧ 검증 — `파일명::테스트이름`

1. 역PRD의 검증 시나리오와 원본 PRD ⑧의 V-행을 매칭한다
2. 매칭되면 실제 테스트의 `파일명::it 블록 이름`을 역PRD 열에 기입한다
3. 테스트가 없는 V-행 → `❌ 테스트 없음`
4. 테스트는 있지만 `// V{n}:` 주석이 없으면 → 증거 링크 기입 후, 주석 누락을 **L1 코드 갭**으로 분류

```markdown
| # | 시나리오 | 예상 결과 | 역PRD |
|---|---------|----------|-------|
| V1 | Tab → 다음 노드 | DFS 이동 | ✅ `cms-tab-flow.integration.test.tsx::Tab moves focus in DFS order` |
| V3 | 마지막 노드 Tab | zone 탈출 | ❌ 테스트 없음 |
```

### ①③④⑤⑥⑦ — 기존과 동일

자유 텍스트로 매칭 결과를 기입한다 (✅ 일치 / ❌ 미구현 / ⚠️ 추가 구현 / 🔀 다르게 구현).

### 양방향 링크 검증

역PRD 열을 기입한 후, 코드/테스트에 양방향 주석이 있는지 확인한다:
- 코드: `// ② PRD파일명` 주석이 해당 export에 있는지
- 테스트: `// V{n}: PRD파일명` 주석이 해당 it 블록에 있는지
- 누락된 주석은 **L1 코드 갭**으로 분류하여 보고서에 포함한다

## Step 2: PRD 항목별 Diff

원본 PRD와 역PRD를 항목별(①~⑧)로 비교한다. **역PRD 열이 채워진 PRD 파일 자체가 비교 결과**이며, 갭은 역PRD 열의 내용과 해당 행의 PRD 내용 차이에서 드러난다.

| 비교 결과 | 의미 |
|----------|------|
| ✅ 일치 | 의도대로 구현됨 |
| ❌ PRD에 있는데 역PRD에 없음 | 의도했지만 구현 안 됨 |
| ⚠️ 역PRD에 있는데 PRD에 없음 | 의도 안 했지만 구현됨 (PRD가 덜 촘촘했다는 신호) |
| 🔀 양쪽 다 있지만 내용이 다름 | 의도와 다르게 구현됨 |

## Step 3: 갭 5계층 분류

각 갭의 **근본 원인**을 분석하여 5계층으로 분류한다.

| 계층 | 갭 원인 | 개선 대상 | 처리 |
|------|---------|----------|------|
| **L1 코드** | 구현 누락, 버그, 잘못된 로직 | 코드 수정 | 즉시 수정 또는 `/backlog`에 저장 |
| **L2 PRD** | PRD 항목이 비어있었음, 체크리스트 누락 | PRD 스킬 보강 | **직접 수정** — 스킬 파일에 체크리스트 추가 |
| **L3 스킬** | Discussion/Plan/Go 스킬의 구조적 약점 | 스킬 파일 수정 | **직접 수정** — 해당 스킬 파일 편집 |
| **L4 지식** | memory/rules에 없는 원칙 | memory/rules 추가 | **직접 수정** — memory 파일 생성/업데이트 |
| **L5 사용자 지시** | 모호한 표현, 암묵적 기대 | 사용자에게 피드백 | 보고서 → 사용자 읽기 |

**분류 기준:**
- "코드에서 뭘 안 했는가?" → L1
- "PRD에 왜 안 적혀 있었는가?" → L2
- "PRD에 적혀야 하는 걸 스킬이 왜 안 물어봤는가?" → L3
- "이런 원칙이 있었다면 잡혔을 텐데?" → L4
- "사용자가 이걸 명시했다면 잡혔을 텐데?" → L5

## Step 4.5: L2~L4 직접 수정

보고서를 생성한 후, L2~L4 갭이 있으면 **보고서 작성과 동시에 해당 파일을 직접 수정**한다.

| 계층 | 수정 대상 | 수정 방법 |
|------|----------|----------|
| L2 | `.claude/skills/prd/SKILL.md` | 누락된 체크리스트 항목 추가 |
| L3 | `.claude/skills/{스킬명}/SKILL.md` | 구조적 약점 보완 |
| L4 | `.claude/projects/*/memory/*.md` | 새 원칙을 memory 파일로 생성, MEMORY.md 인덱스 업데이트 |

수정한 파일 경로를 보고서의 "수정 파일" 필드에 기록한다.

## Step 6: 경험 추출

갭 분석이 끝나면, 이번 retro에서 **프로젝트 고유 경험**을 추출하여 경험 DB에 저장한다.

### 프로젝트 고유 판정 (리트머스)

교훈에 이 프로젝트의 **파일명, 타입명, 아키텍처 패턴**이 들어가면 프로젝트 고유 → 저장.
일반 코딩 지식("변수명은 의미 있게", "에러 핸들링 하라")은 LLM이 이미 알고 있으므로 저장하지 않는다.

예시:
- ✅ "behavior 파일에도 수동 keyMap 바인딩이 있을 수 있다" → behavior가 프로젝트 고유 → 저장
- ✅ "relationships만 바꾸면 expandedIds는 자동 보존된다" → NormalizedData 구조 고유 → 저장
- ❌ "타입을 명시하면 버그가 줄어든다" → 일반론 → 버림

### 저장 위치 및 포맷

`memory/experience_db.md`의 테이블에 행을 추가한다:

```markdown
| # | 도메인 | 상황 | 갭 | 교훈 | 빈도 | 최근 |
|---|--------|------|-----|------|------|------|
| N | plugin/keyMap | 수동 바인딩을 plugin.keyMap으로 교체 | behavior의 바인딩 누락 | behavior도 grep 범위에 포함 | 1 | 2026-03-21 |
```

### 패턴 매칭

새 경험을 추가하기 전에 기존 테이블을 확인한다:
- **같은 교훈이 이미 있으면** → 빈도 +1, 최근 날짜 갱신
- **새로운 교훈이면** → 새 행 추가

### 빈도 기반 승격

빈도가 **3회 이상**이면:
1. 해당 교훈을 `type: feedback` memory 파일로 승격한다 (Why + How to apply 포맷)
2. 경험 DB의 해당 행에 "→ feedback 승격됨" 표시 (행은 이력으로 유지)
3. MEMORY.md 인덱스에 새 feedback 파일 추가

승격은 "삽질이 규칙이 될 만큼 반복됐다"는 신호이므로, AI가 매 세션에서 자동으로 읽는 feedback으로 올리는 것.

## Step 7: Areas 갱신

역PRD(코드에서 역추출한 현재 상태)를 기반으로 **도메인별 living document**를 갱신한다. Areas는 "이 도메인이 지금 어떻게 동작하는가"를 담는 영구 문서이며, PRD(시간축)와 달리 도메인축으로 조직된다.

### 저장 위치

`docs/2-areas/{도메인}.md`

### 도메인 판정

이번 PRD/역PRD가 다루는 **아키텍처 모듈**을 도메인으로 판정한다. 하나의 PRD가 여러 도메인에 걸칠 수 있으며, 해당되는 Areas를 각각 갱신한다.

### 갱신 흐름

1. 이번 PRD의 도메인 판정 (예: plugin-system, focus-management, ...)
2. `docs/2-areas/{도메인}.md` 존재 확인
   - **없으면** → 역PRD 기반으로 초안 생성
   - **있으면** → 역PRD와 비교하여 변경된 부분만 갱신
3. "최종 갱신" 날짜 + retro 출처 기록

### Areas 문서 포맷

```markdown
# [도메인 이름]

> 최종 갱신: YYYY-MM-DD (retro: [feature명])

## 현재 구조

[이 도메인의 핵심 타입, 데이터 흐름, 합성 규칙 등]

## 핵심 규칙

[이 도메인에서 지켜야 하는 불변 조건, 우선순위, 제약]

## 관련 경험

[experience_db에서 이 도메인에 해당하는 행 참조]
```

## Step 8: PRD 아카이브

retro가 완료된 PRD를 `specs/archive/`로 이동하여 `specs/`에는 진행 중인 PRD만 남긴다.

```bash
git mv docs/superpowers/specs/YYYY-MM-DD-feature-prd.md docs/superpowers/specs/archive/
```

PRD는 히스토리 문서로 보존된다 — 갱신하지 않는다. 현재 상태가 궁금하면 Areas를 본다.

## 종료 조건

- 보고서 생성 완료
- L2~L4 직접 수정 완료 (해당 시)
- L1 항목 → 즉시 수정 또는 `/backlog`에 저장
- **경험 DB 업데이트 완료** (프로젝트 고유 경험 추출)
- **Areas 문서 갱신 완료** (도메인별 living document)
- **PRD 아카이브 완료** (specs/ → specs/archive/)
- 사용자에게 요약 제시 (수정 내역 + 경험 추가/승격 + Areas 갱신 + 아카이브 내역 포함)

⛔ **금지:** L5(사용자 피드백)는 자동 반영하지 않는다 — 보고서에 기록만 한다.
