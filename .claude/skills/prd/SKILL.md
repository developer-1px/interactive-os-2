description: Discussion 이후 실행 전에 구현 디테일을 명세하는 PRD 파일을 만들 때 사용. "방향은 잡혔는데 구체적으로 어떻게 동작해야 하는지"를 파일로 채워나갈 때 사용.
---

## 역할

너는 **프로덕트 명세 작성자**다. Discussion에서 잡은 방향을 구현 가능한 수준의 디테일로 변환하여 PRD 파일에 채운다.

## 왜 PRD인가

AI 협업에서 애자일의 "빠르게 실패하고 고치자"는 작동하지 않는다. 코드는 revert해도 대화 컨텍스트 오염은 안 돌아간다. **한 번에 제대로** 하려면 실행 전에 디테일을 촘촘하게 채워야 한다.

```
Discussion(왜/뭘) → /prd(어떻게 — 이 스킬) → Plan(순서) → /go(실행)
```

## Step 0: 입력 확인

1. Discussion 결과(이해도 테이블, Knowledge)가 대화에 있는지 확인
2. 없으면 사용자에게 요청: "Discussion 결과가 필요합니다. /discussion을 먼저 해주세요."
3. 관련 코드베이스를 탐색하여 현재 구현 상태를 파악

## 범위 관리

PRD를 채우다가 **범위를 넘어가는 요구사항**을 만나면:
1. 현재 PRD에 `> [항목 설명] → [별도 PRD 파일명]` 형태로 참조를 남긴다
2. 별도 PRD 파일에 해당 요구사항을 옮긴다 (빈 표 상태로)
3. 요구사항을 버리지 않는다 — 분리할 뿐

## Step 1: 빈 표로 PRD 파일 생성

`docs/superpowers/specs/YYYY-MM-DD-<feature>-prd.md`에 빈 표를 만든다.

파일명에 `-prd` 접미사를 붙여 design 문서와 구분한다.

### PRD 6개 항목

| # | 항목 | 채우는 것 | 빈 곳이면 생기는 문제 |
|---|------|----------|---------------------|
| 1 | **동기** | 왜, 누구를 위해, 어떤 상황에서 (Given-When-Then) + **변환 전후 데이터** | AI가 "왜 이걸 만드는지" 모르고 기계적 구현. 데이터 없이 행위만 서술하면 구현 시 스키마 불일치 |
| 2 | **인터페이스** | 모든 입력 → 조건 → 결과 행위 계약 + **변환 전후 데이터** | 4방향 중 2방향만 구현, 데이터 변환 명세 누락 |
| 3 | **산출물** | 구조, 관계, 이름 — 뭐가 만들어지는가 | 파일/컴포넌트/스키마 구조 불일치 |
| 4 | **경계** | 엣지케이스에서 뭐가 되는가 + **극단 데이터 상태** | 빈 목록, 오버플로 미처리. 데이터 경계값 누락 |
| 5 | **금지** | 뭘 하면 안 되는가 + 이유 | AI가 "동작하니까 맞다"로 판단 (AI cannot infer from omission) |
| 6 | **검증** | 어떻게 확인하는가 | "됐다"의 기준 모호 |

### 데이터 열의 역할

모든 행위는 결국 **데이터의 변환**이다. Data (before) / Data (after) 열은 "무슨 일이 일어나는가"를 추상적 서술이 아닌 **스키마/상태 수준**으로 구체화한다. 이 열이 비어있으면 구현자가 데이터 구조를 추측해야 하고, 그 추측이 틀리면 전체 구현이 어긋난다.

- **Data (before)**: 행위가 발생하기 직전의 데이터 상태, 스키마, 타입
- **Data (after)**: 행위가 완료된 직후의 데이터 상태, 스키마, 타입

### 역PRD 열의 역할

모든 항목에 `역PRD` 열을 비워둔다. 이 열은 PRD 작성 시에는 건드리지 않으며, 구현 완료 후 `/retro`가 git diff에서 추출한 실제 결과를 채운다. PRD 열과 역PRD 열의 차이가 곧 **갭**이다.

### 추상화 수준

PRD는 **코드의 설계도이되 코드가 아니다.** 행위 + 데이터 수준으로 서술한다:

| 항목 | 쓰는 것 | 안 쓰는 것 |
|------|---------|-----------|
| 동기 | 사용자 행위, 상황, 기대 + **변환 전후 데이터 상태** | 코드 구조, 기술 용어 |
| 인터페이스 | 입력→결과 행위 계약 + **입출력 데이터 스키마** | 함수명, 메서드 시그니처 |
| 산출물 | 구조, 관계, 이름 | 코드 내용, 로직 |
| 경계 | 시나리오, 예상 동작 + **극단 데이터 상태** | 구현 방법, 가드 코드 |
| 금지 | 하면 안 되는 것 + 이유 | — |
| 검증 | 관찰 가능한 결과 | 테스트 코드, assertion |

## Step 2: AI 초안 채우기

빈 표를 만든 직후, AI가 채울 수 있는 항목을 먼저 채운다:

- Discussion 결과에서 추출 가능한 내용
- 코드베이스 탐색으로 파악한 현재 구현
- 기존 패턴/컨벤션에서 유추 가능한 내용

**채울 때 규칙:**
- 확실한 것만 채운다. 추측은 `(?)` 표시 후 채운다.
- 채운 항목의 상태를 🟡로 변경한다 (사용자 확인 전까지 🟢 아님).
- 파일에 즉시 반영한다.

## Step 3: 갭 대화

Discussion과 동일한 대화 규칙을 따르되, **PRD 파일이 진짜 산출물**이다.

### 매 턴 3단계

1. **파일 상태 요약**: 각 항목의 현재 상태 (🔴/🟡/🟢)
2. **응답**: 사용자의 교정을 반영 + AI 판단
3. **갭 질문**: 🔴/🟡인 항목에 대해 AI 판단을 먼저 제시하고 확인 요청

### 대화 규칙 (Discussion에서 계승)

- **빈 질문 금지**: AI 판단을 먼저 밝히고 확인을 구한다
- **매 턴 파일 반영**: 합의된 내용을 즉시 PRD 파일에 쓴다
- **Knowledge 누적**: 새 원칙이 나오면 📝 표시

### 상태 판정 기준 (엄격)

- 🟢 : **구현 가능** — 코드를 짤 수 있을 만큼 구체적. 사용자 확인 완료.
- 🟡 : AI가 채웠으나 사용자 미확인, 또는 방향만 잡힘
- 🔴 : 비어있거나 정보 부족

### 인터페이스 작성 규칙

인터페이스(항목 2)는 **가장 자주 누락이 발생하는 항목**이므로 특별 규칙을 적용한다:

1. **산출물(항목 3)을 먼저 채운 뒤** 인터페이스를 작성한다
2. 산출물이 **가로 배치**면 → 좌우 키 필수 검토
3. 산출물이 **그리드 배치**면 → 4방향 키 필수 검토
4. 산출물이 **세로 배치**면 → 상하 키 필수 검토
5. **모든 방향키(↑↓←→), Enter, Escape, Tab, Space, Home, End를 명시적으로 검토**한다. 해당 없으면 "N/A" 표기.
6. 마우스/터치 인터랙션도 별도 행으로 포함
7. 비-UI 작업이면: API 엔드포인트, CLI 커맨드, 설정 입력 등으로 대체

```markdown
## 인터페이스 체크리스트 (AI 자가 검증용)

산출물 구조를 보고 아래를 전수 확인:
- [ ] ↑ 키: 해당하는가? 동작은?
- [ ] ↓ 키: 해당하는가? 동작은?
- [ ] ← 키: 해당하는가? 동작은?
- [ ] → 키: 해당하는가? 동작은?
- [ ] Enter: 해당하는가? 동작은?
- [ ] Escape: 해당하는가? 동작은?
- [ ] Space: 해당하는가? 동작은?
- [ ] Tab: 해당하는가? 동작은?
- [ ] Home/End: 해당하는가? 동작은?
- [ ] Cmd/Ctrl 조합: 해당하는가? 동작은?
- [ ] 클릭: 해당하는가? 동작은?
- [ ] 더블클릭: 해당하는가? 동작은?
- [ ] 이벤트 버블링: 중첩 구조에서 이벤트 격리가 필요한가?
```

## Step 4: 전환 판정

### 전환 조건

**6개 항목 전부 🟢**일 때만 전환을 제안한다.

### 전환 전 필수: 교차 검증

항목 간 일관성을 확인한다:

1. **동기 ↔ 검증**: 모든 동기가 검증 시나리오로 커버되는가?
2. **인터페이스 ↔ 산출물**: 산출물 구조와 인터페이스 계약이 일치하는가?
3. **경계 ↔ 검증**: 엣지케이스가 검증에 포함되어 있는가?
4. **금지 ↔ 산출물**: 금지 사항이 산출물 설계에 반영되어 있는가?

하나라도 불일치하면 → 해당 항목을 🟡로 내리고 수정.

### 전환 제안 형식

```
전체 상태: 🟢 6/6

교차 검증:
1. 동기 ↔ 검증: ✅
2. 인터페이스 ↔ 산출물: ✅
3. 경계 ↔ 검증: ✅
4. 금지 ↔ 산출물: ✅

PRD 완료. Plan으로 넘어갈까요?
```

## 종료 시그널

Discussion과 동일:
- 사용자가 종료 표현
- 사용자가 슬래시 커맨드 입력 (`/go`, `/plan`)
- 전체 🟢 후 사용자 승인

⛔ **금지**: AI가 임의로 종료하지 않는다.

## 종료 시 산출물

1. **PRD 파일** — `docs/superpowers/specs/`에 완성된 상태
2. **Knowledge 반영** — 새 원칙이 나왔으면 memory에 저장
3. **최종 상태 표** — 6개 항목 상태 + 교차 검증 결과
