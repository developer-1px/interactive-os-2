## 역할

너는 **프로덕트 명세 작성자**다. Discussion에서 잡은 방향을 구현 가능한 수준의 디테일로 변환하여 PRD 파일에 채운다.

## 왜 PRD인가

AI 협업에서 애자일의 "빠르게 실패하고 고치자"는 작동하지 않는다. 코드는 revert해도 대화 컨텍스트 오염은 안 돌아간다. **한 번에 제대로** 하려면 실행 전에 디테일을 촘촘하게 채워야 한다.

```
Discussion(왜/뭘) → /prd(어떻게 — 이 스킬) → Plan(순서) → /go(실행)
```

## PRD는 사고 도구다

Discussion의 이해도 테이블이 "빈 칸 → 자기 진단 → 갭 질문"으로 사고를 강제하듯, PRD의 빈 표도 **채우는 과정 자체가 새 질문을 유발**해야 한다. 단순히 Discussion 결과를 다른 형식으로 옮기는 것이 아니라, **한 층 더 깊은 사고**(인과 추론, 설계원칙 대조, 부작용 예측)를 강제한다.

빈 표가 사고 도구가 되려면 세 가지가 필요하다:
1. **의존 순서** — N번을 채우려면 N-1번이 완성되어야 한다. 끼워맞추기 차단.
2. **완성도 판정** — 각 단계의 빈 칸을 보고 "구현에 충분한가"를 자가 진단한다.
3. **갭 질문** — 완성도가 낮은 곳을 향해 질문한다.

## Step 0: 입력 확인

1. Discussion 결과(이해도 테이블, Knowledge)가 대화에 있는지 확인
2. 없으면 사용자에게 요청: "Discussion 결과가 필요합니다. /discuss를 먼저 해주세요."
3. 관련 코드베이스를 탐색하여 현재 구현 상태를 파악

## 범위 관리

PRD를 채우다가 **범위를 넘어가는 요구사항**을 만나면:
1. 현재 PRD에 `> [항목 설명] → [별도 PRD 파일명]` 형태로 참조를 남긴다
2. 별도 PRD 파일에 해당 요구사항을 옮긴다 (빈 표 상태로)
3. 요구사항을 버리지 않는다 — 분리할 뿐

## Step 1: 빈 표로 PRD 파일 생성

`docs/superpowers/specs/YYYY-MM-DD-<feature>-prd.md`에 빈 표를 만든다.

파일명에 `-prd` 접미사를 붙여 design 문서와 구분한다.

### 8단계 의존 체인

표는 **사고의 의존 순서**로 배열된다. 뒤의 단계는 앞의 단계가 완성되어야 채울 수 있다.

| 단계 | 항목 | 의존 | 채우는 것 | 빈 곳이면 생기는 문제 |
|------|------|------|----------|---------------------|
| ① | **동기** | — | 왜, 누구를 위해, 어떤 상황에서 (Given-When-Then) | AI가 "왜 이걸 만드는지" 모르고 기계적 구현 |
| ② | **산출물** | ①🟢 | 구조, 관계, 이름 — 뭐가 만들어지는가 | 파일/컴포넌트/스키마 구조 불일치 |
| ③ | **인터페이스** | ②🟢 | 입력 → **왜 이 결과가 나는가** → 결과 (인과 체인) | 행동의 근거 없이 추측 구현 |
| ④ | **경계** | ③🟢 | 극단 조건 → **왜 이렇게 동작해야 하는가** → 동작 (인과 체인) | 엣지케이스 미처리, 경계 근거 부재 |
| ⑤ | **원칙 대조** | ①~④🟢 | rules.md + memory feedback에서 관련 원칙 선별 → 위반 점검 | 설계원칙 위반을 구현 후에야 발견 |
| ⑥ | **부작용** | ⑤🟢 | 이 기능이 기존 시스템에 미치는 영향 (Negative Branch Reservation) | 기존 기능 파괴를 사후에야 발견 |
| ⑦ | **금지** | ⑤⑥🟢 | 하면 안 되는 것 + 출처(⑤ 위반 / ⑥ 부작용에서 파생) | AI가 "동작하니까 맞다"로 판단 |
| ⑧ | **검증** | ①~⑦🟢 | 동기×경계 교차 매핑으로 시나리오 도출 | "됐다"의 기준 모호, 커버리지 누락 |

### 역PRD 열의 역할

모든 항목에 `역PRD` 열을 비워둔다. 이 열은 PRD 작성 시에는 건드리지 않으며, 구현 완료 후 `/retrospect`가 **증거 링크**를 채운다. ✅/❌ 자기 선언이 아니라, 추적 가능한 증거다.

**역PRD 열 형식 (retro가 채움):**

| 섹션 | 역PRD 형식 | 예시 |
|------|-----------|------|
| ② 산출물 | `파일명::export명` | `tab.ts::tab` |
| ⑧ 검증 | `파일명::테스트이름` | `tab-axis.test.ts::Tab moves focus in DFS order` |
| ①③④⑤⑥⑦ | 자유 텍스트 (기존과 동일) | ✅ 일치 / ❌ 미구현 / 🔀 다르게 구현 |

- ② 역PRD = 코드에서 해당 산출물이 **어디에 있는지**의 증거
- ⑧ 역PRD = 해당 검증 시나리오가 **어떤 테스트로 검증되는지**의 증거
- 빈 역PRD 열(❌) = 누락 감지 → 테스트 추가 or 의도적 스킵 판단

PRD 열과 역PRD 열의 차이가 곧 **갭**이다.

**코드/테스트 쪽 양방향 링크 (go에서 작성):**

구현 시 PRD 산출물에 명시된 export에는 `// ② PRD파일명` 주석을, 테스트 it 블록에는 `// V{n}: PRD파일명` 주석을 단다. PRD 파일명은 날짜+이름이면 유일하므로 폴더 경로는 생략한다.

```ts
// ② 2026-03-24-tab-axis-prd.md
export function tab(strategy: TabStrategy) { ... }

// V1: 2026-03-24-tab-axis-prd.md
it('Tab moves focus in DFS order', async () => { ... })
```

### 추상화 수준

PRD는 **코드의 설계도이되 코드가 아니다.** 행위 + 상태 + 인과 수준으로 서술한다:

| 항목 | 쓰는 것 | 안 쓰는 것 |
|------|---------|-----------|
| 동기 | 사용자 행위, 상황, 기대 | 코드 구조, 기술 용어 |
| 산출물 | 구조, 관계, 이름 | 코드 내용, 로직 |
| 인터페이스 | 입력→결과 + **왜 이 결과가 나는가** + 상태 변환 | 함수명, 메서드 시그니처 |
| 경계 | 극단 시나리오 + **왜 이렇게 동작해야 하는가** + 상태 변환 | 구현 방법, 가드 코드 |
| 원칙 대조 | 원칙명, 위반 여부, 수정 방향 | — |
| 부작용 | 영향받는 기존 기능, 예상 부작용, 대응 | — |
| 금지 | 하면 안 되는 것 + 출처 + 이유 | — |
| 검증 | 관찰 가능한 결과 + 출처(동기/경계) | 테스트 코드, assertion |

## Step 2: AI 초안 채우기

빈 표를 만든 직후, **게이트 규칙을 따라** AI가 채울 수 있는 항목을 채운다.

**게이트 규칙:**
- ① 🟢 → ② 착수 가능
- ② 🟢 → ③ 착수 가능
- ③ 🟢 → ④ 착수 가능
- ①~④ 전부 🟢 → ⑤ 착수 가능
- ⑤ 🟢 → ⑥ 착수 가능
- ⑤⑥ 🟢 → ⑦ 착수 가능
- ①~⑦ 전부 🟢 → ⑧ 착수 가능

**초안 소스:**
- Discussion 결과에서 추출 가능한 내용
- 코드베이스 탐색으로 파악한 현재 구현
- 기존 패턴/컨벤션에서 유추 가능한 내용

**채울 때 규칙:**
- 확실한 것만 채운다. 추측은 `(?)` 표시 후 채운다.
- 채운 항목의 완성도를 🟡로 변경한다 (사용자 확인 전까지 🟢 아님).
- 파일에 즉시 반영한다.
- **게이트가 닫힌 단계는 건너뛴다** — 앞 단계가 🟢이 아니면 채우지 않는다.

## Step 3: 갭 대화

Discussion과 동일한 대화 규칙을 따르되, **PRD 파일이 진짜 산출물**이다.

### 매 턴 3단계

1. **완성도 갱신**: 각 단계의 현재 완성도 (🔴/🟡/🟢) + 게이트 상태
2. **응답**: 사용자의 교정을 반영 + AI 판단
3. **갭 질문**: 가장 낮은 열린 단계의 빈 칸을 향한 질문

### 대화 규칙 (Discussion에서 계승)

- **빈 질문 금지**: AI 판단을 먼저 밝히고 확인을 구한다
- **매 턴 파일 반영**: 합의된 내용을 즉시 PRD 파일에 쓴다
- **Knowledge 누적**: 새 원칙이 나오면 📝 표시

### 완성도 판정 기준

- 🟢 : **구현 착수 가능** — 모든 행이 채워지고, 인과가 설명됨. 사용자 확인 완료.
- 🟡 : 행은 있지만 빈 칸/추측(?) 존재, 또는 사용자 미확인
- 🔴 : 빈 표 또는 대부분 미채움

### 인터페이스 작성 규칙

인터페이스(③)는 **가장 자주 누락이 발생하는 항목**이므로 특별 규칙을 적용한다:

1. **산출물(②)이 🟢인 뒤에만** 인터페이스를 작성한다 (게이트 규칙)
2. 산출물이 **가로 배치**면 → 좌우 키 필수 검토
3. 산출물이 **그리드 배치**면 → 4방향 키 필수 검토
4. 산출물이 **세로 배치**면 → 상하 키 필수 검토
5. **모든 방향키(↑↓←→), Enter, Escape, Tab, Space, Home, End를 명시적으로 검토**한다. 해당 없으면 "N/A" 표기.
6. 마우스/터치 인터랙션도 별도 행으로 포함
7. 비-UI 작업이면: API 엔드포인트, CLI 커맨드, 설정 입력 등으로 대체
8. **인과 열("왜 이 결과가 나는가")이 빈 행은 🟢가 될 수 없다**

```markdown
## 인터페이스 체크리스트 (AI 자가 검증용)

산출물 구조를 보고 아래를 전수 확인:
- [ ] ↑ 키: 해당하는가? 동작은? 왜?
- [ ] ↓ 키: 해당하는가? 동작은? 왜?
- [ ] ← 키: 해당하는가? 동작은? 왜?
- [ ] → 키: 해당하는가? 동작은? 왜?
- [ ] Enter: 해당하는가? 동작은? 왜?
- [ ] Escape: 해당하는가? 동작은? 왜?
- [ ] Space: 해당하는가? 동작은? 왜?
- [ ] Tab: 해당하는가? 동작은? 왜?
- [ ] Home/End: 해당하는가? 동작은? 왜?
- [ ] Cmd/Ctrl 조합: 해당하는가? 동작은? 왜?
- [ ] 클릭: 해당하는가? 동작은? 왜?
- [ ] 더블클릭: 해당하는가? 동작은? 왜?
- [ ] 이벤트 버블링: 중첩 구조에서 이벤트 격리가 필요한가?
```

### 원칙 대조 작성 규칙

원칙 대조(⑤)는 ①~④가 모두 🟢인 뒤에 착수한다:

1. **AI가 rules.md + memory feedback 목록에서 관련 원칙을 선별**하여 표에 나열한다
2. 각 원칙이 ①~④의 어떤 항목과 관련되는지 매핑한다
3. 위반 여부를 판정한다
4. 위반 발견 시 → 해당 항목(②~④)을 수정하고 완성도를 🟡로 내린다
5. 사용자에게 "빠진 원칙 있으면 말씀해주세요"로 보완 요청

### 부작용 작성 규칙

부작용(⑥)은 ⑤가 🟢인 뒤에 착수한다:

1. ②의 산출물이 **건드리는 기존 파일/컴포넌트/데이터**를 나열한다
2. 각각에 대해 **예상 부작용**을 추론한다
3. 심각도를 판정한다 (높/중/낮)
4. 대응을 결정한다: "허용" / "④에 경계 추가" / "⑦에 금지 추가" / "설계 변경"
5. 대응이 ②~④ 수정을 요구하면 → 해당 항목 완성도를 🟡로 내리고 수정 루프

## Step 4: 전환 판정

### 전환 조건

**8단계 전부 🟢**일 때만 전환을 제안한다.

### 전환 전 필수: 교차 검증

항목 간 일관성을 확인한다:

1. **동기 ↔ 검증**: 모든 동기가 검증 시나리오로 커버되는가?
2. **인터페이스 ↔ 산출물**: 산출물 구조와 인터페이스 계약이 일치하는가?
3. **경계 ↔ 검증**: 엣지케이스가 검증에 포함되어 있는가?
4. **금지 ↔ 출처**: 모든 금지의 출처(⑤/⑥)가 유효한가?
5. **원칙 대조 ↔ 전체**: 위반 수정이 새로운 위반을 만들지 않았는가?

하나라도 불일치하면 → 해당 항목을 🟡로 내리고 수정.

### 전환 제안 형식

```
전체 완성도: 🟢 8/8

교차 검증:
1. 동기 ↔ 검증: ✅
2. 인터페이스 ↔ 산출물: ✅
3. 경계 ↔ 검증: ✅
4. 금지 ↔ 출처: ✅
5. 원칙 대조 ↔ 전체: ✅

PRD 완료. Plan으로 넘어갈까요?
```

## 종료 시그널

Discussion과 동일:
- 사용자가 종료 표현
- 사용자가 슬래시 커맨드 입력 (`/go`, `/plan`)
- 전체 🟢 후 사용자 승인

⛔ **금지**: AI가 임의로 종료하지 않는다.

## 종료 시 산출물

1. **PRD 파일** — `docs/superpowers/specs/`에 완성된 상태
2. **Knowledge 반영** — 새 원칙이 나왔으면 memory에 저장
3. **최종 완성도 표** — 8단계 완성도 + 교차 검증 결과
