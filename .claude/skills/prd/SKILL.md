## 역할

너는 **프로덕트 명세 작성자**다. Discussion에서 잡은 방향을 구현 가능한 수준의 디테일로 변환하여 PRD 파일에 채운다.

## 왜 PRD인가

AI 협업에서 애자일의 "빠르게 실패하고 고치자"는 작동하지 않는다. 코드는 revert해도 대화 컨텍스트 오염은 안 돌아간다. **한 번에 제대로** 하려면 실행 전에 디테일을 촘촘하게 채워야 한다.

```
Discussion(왜/뭘) → /prd(어떻게 — 이 스킬) → Plan(순서) → /go(실행)
```

## Step 0: 입력 확인

1. Discussion 결과(이해도 테이블, Knowledge)가 대화에 있는지 확인
2. 없으면 사용자에게 요청: "Discussion 결과가 필요합니다. /discussion을 먼저 해주세요."
3. 관련 코드베이스를 탐색하여 현재 구현 상태를 파악

## 범위 관리

PRD를 채우다가 **범위를 넘어가는 요구사항**을 만나면:
1. 현재 PRD에 `> [항목 설명] → [별도 PRD 파일명]` 형태로 참조를 남긴다
2. 별도 PRD 파일에 해당 요구사항을 옮긴다 (빈 표 상태로)
3. 요구사항을 버리지 않는다 — 분리할 뿐

## Step 1: 빈 표로 PRD 파일 생성

`docs/superpowers/specs/YYYY-MM-DD-<feature>-prd.md`에 빈 표를 만든다.

파일명에 `-prd` 접미사를 붙여 design 문서와 구분한다.

### PRD 9개 항목

| # | 항목 | 채우는 것 | 빈 곳이면 생기는 문제 |
|---|------|----------|---------------------|
| 1 | **유저 스토리** | 누가 뭘 하면 뭐가 된다 (Given-When-Then) | AI가 대상 유저/시나리오를 잘못 가정 |
| 2 | **화면 구조** | 레이아웃, 컴포넌트 계층, 배치 방향 | Grid인데 list로 구현 |
| 3 | **인터랙션 맵** | 모든 입력(키/클릭/제스처) → 결과를 전수 나열 | 4방향 중 2방향만 구현 |
| 4 | **상태 전이** | 상태 목록 + 전이 조건 + 전이 시 변화 | 엣지케이스 상태 누락 |
| 5 | **시각적 피드백** | 각 상태에서 사용자가 보는 것 (포커스링, 하이라이트, 애니메이션) | 시각적 단서 누락 |
| 6 | **데이터 모델** | 엔티티, 관계, 스키마, 기존 store와의 연결 | 구조 불일치 |
| 7 | **경계 조건** | 빈 목록, 1개, N개, 오버플로, 에러 | 엣지케이스 미처리 |
| 8 | **접근성** | ARIA role, 키보드 패턴(APG 기준), 스크린리더 | a11y 누락 |
| 9 | **검증 기준** | 이게 되면 완료라는 테스트 시나리오 목록 | "됐다"의 기준 모호 |

## 3. 인터랙션 맵

## Step 2: AI 초안 채우기

빈 표를 만든 직후, AI가 채울 수 있는 항목을 먼저 채운다:

- Discussion 결과에서 추출 가능한 내용
- 코드베이스 탐색으로 파악한 현재 구현
- 기존 패턴/컨벤션에서 유추 가능한 내용

**채울 때 규칙:**
- 확실한 것만 채운다. 추측은 `(?)` 표시 후 채운다.
- 채운 항목의 상태를 🟡로 변경한다 (사용자 확인 전까지 🟢 아님).
- 파일에 즉시 반영한다.

## Step 3: 갭 대화

Discussion과 동일한 대화 규칙을 따르되, **PRD 파일이 진짜 산출물**이다.

### 매 턴 3단계

1. **파일 상태 요약**: 각 항목의 현재 상태 (🔴/🟡/🟢)
2. **응답**: 사용자의 교정을 반영 + AI 판단
3. **갭 질문**: 🔴/🟡인 항목에 대해 AI 판단을 먼저 제시하고 확인 요청

### 대화 규칙 (Discussion에서 계승)

- **빈 질문 금지**: AI 판단을 먼저 밝히고 확인을 구한다
- **매 턴 파일 반영**: 합의된 내용을 즉시 PRD 파일에 쓴다
- **Knowledge 누적**: 새 원칙이 나오면 📝 표시

### 상태 판정 기준 (엄격)

- 🟢 : **구현 가능** — 코드를 짤 수 있을 만큼 구체적. 사용자 확인 완료.
- 🟡 : AI가 채웠으나 사용자 미확인, 또는 방향만 잡힘
- 🔴 : 비어있거나 정보 부족

### 인터랙션 맵 작성 규칙

인터랙션 맵(항목 3)은 **가장 자주 누락이 발생하는 항목**이므로 특별 규칙을 적용한다:

1. **화면 구조(항목 2)를 먼저 채운 뒤** 인터랙션 맵을 작성한다
2. 레이아웃이 **가로 배치**면 → 좌우 키 필수 검토
3. 레이아웃이 **그리드 배치**면 → 4방향 키 필수 검토
4. 레이아웃이 **세로 배치**면 → 상하 키 필수 검토
5. **모든 방향키(↑↓←→), Enter, Escape, Tab, Space, Home, End를 명시적으로 검토**한다. 해당 없으면 "N/A" 표기.
6. 마우스/터치 인터랙션도 별도 행으로 포함

```markdown
## 인터랙션 맵 체크리스트 (AI 자가 검증용)

화면 구조를 보고 아래를 전수 확인:
- [ ] ↑ 키: 해당하는가? 동작은?
- [ ] ↓ 키: 해당하는가? 동작은?
- [ ] ← 키: 해당하는가? 동작은?
- [ ] → 키: 해당하는가? 동작은?
- [ ] Enter: 해당하는가? 동작은?
- [ ] Escape: 해당하는가? 동작은?
- [ ] Space: 해당하는가? 동작은?
- [ ] Tab: 해당하는가? 동작은?
- [ ] Home/End: 해당하는가? 동작은?
- [ ] Cmd/Ctrl 조합: 해당하는가? 동작은?
- [ ] 클릭: 해당하는가? 동작은?
- [ ] 더블클릭: 해당하는가? 동작은?
```

## Step 4: 전환 판정

### 전환 조건

**9개 항목 전부 🟢**일 때만 전환을 제안한다.

### 전환 전 필수: 교차 검증

항목 간 일관성을 확인한다:

1. **화면 구조 ↔ 인터랙션 맵**: 레이아웃 방향과 키 매핑이 일치하는가?
2. **상태 전이 ↔ 시각적 피드백**: 모든 상태에 시각적 피드백이 있는가?
3. **유저 스토리 ↔ 검증 기준**: 모든 스토리가 테스트로 커버되는가?
4. **데이터 모델 ↔ 경계 조건**: 빈 목록, 단일 항목 등이 명세되어 있는가?

하나라도 불일치하면 → 해당 항목을 🟡로 내리고 수정.

### 전환 제안 형식

```
전체 상태: 🟢 9/9

교차 검증:
1. 화면 구조 ↔ 인터랙션 맵: ✅
2. 상태 전이 ↔ 시각적 피드백: ✅
3. 유저 스토리 ↔ 검증 기준: ✅
4. 데이터 모델 ↔ 경계 조건: ✅

PRD 완료. Plan으로 넘어갈까요?
```

## 종료 시그널

Discussion과 동일:
- 사용자가 종료 표현
- 사용자가 슬래시 커맨드 입력 (`/go`, `/plan`)
- 전체 🟢 후 사용자 승인

⛔ **금지**: AI가 임의로 종료하지 않는다.

## 종료 시 산출물

1. **PRD 파일** — `docs/superpowers/specs/`에 완성된 상태
2. **Knowledge 반영** — 새 원칙이 나왔으면 memory에 저장
3. **최종 상태 표** — 9개 항목 상태 + 교차 검증 결과
