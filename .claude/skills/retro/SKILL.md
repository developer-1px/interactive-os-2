description: 구현 사이클 완료 후 회고. git diff에서 역PRD를 생성하고 원본 PRD와 비교하여 갭을 자동 감지한다. "/retro", "회고", "뭐가 빠졌지", "결과 점검" 등을 말할 때 사용.
---

## 역할

너는 **blind reviewer**다. 코드 변경만 보고 "이 코드가 뭘 하는가"를 역으로 정리한 뒤, 원래 의도(PRD)와 대조하여 갭을 찾는다.

## 왜 회고인가

AI 협업에서 테스트 통과 ≠ 의도 충족. 코드가 돌아가도 PRD의 의도와 다를 수 있다. 사용자가 안 짚은 문제는 영원히 모르기 때문에, AI가 자동으로 갭을 감지하는 메커니즘이 필요하다.

```
구현 완료 → /retro → 역PRD ↔ 원본PRD diff → 갭 보고서 → 백로그 → 처리
```

## Step 0: 입력 확인

1. **PRD 파일 경로** — `docs/superpowers/specs/*-prd.md` 중 가장 최근, 또는 사용자 지정
2. **git diff 범위** — Plan 파일 생성 커밋부터 현재 HEAD까지. Plan이 없으면 사용자에게 범위 요청
3. PRD와 diff가 모두 있으면 다음 단계로

## Step 1: Blind 역PRD 생성

**fresh subagent**를 디스패치한다. 이 에이전트에게는:
- ✅ git diff (소스 코드 변경분만)
- ✅ PRD의 9개 항목 템플릿 (빈 표)
- ❌ 원본 PRD 내용은 주지 않는다 (blind)

서브에이전트 프롬프트:

```
당신은 코드 리버스 엔지니어입니다. 아래 git diff를 읽고, 이 코드가 "무엇을 하는가"를 역으로 정리하세요.

아래 9개 항목의 빈 표를 채우세요. 코드에서 파악할 수 있는 것만 채우고, 파악할 수 없는 것은 "코드에서 파악 불가"로 표기하세요.

## 1. 유저 스토리
| # | Given | When | Then |
|---|-------|------|------|

## 2. 화면 구조
## 3. 인터랙션 맵
| 입력 | 현재 상태 | 결과 |
|------|----------|------|

## 4. 상태 전이
## 5. 시각적 피드백
## 6. 데이터 모델
## 7. 경계 조건
## 8. 접근성
## 9. 검증 기준

[git diff 내용]
```

서브에이전트가 역PRD를 반환하면 파일로 저장하지 않고 메모리에 보관한다.

## Step 2: PRD 항목별 Diff

원본 PRD와 역PRD를 항목별(1~9)로 비교한다.

| 비교 결과 | 의미 |
|----------|------|
| ✅ 일치 | 의도대로 구현됨 |
| ❌ PRD에 있는데 역PRD에 없음 | 의도했지만 구현 안 됨 |
| ⚠️ 역PRD에 있는데 PRD에 없음 | 의도 안 했지만 구현됨 (PRD가 덜 촘촘했다는 신호) |
| 🔀 양쪽 다 있지만 내용이 다름 | 의도와 다르게 구현됨 |

## Step 3: 갭 5계층 분류

각 갭의 **근본 원인**을 분석하여 5계층으로 분류한다.

| 계층 | 갭 원인 | 개선 대상 | 처리 |
|------|---------|----------|------|
| **L1 코드** | 구현 누락, 버그, 잘못된 로직 | 코드 수정 | 백로그 → 랄프루프 자율 |
| **L2 PRD** | PRD 항목이 비어있었음, 체크리스트 누락 | PRD 스킬 보강 제안 | 보고서 → 사용자 확인 |
| **L3 스킬** | Discussion/Plan/Go 스킬의 구조적 약점 | 스킬 파일 수정 제안 | 보고서 → 사용자 확인 |
| **L4 지식** | memory/rules에 없는 원칙 | memory/rules 추가 제안 | 보고서 → 사용자 확인 |
| **L5 사용자 지시** | 모호한 표현, 암묵적 기대 | 사용자에게 피드백 | 보고서 → 사용자 읽기 |

**분류 기준:**
- "코드에서 뭘 안 했는가?" → L1
- "PRD에 왜 안 적혀 있었는가?" → L2
- "PRD에 적혀야 하는 걸 스킬이 왜 안 물어봤는가?" → L3
- "이런 원칙이 있었다면 잡혔을 텐데?" → L4
- "사용자가 이걸 명시했다면 잡혔을 텐데?" → L5

## Step 4: 보고서 생성

`docs/0-inbox/`에 보고서 파일을 생성한다. 파일명: `{순번}-[retro]{feature명}.md`

### 보고서 템플릿

```markdown
# Retro: [feature명] — YYYY-MM-DD

## 비교 기준
- **PRD:** [경로]
- **Diff 범위:** [시작 커밋..HEAD]
- **커밋 수:** N
- **변경 파일:** N

## 항목별 비교

| # | PRD 항목 | 일치 | 갭 요약 | 계층 |
|---|---------|------|---------|------|
| 1 | 유저 스토리 | ✅/❌/⚠️/🔀 | ... | L? |
| 2 | 화면 구조 | ... | ... | ... |
| ... | ... | ... | ... | ... |

**일치율:** N/9

## 갭 상세

### ❌ PRD에 있는데 구현 안 됨
- ...

### ⚠️ 구현됐는데 PRD에 없었음
- ...

### 🔀 의도와 다르게 구현됨
- ...

## 계층별 개선 제안

### L1 코드 — 백로그 (자율 처리 가능)
- [ ] [구체적 수정 내용]
- [ ] ...

### L2 PRD 스킬 — 제안 (확인 필요)
- PRD 스킬의 [항목]에 [체크리스트] 추가 권장
- 이유: [이번에 이것 때문에 누락됨]

### L3 스킬 — 제안 (확인 필요)
- [스킬명]의 [부분]을 [이렇게] 수정 권장
- 이유: ...

### L4 지식 — 제안 (확인 필요)
- memory에 "[원칙]" 추가 권장
- 이유: ...

### L5 사용자 피드백
- "💬 [상황 설명] — 다음에 [이렇게] 명시해주시면 더 정확한 결과를 낼 수 있습니다."
- ...

## 다음 행동
- L1 항목이 있으면 → 랄프루프로 처리 가능
- L2~L4 제안이 있으면 → 사용자 확인 후 반영
- L5 피드백이 있으면 → 사용자가 참고
```

## Step 5: 사용자에게 보고서 요약 제시

보고서를 생성한 후, 대화에 핵심 요약을 보여준다:

```
## Retro 결과

일치율: 7/9

### 주요 갭
- ❌ [항목]: [요약]
- ⚠️ [항목]: [요약]

### L1 백로그 (자율 처리 가능)
- [ ] ...

### 사용자 피드백
- 💬 ...

보고서: docs/0-inbox/N-[retro]feature.md
L1 백로그를 랄프루프로 처리할까요?
```

## 종료 조건

- 보고서 생성 완료
- 사용자에게 요약 제시
- 사용자가 백로그 처리를 승인하면 → 랄프루프 기동
- 사용자가 L2~L5 제안을 확인하면 → 해당 파일에 반영

⛔ **금지:** L2~L5 개선 사항을 사용자 확인 없이 자동 반영하지 않는다.
