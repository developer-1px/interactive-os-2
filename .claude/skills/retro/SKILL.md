## 역할

너는 **blind reviewer**다. 코드 변경만 보고 "이 코드가 뭘 하는가"를 역으로 정리한 뒤, 원래 의도(PRD)와 대조하여 갭을 찾는다.

## 왜 회고인가

AI 협업에서 테스트 통과 ≠ 의도 충족. 코드가 돌아가도 PRD의 의도와 다를 수 있다. 사용자가 안 짚은 문제는 영원히 모르기 때문에, AI가 자동으로 갭을 감지하는 메커니즘이 필요하다.

```
구현 완료 → /retro → 역PRD ↔ 원본PRD diff → 갭 보고서 → 백로그 → 처리
```

## Step 0: 입력 확인

1. **PRD 파일 경로** — `docs/superpowers/specs/*-prd.md` 중 가장 최근, 또는 사용자 지정
2. **git diff 범위** — Plan 파일 생성 커밋부터 현재 HEAD까지. Plan이 없으면 사용자에게 범위 요청
3. PRD와 diff가 모두 있으면 다음 단계로

## Step 1: Blind 역PRD 생성

**fresh subagent**를 디스패치한다. 이 에이전트에게는:
- ✅ git diff (소스 코드 변경분만)
- ✅ PRD의 6개 항목 템플릿 (빈 표 — 원본 PRD와 동일 구조)
- ❌ 원본 PRD 내용은 주지 않는다 (blind)

서브에이전트 프롬프트:

```
당신은 코드 리버스 엔지니어입니다. 아래 git diff를 읽고, 이 코드가 "무엇을 하는가"를 역으로 정리하세요.

아래 6개 항목의 빈 표를 채우세요. 코드에서 파악할 수 있는 것만 채우고, 파악할 수 없는 것은 "코드에서 파악 불가"로 표기하세요.

## 1. 동기
| # | Given | Data (before) | When | Then | Data (after) |
|---|-------|--------------|------|------|-------------|

## 2. 인터페이스
| 입력 | Data (before) | 조건 | 결과 | Data (after) |
|------|--------------|------|------|-------------|

## 3. 산출물
| 산출물 | 설명 |
|--------|------|

## 4. 경계
| 조건 | Data (before) | 예상 동작 | Data (after) |
|------|--------------|----------|-------------|

## 5. 금지
| # | 하면 안 되는 것 | 이유 |
|---|---------------|------|

## 6. 검증
| # | 시나리오 | 예상 결과 |
|---|---------|----------|

[git diff 내용]
```

서브에이전트가 역PRD를 반환하면 메모리에 보관한다.

## Step 1.5: 역PRD 열 기입

서브에이전트가 반환한 역PRD의 각 항목을 **원본 PRD 파일의 역PRD 열에 직접 기입**한다.

- 원본 PRD의 각 행과 역PRD의 대응 행을 매칭하여 역PRD 열을 채운다
- 매칭되지 않는 역PRD 행(PRD에 없는 것이 구현됨)은 ⚠️ 표시와 함께 해당 항목 표 아래에 추가한다
- 매칭되지 않는 PRD 행(구현되지 않은 것)의 역PRD 열에는 "❌ 미구현"을 기입한다

## Step 2: PRD 항목별 Diff

원본 PRD와 역PRD를 항목별(1~6)로 비교한다. **역PRD 열이 채워진 PRD 파일 자체가 비교 결과**이며, 갭은 역PRD 열의 내용과 해당 행의 PRD 내용 차이에서 드러난다.

| 비교 결과 | 의미 |
|----------|------|
| ✅ 일치 | 의도대로 구현됨 |
| ❌ PRD에 있는데 역PRD에 없음 | 의도했지만 구현 안 됨 |
| ⚠️ 역PRD에 있는데 PRD에 없음 | 의도 안 했지만 구현됨 (PRD가 덜 촘촘했다는 신호) |
| 🔀 양쪽 다 있지만 내용이 다름 | 의도와 다르게 구현됨 |

## Step 3: 갭 5계층 분류

각 갭의 **근본 원인**을 분석하여 5계층으로 분류한다.

| 계층 | 갭 원인 | 개선 대상 | 처리 |
|------|---------|----------|------|
| **L1 코드** | 구현 누락, 버그, 잘못된 로직 | 코드 수정 | 즉시 수정 또는 `/backlog`에 저장 |
| **L2 PRD** | PRD 항목이 비어있었음, 체크리스트 누락 | PRD 스킬 보강 | **직접 수정** — 스킬 파일에 체크리스트 추가 |
| **L3 스킬** | Discussion/Plan/Go 스킬의 구조적 약점 | 스킬 파일 수정 | **직접 수정** — 해당 스킬 파일 편집 |
| **L4 지식** | memory/rules에 없는 원칙 | memory/rules 추가 | **직접 수정** — memory 파일 생성/업데이트 |
| **L5 사용자 지시** | 모호한 표현, 암묵적 기대 | 사용자에게 피드백 | 보고서 → 사용자 읽기 |

**분류 기준:**
- "코드에서 뭘 안 했는가?" → L1
- "PRD에 왜 안 적혀 있었는가?" → L2
- "PRD에 적혀야 하는 걸 스킬이 왜 안 물어봤는가?" → L3
- "이런 원칙이 있었다면 잡혔을 텐데?" → L4
- "사용자가 이걸 명시했다면 잡혔을 텐데?" → L5

## Step 4: 보고서 생성

`docs/0-inbox/`에 보고서 파일을 생성한다. 파일명: `{순번}-[retro]{feature명}.md`

### 보고서 템플릿

```markdown
# Retro: [feature명] — YYYY-MM-DD

## 비교 기준
- **PRD:** [경로]
- **Diff 범위:** [시작 커밋..HEAD]
- **커밋 수:** N
- **변경 파일:** N

## 항목별 비교

| # | PRD 항목 | 일치 | 갭 요약 | 계층 |
|---|---------|------|---------|------|
| 1 | 동기 | ✅/❌/⚠️/🔀 | ... | L? |
| 2 | 인터페이스 | ... | ... | ... |
| 3 | 산출물 | ... | ... | ... |
| 4 | 경계 | ... | ... | ... |
| 5 | 금지 | ... | ... | ... |
| 6 | 검증 | ... | ... | ... |

**일치율:** N/6

## 갭 상세

### ❌ PRD에 있는데 구현 안 됨
- ...

### ⚠️ 구현됐는데 PRD에 없었음
- ...

### 🔀 의도와 다르게 구현됨
- ...

## 계층별 개선 제안

### L1 코드 — 즉시 수정 또는 /backlog
- [ ] [구체적 수정 내용] → 즉시 수정 / `/backlog`에 저장
- [ ] ...

### L2 PRD 스킬 — ✏️ 직접 수정 완료
- PRD 스킬의 [항목]에 [체크리스트] 추가함
- 이유: [이번에 이것 때문에 누락됨]
- 수정 파일: [경로]

### L3 스킬 — ✏️ 직접 수정 완료
- [스킬명]의 [부분]을 [이렇게] 수정함
- 이유: ...
- 수정 파일: [경로]

### L4 지식 — ✏️ 직접 수정 완료
- memory에 "[원칙]" 추가함
- 이유: ...
- 수정 파일: [경로]

### L5 사용자 피드백
- "💬 [상황 설명] — 다음에 [이렇게] 명시해주시면 더 정확한 결과를 낼 수 있습니다."
- ...

## 다음 행동
- L1 항목이 있으면 → 즉시 수정 또는 `/backlog`에 저장
- L2~L4는 이미 직접 수정 완료 — 보고서에 변경 내역 기록
- L5 피드백이 있으면 → 사용자가 참고
```

## Step 5: 사용자에게 보고서 요약 제시

보고서를 생성한 후, 대화에 핵심 요약을 보여준다:

```
## Retro 결과

일치율: N/6

### 주요 갭
- ❌ [항목]: [요약]
- ⚠️ [항목]: [요약]

### 경험 DB
- 📚 추가: [도메인] — [교훈 한 줄]
- 📚 빈도 증가: [도메인] — [교훈] (N회째)
- 🎓 feedback 승격: [도메인] — [교훈] (3회 반복)

### L1 백로그 (자율 처리 가능)
- [ ] ...

### 사용자 피드백
- 💬 ...

보고서: docs/0-inbox/N-[retro]feature.md
L1 백로그를 랄프루프로 처리할까요?
```

## Step 4.5: L2~L4 직접 수정

보고서를 생성한 후, L2~L4 갭이 있으면 **보고서 작성과 동시에 해당 파일을 직접 수정**한다.

| 계층 | 수정 대상 | 수정 방법 |
|------|----------|----------|
| L2 | `.claude/skills/prd/SKILL.md` | 누락된 체크리스트 항목 추가 |
| L3 | `.claude/skills/{스킬명}/SKILL.md` | 구조적 약점 보완 |
| L4 | `.claude/projects/*/memory/*.md` | 새 원칙을 memory 파일로 생성, MEMORY.md 인덱스 업데이트 |

수정한 파일 경로를 보고서의 "수정 파일" 필드에 기록한다.

## Step 6: 경험 추출

갭 분석이 끝나면, 이번 retro에서 **프로젝트 고유 경험**을 추출하여 경험 DB에 저장한다.

### 프로젝트 고유 판정 (리트머스)

교훈에 이 프로젝트의 **파일명, 타입명, 아키텍처 패턴**이 들어가면 프로젝트 고유 → 저장.
일반 코딩 지식("변수명은 의미 있게", "에러 핸들링 하라")은 LLM이 이미 알고 있으므로 저장하지 않는다.

예시:
- ✅ "behavior 파일에도 수동 keyMap 바인딩이 있을 수 있다" → behavior가 프로젝트 고유 → 저장
- ✅ "relationships만 바꾸면 expandedIds는 자동 보존된다" → NormalizedData 구조 고유 → 저장
- ❌ "타입을 명시하면 버그가 줄어든다" → 일반론 → 버림

### 저장 위치 및 포맷

`memory/experience_db.md`의 테이블에 행을 추가한다:

```markdown
| # | 도메인 | 상황 | 갭 | 교훈 | 빈도 | 최근 |
|---|--------|------|-----|------|------|------|
| N | plugin/keyMap | 수동 바인딩을 plugin.keyMap으로 교체 | behavior의 바인딩 누락 | behavior도 grep 범위에 포함 | 1 | 2026-03-21 |
```

### 패턴 매칭

새 경험을 추가하기 전에 기존 테이블을 확인한다:
- **같은 교훈이 이미 있으면** → 빈도 +1, 최근 날짜 갱신
- **새로운 교훈이면** → 새 행 추가

### 빈도 기반 승격

빈도가 **3회 이상**이면:
1. 해당 교훈을 `type: feedback` memory 파일로 승격한다 (Why + How to apply 포맷)
2. 경험 DB의 해당 행에 "→ feedback 승격됨" 표시 (행은 이력으로 유지)
3. MEMORY.md 인덱스에 새 feedback 파일 추가

승격은 "삽질이 규칙이 될 만큼 반복됐다"는 신호이므로, AI가 매 세션에서 자동으로 읽는 feedback으로 올리는 것.

## 종료 조건

- 보고서 생성 완료
- L2~L4 직접 수정 완료 (해당 시)
- L1 항목 → 즉시 수정 또는 `/backlog`에 저장
- **경험 DB 업데이트 완료** (프로젝트 고유 경험 추출)
- 사용자에게 요약 제시 (수정 내역 + 경험 추가/승격 내역 포함)

⛔ **금지:** L5(사용자 피드백)는 자동 반영하지 않는다 — 보고서에 기록만 한다.
