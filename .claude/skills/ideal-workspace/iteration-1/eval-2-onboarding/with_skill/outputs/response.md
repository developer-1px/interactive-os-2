## 시뮬레이션: 경험

도메인: **경험** — 새 사용자의 Visual CMS 에디터 첫 진입 저니맵.

### 맥락

"에디터"는 interactive-os 기반 Visual CMS다. 구조:
- **ActivityBar** (좌측) — CMS / Viewer / Agent / UI 등 8개 레이어
- **CMS 진입 시** — TopToolbar + Sidebar(섹션 아웃라인) + Canvas(공간 탐색 편집) + DetailPanel(속성 편집)
- 키보드 우선 설계: 화살표로 공간 이동, Enter로 drill-in, Escape로 복귀, F2로 rename
- 플러그인: undo/redo, clipboard, dnd, rename, CRUD, spatial navigation

"새 사용자"는 CMS를 처음 여는 사람이다. 개발자가 아니라 **콘텐츠 관리자** — interactive-os의 내부 구조를 모른다.

### 저니맵

| 단계 | 행동 | 생각/감정 | 터치포인트 | 문제점 |
|------|------|----------|-----------|--------|
| 1. 진입 | URL 접속, CMS 화면 로드 | "이게 뭐지? 어디서부터 시작하지?" | ActivityBar에서 CMS 아이콘이 기본 선택, Canvas에 샘플 데이터 표시 | 빈 캔버스라면 첫 행동이 없다. 샘플 데이터가 있어도 "이것이 내 데이터인가 예시인가" 구분 불가 |
| 2. 시각 탐색 | 눈으로 화면 구조 파악 — Sidebar, Canvas, DetailPanel 3영역 | "왼쪽이 목차고, 가운데가 편집 영역이겠지" | Sidebar 섹션 목록, Canvas의 시각적 블록 배치 | Sidebar가 읽기 전용 아웃라인이라는 것을 모름. 클릭해서 편집하려 할 수 있음 |
| 3. 첫 클릭 | Canvas의 아무 블록을 클릭 | "이걸 선택한 건가?" 포커스 아웃라인이 보이면 안심 | 포커스 outline(CSS), FloatingToolbar 표시 | FloatingToolbar의 아이콘이 무슨 뜻인지 모름. 툴팁이 있지만 발견하기 전까지 의미 불투명 |
| 4. 첫 편집 시도 | 텍스트를 바꾸고 싶어 더블클릭 또는 타이핑 시작 | "어떻게 고치지?" | F2(rename) 또는 DetailPanel의 필드 | **F2 단축키를 모르면 편집 진입 불가**. DetailPanel이 우측에 있다는 것을 모르면 막힘. 인라인 편집 기대 vs 실제 Form 기반 편집의 괴리 |
| 5. 구조 변경 시도 | 블록 순서를 바꾸거나 새 블록 추가 | "드래그하면 되나? 메뉴가 있나?" | Mod+ArrowUp/Down(이동), Mod+D(복제), Delete(삭제) | **키보드 단축키를 모르면 구조 변경 방법을 전혀 발견하지 못함**. 드래그 앤 드롭은 있지만, 우클릭 컨텍스트 메뉴가 없음 |
| 6. 실수 복구 | 잘못 삭제하고 당황 | "되돌리기 되나?" | Mod+Z(undo) | Undo는 직관적이라 이건 괜찮음. 단 undo 후 포커스가 복구되는지가 안심/불안의 분기점 |
| 7. 멀티 콘텐츠 관리 | 다른 섹션, 다른 locale 편집 | "한국어 바꿨는데 영어는 어디서?" | TopToolbar의 locale switcher, I18nSheet | Locale switcher가 toolbar에 있다는 걸 발견해야 함. I18nSheet의 존재를 모르면 하나씩 전환하며 편집 |
| 8. 확인/발행 | 완성된 결과를 보고 싶음 | "실제로 어떻게 보이지?" | Present 모드 (Presentation 아이콘) | Present 버튼의 발견성. "발행"과 "미리보기"의 차이 불명확 |

### 발견된 갭

- **갭 1: 편집 진입 경로의 발견성(discoverability) 부재.** F2로 rename, DetailPanel로 속성 편집 — 두 경로 모두 사용자가 스스로 발견해야 한다. 첫 클릭 후 "지금 여기서 뭘 할 수 있는지" 알려주는 어포던스가 없다. FloatingToolbar가 이 역할을 할 수 있지만, 현재는 아이콘만 나열되어 있다.

- **갭 2: 구조 변경 수단의 발견성 부재.** Mod+D(복제), Delete(삭제), Mod+Arrow(이동)는 숙련자용이다. 새 사용자에게는 우클릭 컨텍스트 메뉴 또는 "+" 버튼 같은 시각적 진입점이 필요하지만, ContextMenu는 현재 미구현(PROGRESS.md 확인).

- **갭 3: "첫 화면"에서 할 일이 없다.** 샘플 데이터가 로드되지만, 이것이 튜토리얼인지 내 데이터인지 구분이 안 된다. "여기를 클릭해서 시작하세요" 같은 온보딩 가이드가 없다.

- **갭 4: Sidebar 역할의 모호성.** Sidebar는 읽기 전용 아웃라인(내비게이션)이지만, 새 사용자는 Sidebar에서 CRUD를 시도할 수 있다. "탭 CRUD는 캔버스 전용"이라는 설계 원칙이 UI에 드러나지 않는다.

- **갭 5: DetailPanel의 존재 인지 실패.** 우측 패널이 접혀 있거나 작으면 못 볼 수 있다. 첫 포커스 시 DetailPanel이 활성화되며 "여기서 편집하세요"를 암시해야 하는데, 현재는 수동 발견에 의존한다.

### 질문

제 판단: 갭 1(편집 진입 발견성)과 갭 2(구조 변경 발견성)가 가장 치명적이다. 이유: 새 사용자가 "첫 편집"과 "첫 구조 변경"을 못 하면, 에디터의 가치를 체험하기 전에 이탈한다. 나머지 갭은 한 번 배우면 해결되지만, 이 두 가지는 배울 기회 자체가 없다.

구체화할 방향이 두 가지 있다:

1. **발견성 우선 — 기존 UI에 어포던스 추가.** FloatingToolbar에 레이블 추가, 첫 포커스 시 "F2로 편집" 같은 힌트 표시, ContextMenu 구현. 에디터의 구조를 바꾸지 않고 "안내"만 추가하는 접근.

2. **동선 재설계 — 첫 진입 시퀀스를 설계.** 빈 캔버스 + 템플릿 선택(CmsTemplatePicker가 이미 존재)으로 시작하여 "첫 행동"을 유도하고, 그 과정에서 자연스럽게 편집/구조변경을 경험하게 하는 접근.
