좋은 질문이다. 현재 프로젝트 구조를 기반으로 "새 사용자의 이상적 첫 경험"을 구체화해 보겠다.

---

## 현재 상태 진단

현재 `/` (루트)는 CMS가 랜딩이다. 새 사용자가 들어오면 CMS 편집 화면을 바로 보게 된다. 문제는:

1. **CMS는 "편집"이지 "이해"가 아니다** -- 사용자가 interactive-os가 뭔지 모르는 상태에서 편집 화면부터 보게 된다
2. **ActivityBar에 14개 항목이 나열된다** -- CMS, Viewer, Agent, UI, Theme, Store, Engine, Axis, Pattern, Plugin, Collection, Components, Area. 어디부터 봐야 할지 모른다
3. **계층이 보이지 않는다** -- App(CMS/Viewer/Agent) vs OS(내부 구조) vs Util(테마)의 구분이 시각적으로 separator 하나뿐이다

## 사용자 페르소나 분류

"에디터"라고 했을 때 두 종류의 새 사용자가 있다:

### A. 라이브러리 사용자 (개발자)
- interactive-os를 npm에서 설치해서 자기 앱에 쓰려는 사람
- 관심사: "treegrid에 키보드+undo+클립보드를 어떻게 한 줄로 붙이지?"
- 이상적 첫 경험: **동작하는 데모를 직접 조작** -> 코드 확인 -> 내 앱에 복사

### B. CMS 편집자 (비개발자 또는 콘텐츠 관리자)
- Visual CMS를 써서 콘텐츠를 편집하려는 사람
- 관심사: "이 화면에서 뭘 어떻게 바꾸지?"
- 이상적 첫 경험: **샘플 콘텐츠가 이미 있고, 하나를 클릭해서 즉시 편집**

## 이상적 온보딩 경험 제안

### 1단계: 랜딩 = Viewer (이미 설계 방향과 일치)

메모리에 "Viewer가 메타 쇼케이스이자 default landing인 이유"가 기록되어 있다. 이 방향이 맞다.

- **Viewer가 랜딩인 이유**: interactive-os 자체로 만든 UI(트리 탐색, 코드 뷰어, Quick Open, 의존성 그래프)를 보여준다. "이 라이브러리로 만든 것을 이 라이브러리 안에서 쓰고 있다"는 자기 증명
- 사용자가 처음 보는 것: 마크다운 문서 뷰어 + 좌측 파일 트리 + `Cmd+P`로 Quick Open
- 즉시 조작 가능: 키보드로 트리 탐색, 파일 열기, 코드 블록 복사

### 2단계: 조작해 보기 (발견 가능한 인터랙션)

새 사용자가 "이거 어떻게 쓰는 거지?"라고 묻기 전에 답이 와야 한다:

| 발견 순서 | 인터랙션 | 어떻게 발견하나 |
|-----------|---------|---------------|
| 1 | 화살표로 파일 트리 탐색 | 첫 포커스가 트리에 있으므로 자연스럽게 화살표를 눌러 본다 |
| 2 | Enter로 파일 열기 | 트리 항목이 포커스되면 Enter가 본능적 |
| 3 | `Cmd+P` Quick Open | 상단에 "Cmd+P" 힌트가 보인다 |
| 4 | 사이드바 크기 조절 | 경계선에 커서 변경 |
| 5 | ActivityBar로 다른 레이어 탐색 | 좌측 아이콘을 호버하면 툴팁 |

핵심 원칙: **모든 인터랙션이 keyboard-first로 동작하고, 되돌릴 수 있다(가역적 동선).**

### 3단계: CMS 편집 진입

Viewer에서 "CMS" 아이콘을 눌러 전환하면:

- 샘플 슬라이드가 이미 들어 있다 (빈 화면 금지)
- 첫 포커스가 캔버스의 첫 섹션에 간다
- 화살표로 섹션 간 이동, Enter로 편집 진입, Escape로 빠져나오기
- Undo(`Cmd+Z`)가 즉시 동작한다는 것을 경험

### 4단계: "어떻게 만들어진 거지?" (개발자 경로)

개발자라면 여기서 호기심이 생긴다:

- ActivityBar의 OS 영역(Store, Engine, Axis, Pattern, Plugin, Collection, Components)으로 이동
- 각 쇼케이스에서 동작을 직접 체험하고, Area 문서에서 설계를 읽는다

## 현재 구현과의 갭

| 항목 | 현재 | 이상 |
|------|------|------|
| 랜딩 페이지 | CMS (`/`) | Viewer (`/viewer`) 또는 Overview |
| 초기 포커스 | 없음 (ActivityBar가 받을 수도 있음) | Viewer 트리의 첫 항목 |
| 키보드 힌트 | 없음 | `Cmd+P`, 화살표 등 최소한의 시각적 힌트 |
| 빈 상태 가이드 | CMS에 샘플 데이터 있음 | 유지 (좋은 선택) |
| ActivityBar 계층 | separator 1개 | App / OS 레이블 또는 시각적 구분 강화 |
| 온보딩 투어 | 없음 | 불필요 -- 인터랙션 자체가 온보딩이어야 함 |

## 핵심 통찰

온보딩의 본질은 "설명"이 아니라 "첫 성공 경험"이다.

- **첫 성공 = 키보드만으로 파일을 열어 읽는 것** (Viewer)
- **두 번째 성공 = 콘텐츠를 편집하고 Undo하는 것** (CMS)
- **세 번째 성공 = 같은 패턴이 모든 곳에서 동작한다는 것을 발견** (쇼케이스)

interactive-os의 가치 제안("비용 제거")이 온보딩 자체에서 증명되어야 한다. 사용자가 키보드 네비게이션, undo, 클립보드를 "배우지 않고 그냥 쓰는" 순간이 온보딩의 완료 시점이다.

- 랜딩 페이지 전환 (`/` -> Viewer)을 실제로 구현할지
- ActivityBar의 시각적 계층 강화를 디자인할지
- 초기 포커스 정책을 어떤 규칙으로 설계할지

어떤 방향이 지금 가장 의미 있는지 골라 달라.
