## 대립해소도: 컴포넌트 범용성 vs 단순성

### Step 0: 진입

딜레마: "컴포넌트를 범용으로 만들면 설정이 복잡해지고, 단순하게 만들면 재사용이 안 돼"

두 방향이 충돌한다:
- **범용으로 만든다** → 설정(props, config)이 복잡해진다
- **단순하게 만든다** → 재사용이 안 된다

### Step 1: 공통 목표 (A)

```
A: "적은 노력으로 다양한 UI를 일관되게 제공한다"
```

양쪽 모두 원하는 것은 같다. 범용성을 추구하는 이유도, 단순성을 추구하는 이유도, 결국 "개발 효율과 사용자 경험의 일관성"을 동시에 달성하기 위해서다.

### Step 2: 필요조건 (B, C)

```
B: "다양한 맥락에서 재사용 가능해야 한다" (A를 위해)
C: "사용자(개발자)가 쉽게 사용할 수 있어야 한다" (A를 위해)
```

B와 C 자체는 충돌하지 않는다. 둘 다 A를 위해 필요하다.

### Step 3: 행동 (D, D') + 충돌 확인

```
D: "컴포넌트에 많은 옵션/props를 노출한다" (B를 충족하기 위해)
D': "컴포넌트의 인터페이스를 최소화한다" (C를 충족하기 위해)
D와 D'는 양립할 수 없다. — 옵션을 늘리면 인터페이스가 커지고, 최소화하면 커버리지가 줄어든다.
```

```
              [A: 적은 노력으로 다양한 UI를 일관되게 제공]
             /                                           \
      [B: 다양한 맥락에서                        [C: 개발자가 쉽게
       재사용 가능]                                사용 가능]
             \                                           /
  [D: 옵션/props를        ↔충돌↔        [D': 인터페이스를
   많이 노출]                              최소화]
```

### Step 4: 전제 분석

| 화살표 | 연결 | 전제 (왜 필수인가?) |
|--------|------|---------------------|
| ① | A → B | **①-1**: 일관된 UI를 위해서는 같은 컴포넌트를 여러 곳에서 써야 한다. 컴포넌트마다 따로 만들면 불일치가 생기니까. |
| ① | A → B | **①-2**: "적은 노력"을 위해 한 번 만들고 여러 번 쓸 수 있어야 한다. |
| ② | A → C | **②-1**: 사용이 어려우면 개발 속도가 떨어지고, 잘못 사용해서 불일치가 생긴다. |
| ② | A → C | **②-2**: 쉬워야 팀 누구나 쓸 수 있다. 진입장벽이 높으면 실질적으로 재사용이 안 된다. |
| ③ | B → D | **③-1**: 재사용하려면 각 맥락의 차이를 흡수해야 하고, 차이를 흡수하는 방법은 props/옵션뿐이다. |
| ③ | B → D | **③-2**: 컴포넌트 하나가 모든 변형을 포함해야 재사용이다. |
| ④ | C → D' | **④-1**: 옵션이 많으면 어떤 조합이 유효한지 파악하기 어렵다. 적을수록 쉽다. |
| ④ | C → D' | **④-2**: 인터페이스가 작아야 문서를 읽지 않아도 사용할 수 있다. |
| ⑤ | D ↔ D' | **⑤-1**: 옵션을 늘리는 것과 줄이는 것은 같은 축의 반대 방향이므로 동시에 할 수 없다. |
| ⑤ | D ↔ D' | **⑤-2**: 하나의 컴포넌트에 대한 하나의 인터페이스만 존재한다. |

### Step 5: 전제 깨기 → Injection (돌파구)

#### 전제 ③-1 깨기: "차이를 흡수하는 방법은 props/옵션뿐이다"

이것이 가장 깨기 쉬운 전제다. props가 아닌 다른 방식으로 변형을 흡수할 수 있다.

```
전제 ③-1: "재사용하려면 각 맥락의 차이를 props/옵션으로 흡수해야 한다"
→ 깨기: "Composition(합성)으로 차이를 흡수하면 props가 불필요하다"
→ Injection: 컴포넌트를 모놀리식 블록이 아닌 합성 가능한 작은 단위(headless primitives)로 분해한다.
  각 primitive는 인터페이스가 최소이면서, 조합으로 다양한 맥락을 커버한다.
→ 검증: B 충족 ✅ (합성으로 다양한 맥락 커버) / C 충족 ✅ (각 단위가 단순) / 부작용 없음 ✅
```

#### 전제 ③-2 깨기: "컴포넌트 하나가 모든 변형을 포함해야 재사용이다"

```
전제 ③-2: "하나의 컴포넌트가 모든 변형을 담아야 재사용"
→ 깨기: "행동(behavior)을 공유하고 표현(rendering)은 각자 결정하면, 행동 수준에서 재사용된다"
→ Injection: behavior/hook 레이어를 분리한다. useCombobox, useTreeGrid처럼
  ARIA 행동을 hook으로 제공하고, 렌더링은 소비자가 결정한다.
  재사용 단위가 "완성된 컴포넌트"에서 "행동 로직"으로 이동한다.
→ 검증: B 충족 ✅ (행동이 재사용됨) / C 충족 ✅ (hook 인터페이스는 작음) / 부작용 없음 ✅
```

#### 전제 ⑤-2 깨기: "하나의 컴포넌트에 대한 하나의 인터페이스만 존재한다"

```
전제 ⑤-2: "하나의 컴포넌트 = 하나의 인터페이스"
→ 깨기: "레이어별로 인터페이스를 분리하면 양쪽이 공존한다"
→ Injection: 기본 인터페이스(simple API)와 확장 인터페이스(advanced API)를 분리한다.
  80%의 사용은 기본 인터페이스로 커버하고, 나머지 20%만 확장 인터페이스를 사용한다.
  (예: <Combobox items={[...]} /> 기본 vs <Combobox.Root><Combobox.Input /><Combobox.List>...</Combobox.List></Combobox.Root> 확장)
→ 검증: B 충족 ✅ (확장 인터페이스로 커버) / C 충족 ✅ (기본 인터페이스는 단순) / 부작용 없음 ✅
```

### 결론

세 전제를 종합하면 하나의 돌파구로 수렴한다:

**"범용성의 단위를 완성된 컴포넌트에서 합성 가능한 행동(behavior)으로 바꾼다."**

- 행동(hook)은 범용이고 인터페이스가 작다 — B와 C가 동시에 충족된다.
- 컴포넌트는 행동을 조합한 결과물이므로, 각 맥락에 맞게 조합하되 행동 로직은 재사용된다.
- 필요하면 자주 쓰는 조합을 "기본 인터페이스"로 포장해서 단순성까지 확보한다.

충돌은 "재사용 단위 = 완성된 컴포넌트"라는 전제에서 비롯되었다. 재사용 단위를 행동으로 내리면 충돌이 증발한다.
