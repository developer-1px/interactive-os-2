## 대립해소도: 타입 안전성 vs 유연성

### Step 0: 진입

독립 호출. 사용자가 제시한 딜레마:
> "타입 안전성을 높이면 유연성이 떨어지고, 유연성을 높이면 타입이 느슨해져."

타입 시스템의 엄격함과 코드의 유연한 확장성이 양립 불가능하다고 느끼는 상황이다.

---

### Step 1: 공통 목표 (A)

**A: 변경에 강하고 안전한 코드베이스를 유지한다.**

양쪽 모두 원하는 것은 같다. 버그 없이 안정적이면서, 새로운 요구사항에도 쉽게 대응할 수 있는 코드다. 타입 안전성도, 유연성도 결국 이 목표를 위한 수단이다.

---

### Step 2: 필요조건 (B, C)

**B: 컴파일 타임에 오류를 잡아야 한다.** (타입 안전성 방향)
- A를 위해 B가 필요하다: 런타임 버그는 발견이 늦고 비용이 크다. 컴파일 타임에 잡을수록 변경에 강해진다.

**C: 다양한 형태의 데이터와 동작을 수용해야 한다.** (유연성 방향)
- A를 위해 C가 필요하다: 요구사항은 계속 변하고, 확장 불가능한 코드는 변경 비용이 폭발한다.

B와 C 자체는 충돌하지 않는다. 둘 다 "변경에 강한 코드"를 위해 필요한 것이다.

---

### Step 3: 행동 (D, D') + 충돌 확인

**D: 타입을 좁고 구체적으로 정의한다.** (B를 충족하는 행동)
- 구체적 타입, 리터럴 유니온, 엄격한 인터페이스로 가능한 값의 범위를 최소화한다.

**D': 타입을 넓고 일반적으로 정의한다.** (C를 충족하는 행동)
- `any`, `Record<string, unknown>`, 느슨한 제네릭으로 어떤 형태든 받을 수 있게 한다.

**D와 D'는 양립할 수 없다.** 같은 위치에서 타입을 동시에 좁히고 넓힐 수 없다.

```
              [A: 변경에 강하고 안전한 코드베이스]
             /                                  \
      [B: 컴파일 타임                    [C: 다양한 형태의
       오류 포착]                         데이터/동작 수용]
             \                                  /
  [D: 타입을 좁고        ↔ 충돌 ↔     [D': 타입을 넓고
   구체적으로 정의]                     일반적으로 정의]
```

---

### Step 4: 전제 분석

| 화살표 | 연결 | 전제 (왜 필수인가?) |
|--------|------|---------------------|
| ① | A → B | P1: 컴파일 타임 검증 없이는 변경의 안전성을 보장할 수 없다. |
| | | P2: 타입 검사가 오류 포착의 주된 수단이다. |
| ② | A → C | P3: 미래 요구사항을 예측할 수 없으므로 유연해야 한다. |
| | | P4: 유연하지 않으면 변경할 때마다 대규모 리팩토링이 필요하다. |
| ③ | B → D | **P5: 컴파일 타임에 오류를 잡으려면 타입을 좁고 구체적으로 써야만 한다.** |
| | | P6: 넓은 타입은 컴파일러가 검증할 수 있는 정보가 없다. |
| ④ | C → D' | **P7: 다양한 형태를 수용하려면 타입을 넓게 풀 수밖에 없다.** |
| | | P8: 구체적 타입은 새로운 형태가 추가될 때마다 수정이 필요하다. |
| ⑤ | D ↔ D' | P9: 하나의 타입 선언은 좁거나 넓거나 둘 중 하나다. |
| | | P10: 같은 코드 위치에서 두 전략을 동시에 적용할 수 없다. |

---

### Step 5: 전제 깨기 -> Injection (돌파구)

#### 전제 P7 깨기 (④: C → D')

> P7: "다양한 형태를 수용하려면 타입을 넓게 풀 수밖에 없다."

이 전제는 **거짓**이다. 제네릭(Generic)과 조건부 타입(Conditional Types)은 타입을 넓게 풀지 않으면서도 다양한 형태를 수용한다. `T extends Base`로 제약하면 컴파일러는 `T`의 구체적 정보를 보존하면서도 `Base`를 만족하는 어떤 형태든 받을 수 있다.

#### 전제 P5 깨기 (③: B → D)

> P5: "컴파일 타임에 오류를 잡으려면 타입을 좁고 구체적으로 써야만 한다."

이 전제도 **부분적으로 거짓**이다. 제네릭 제약(`T extends X`)은 "좁지만 구체적"이 아니라 "구조적으로 제약된 다형성"이다. 타입이 좁은 게 아니라 **규칙이 명확한 것**이고, 규칙이 명확하면 컴파일러는 검증할 수 있다.

#### 전제 P9 깨기 (⑤: D ↔ D')

> P9: "하나의 타입 선언은 좁거나 넓거나 둘 중 하나다."

이 전제도 **거짓**이다. 타입스크립트의 제네릭은 **선언 시점에서는 넓고, 사용 시점에서는 좁다.** `function process<T extends Item>(items: T[]): T[]`는 선언에서는 "Item을 만족하는 모든 T"를 수용하지만(유연), 호출 시점에서는 T가 구체적 타입으로 추론되어 좁아진다(안전). 하나의 선언이 동시에 양쪽을 만족한다.

---

### Injection (돌파구)

**깨진 전제:** P7 + P5 + P9 -- "유연하려면 넓어야 하고, 안전하려면 좁아야 하며, 둘은 동시에 불가능하다"

**돌파구: 제네릭 제약(Constrained Generics)으로 "선언은 열고, 사용은 닫는다."**

구체적으로:

1. **인터페이스는 최소한의 구조적 계약으로 정의한다** -- `interface HasId { id: string }` 수준의 계약. 넓지도 좁지도 않은 "필요충분 제약".
2. **제네릭 파라미터에 계약을 건다** -- `<T extends HasId>`. 어떤 형태든 수용하되(유연), `id`가 있다는 건 보장한다(안전).
3. **타입 추론에 의존한다** -- 호출 시점에서 TypeScript가 T를 구체 타입으로 추론하므로, 사용하는 쪽에서는 좁은 타입의 이점을 그대로 누린다.
4. **discriminated union으로 확장한다** -- 새 형태 추가 시 유니온에 멤버 하나만 추가하면 되고, `switch` 문의 exhaustive check가 컴파일 타임 안전성을 보장한다.

**D'' (새로운 행동):** 타입을 좁지도 넓지도 않게, **구조적 계약 + 제네릭 제약**으로 정의한다.

### 검증

| 기준 | 결과 |
|------|------|
| B 충족 (컴파일 타임 오류 포착) | 제네릭 제약과 discriminated union은 컴파일러가 완전히 검증 가능하다. `any`처럼 정보가 사라지지 않는다. |
| C 충족 (다양한 형태 수용) | 제네릭은 계약만 만족하면 어떤 형태든 받는다. 유니온에 멤버 추가도 기존 코드 수정 없이 가능하다. |
| 새로운 충돌 없음 | 제네릭 제약은 TypeScript의 핵심 기능이며, 학습 비용 외에 런타임 오버헤드나 구조적 부작용이 없다. |

**결론:** "좁게 vs 넓게"는 잘못된 이분법이었다. 타입의 폭을 선택하는 것이 아니라, **계약의 정밀도를 선택하는 것**이다. 구조적 계약이 명확하면 컴파일러는 안전성을 보장하고, 제네릭이 유연성을 보장한다. 충돌은 증발한다.
