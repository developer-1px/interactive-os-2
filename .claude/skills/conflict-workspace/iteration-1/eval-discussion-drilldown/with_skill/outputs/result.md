# /conflict 드릴다운: "매 턴 3단계 UX를 유지해야 한다"는 진짜 제약인가?

> /discussion에서 라우팅됨. 제약이 2턴 연속 정체(노란불)하여 대립해소도로 분석.

## Step 0: 진입 — 충돌 지점 파악

discussion에서 정체된 제약: **"매 턴 3단계 UX(탐색 -> 선택 -> 조작)를 유지해야 한다."**

이 제약이 정체된 이유: 칸반 쇼케이스에서 `Alt+Arrow`(조작)는 탐색/선택 없이 바로 실행된다. 카드에 포커스가 있으면 즉시 컨테이너 간 이동이 일어난다. 이것이 "매 턴 3단계"와 충돌하는 것처럼 보이면서도, 3단계를 강제하면 조작이 번거로워진다.

## Step 1: 공통 목표 (A)

> **A: 키보드 사용자가 최소 인지 부하로 데이터를 확실하게 조작할 수 있어야 한다.**

양쪽 모두 원하는 것: 키보드 UX가 직관적이고 예측 가능하되, 사용자가 실수하지 않아야 한다.

## Step 2: 필요조건 (B, C)

> **B: 사용자가 현재 맥락(어디에 있고, 무엇을 하려는지)을 항상 파악해야 한다.** (A를 위해)

> **C: 조작이 최소 키 입력으로 즉시 완료되어야 한다.** (A를 위해)

B는 "인지적 안전성" 측면 -- 3단계가 주는 가치. C는 "효율성" 측면 -- 즉시 조작이 주는 가치.

## Step 3: 행동 (D, D') + 충돌

> **D: 모든 조작에 탐색->선택->실행 3단계를 강제한다.** (B를 충족하려면)

> **D': 포커스 위치에서 단축키로 즉시 조작한다 (1단계).** (C를 충족하려면)

> **D와 D'는 양립할 수 없다.** -- 3단계 강제 vs 1단계 즉시 실행

```
              [A: 최소 인지 부하로 확실한 조작]
             /                                \
      [B: 현재 맥락을                    [C: 최소 키 입력으로
       항상 파악]                         즉시 조작 완료]
             \                                /
       [D: 매 턴 3단계       ↔충돌↔    [D': 포커스 위치에서
        탐색->선택->실행                  단축키 즉시 실행]
        강제]
```

## Step 4: 전제 나열

| 화살표 | 연결 | 전제 (왜 필수인가?) |
|--------|------|---------------------|
| ① | A -> B | **전제 ①-1:** 맥락을 모르면 의도하지 않은 조작을 하게 된다. **전제 ①-2:** 실수를 되돌리는 비용이 크다. |
| ② | A -> C | **전제 ②-1:** 키 입력이 많을수록 인지 부하가 증가한다. **전제 ②-2:** 키보드 우선 UX의 가치는 마우스보다 빠른 조작에 있다. |
| ③ | B -> D | **전제 ③-1:** 맥락 파악은 명시적 단계(탐색->선택)를 거쳐야만 가능하다. **전제 ③-2:** 포커스 위치만으로는 사용자가 "어디에 있는지" 충분히 인지하지 못한다. |
| ④ | C -> D' | **전제 ④-1:** 즉시 조작하려면 중간 단계를 생략해야 한다. |
| ⑤ | D ↔ D' | **전제 ⑤-1:** 3단계와 1단계는 동시에 존재할 수 없다 (모든 조작에 동일 규칙을 적용해야 한다). |

## Step 5: 전제 깨기 -> Injection

### 깨기 1: 전제 ③-1 (가장 취약)

> **전제 ③-1: "맥락 파악은 명시적 단계(탐색->선택)를 거쳐야만 가능하다"**

이 전제는 **거짓이다.**

interactive-os에서 포커스 자체가 이미 맥락 정보를 제공한다:
- roving-tabindex로 포커스된 카드가 시각적으로 명확하다
- `aria-activedescendant` 또는 `tabindex="0"`으로 현재 위치가 스크린 리더에도 노출된다
- 칸반에서 `Alt+Arrow`를 누르는 시점에 사용자는 이미 화살표로 탐색하여 해당 카드에 도달한 상태다

즉, **포커스 도달 과정 자체가 암묵적 탐색+선택이다.** 별도의 명시적 "선택 단계"가 없어도 맥락은 파악된 상태.

### 깨기 2: 전제 ⑤-1 (보강)

> **전제 ⑤-1: "모든 조작에 동일 규칙을 적용해야 한다"**

이 전제도 **거짓이다.**

실제로 데스크톱 앱들은 조작 유형에 따라 단계 수가 다르다:
- 파괴적 조작(삭제): 선택 -> 확인 -> 실행 (3단계)
- 이동 조작(reorder, moveTo): 포커스 + 즉시 실행 (1단계) -- 가역적이므로 안전
- 이름 변경(rename): 진입 -> 편집 -> 확정 (3단계)

핵심은 **가역성(reversibility)**이다. 가역적 조작은 단계를 줄여도 안전하다.

### Injection (돌파구)

> **"매 턴 3단계 UX"가 아니라 "가역성에 비례한 단계 수"를 적용한다.**

구체적으로:
- **가역적 조작** (`Alt+Arrow` 이동, reorder): 포커스 위치에서 즉시 실행 (1단계). `Alt+반대방향`으로 원래 위치 복귀 가능하고, `Mod+Z`로도 undo 가능.
- **비가역적/파괴적 조작** (삭제, 잘라내기): 선택을 명시한 후 실행 (2~3단계). 단, undo가 있으므로 확인 dialog까지는 불필요.
- **모드 전환 조작** (rename): 진입->편집->확정 (3단계). 모드 전환은 맥락이 바뀌므로 단계가 필요.

### Injection 검증

1. **B를 충족하는가? (맥락 파악)** -- 포커스 + 시각적 피드백으로 맥락이 항상 명확. 파괴적 조작은 여전히 명시적 선택 요구. **통과.**
2. **C를 충족하는가? (최소 키 입력)** -- 빈번한 조작(이동, reorder)이 즉시 실행. **통과.**
3. **새로운 충돌을 만들지 않는가?** -- 칸반 PRD의 기존 인터페이스 테이블과 완전히 일치. `Alt+Arrow`는 즉시 실행, `Delete`는 선택 필요, `F2`는 모드 전환. 이미 이 원칙대로 설계되어 있었다. **통과.**

## /discussion 복귀용 요약

### 깨진 전제

> "매 턴 3단계 UX"는 **전제 ③-1**(맥락 파악에 명시적 단계 필수)과 **전제 ⑤-1**(모든 조작에 동일 규칙 필수)에서 비롯된 착시.

### 제약 갱신

| 기존 제약 (정체) | 갱신된 제약 |
|---|---|
| 매 턴 3단계 UX를 유지해야 한다 | **조작의 단계 수는 가역성에 비례한다** -- 가역적 조작은 즉시 실행, 비가역적 조작은 명시적 선택 요구 |

### 목표/해결책 반영

이 갱신된 제약은 이미 칸반 PRD의 인터페이스 설계와 일치한다:
- `Alt+Arrow` (이동): 즉시 실행 + 가역적 (`Alt+반대`, `Mod+Z`)
- `Delete`: 선택 후 실행
- `F2`/`Enter` (rename): 모드 진입 -> 편집 -> 확정/취소

"3단계 UX"는 진짜 제약이 아니었다. 진짜 제약은 **"가역성이 보장되지 않는 조작에는 명시적 확인 단계가 필요하다"**이다.
