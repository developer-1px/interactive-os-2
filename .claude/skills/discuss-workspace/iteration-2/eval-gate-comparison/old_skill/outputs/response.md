# 전환 판정: CMS ContextMenu(우클릭 메뉴) 추가

## 이해도 테이블

| # | 요소 | 내용 | 이해도 |
|---|------|------|--------|
| 1 | 목적 | CMS 캔버스에서 우클릭으로 컨텍스트 메뉴 제공 — 마우스 사용자의 기능 접근성 확보 | 🟢 95% |
| 2 | 배경 | 현재 키보드 단축키만 존재, 마우스 우클릭 시 브라우저 기본 메뉴가 뜸 | 🟢 95% |
| 3 | 이상적 결과 | 우클릭 시 노드 맥락에 맞는 메뉴(복사/붙여넣기/삭제/이름변경) 표시 | 🟢 90% |
| 4 | 현실 | MenuList UI 완성품 있음, popover API 사용 가능, 우클릭 이벤트 미연결 | 🟢 92% |
| 5 | 문제 | 마우스 사용자에게 기능 접근성 부족 — 키보드 사용자만 단축키로 접근 가능 | 🟢 90% |
| 6 | 원인 | 우클릭 이벤트 → engine 연결 경로가 없음 | 🟢 90% |
| 7 | 제약 | engine을 우회하면 안 됨, 키보드 메뉴(Shift+F10)도 동일 동작 필수 | 🟢 92% |
| 8 | 목표 | contextmenu 이벤트 → MenuList 팝업 연결, Shift+F10도 동일 트리거 | 🟢 92% |
| 9 | 해결 | ContextMenu UI 완성품 = MenuList + popover + 우클릭/Shift+F10 트리거 | 🟢 90% |
| 10 | 부작용 | popover 겹침 가능성, 기존 브라우저 메뉴 차단 | 🟡 70% |

## 전환 제안

핵심 요소 전부 🟢, 부작용 🟡 이상

구체성 체크:
1. **입력**: 사용자가 CMS 캔버스의 노드 위에서 우클릭(contextmenu 이벤트) 또는 Shift+F10을 누른다. 트리거 대상은 현재 포커스된 노드 또는 마우스 포인터 아래의 노드.
2. **출력**: 해당 노드 맥락의 MenuList가 popover로 표시된다. 메뉴 항목(복사/붙여넣기/삭제/이름변경)은 기존 키보드 단축키와 동일한 engine 커맨드를 실행한다. 브라우저 기본 컨텍스트 메뉴는 preventDefault로 차단된다.
3. **구조**: ContextMenu UI 완성품이 MenuList를 감싸고, popover API로 위치를 잡는다. 트리거는 contextmenu 이벤트 + Shift+F10 keyMap 두 경로. 메뉴 항목은 engine의 기존 commands(copy, paste, delete, rename)를 호출한다. engine 우회 없이 동일한 커맨드 경로를 탄다.
4. **검증**: 통합 테스트로 검증 — (1) 노드 우클릭 시 메뉴 표시 + role="menu" ARIA 상태 확인, (2) Shift+F10으로 동일 메뉴 표시, (3) 메뉴 항목 클릭 시 engine 커맨드 실행 결과(DOM 변화) 확인, (4) 메뉴 밖 클릭 시 닫힘, (5) 다른 popover가 열린 상태에서의 겹침 처리.

제 판단: `/prd`로 전환. 이유: UI 완성품(ContextMenu)의 인터페이스 경계, popover 겹침 처리 규칙, engine 커맨드 연결 방식 등 구현 디테일을 명세해야 한다. 기존 MenuList와 popover API가 있어 구조는 명확하지만, 노드 맥락별 메뉴 항목 결정 로직과 위치 계산 규칙이 PRD에서 확정되어야 구현이 수렴한다.
