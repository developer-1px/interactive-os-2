좋은 질문이네요. 선행 조사부터 하겠습니다.

현재 프로젝트에 이미 `dnd` 플러그인이 있고(`src/interactive-os/plugins/dnd.ts`), **키보드 기반 리오더**(moveUp/moveDown/moveIn/moveOut/moveTo) 5개 커맨드가 Command 패턴으로 구현되어 있습니다. 모두 `moveNode` 유틸리티를 통해 normalized store를 조작하고, undo가 가능합니다. 마우스 드래그앤드롭은 아직 구현되어 있지 않습니다.

## 응답

**제 판단: 같은 모델을 공유해야 합니다.** 이유는 세 가지입니다.

**1. 이미 정답이 코드에 있다.** 현재 `dnd.ts`의 `moveTo(nodeId, targetParentId, index)` 커맨드가 "임의 위치로 이동"을 추상화하고 있습니다. 마우스 DnD의 drop 결과도 결국 "이 노드를 저 부모의 n번째로 옮겨라"이므로, drop 시점에 `moveTo` 커맨드를 dispatch하면 됩니다. 마우스 DnD가 필요로 하는 별도 모델은 **drag 중의 시각적 피드백**(ghost, drop indicator)뿐이고, 이건 view 레이어의 일시적 상태이지 store 모델이 아닙니다.

**2. undo/redo 통합이 자동으로 따라온다.** Command 패턴을 재사용하면 history 플러그인이 마우스 DnD도 키보드 리오더와 동일하게 undo/redo합니다. 별도 모델로 가면 history 통합을 이중으로 구현해야 합니다.

**3. 가역적 동선 원칙과 일치한다.** 프로젝트 원칙상 이동 조작은 역방향으로 원래 위치를 복구할 수 있어야 합니다. Command의 `undo`가 이를 보장합니다.

**구조적으로 분리해야 하는 것은 "모델"이 아니라 "인터랙션 단계"입니다:**

- 키보드: 키 입력 → 즉시 Command dispatch (단일 단계)
- 마우스: drag start → drag over(시각 피드백) → drop → Command dispatch (다단계)

drag 중의 중간 상태(어떤 노드를 들고 있고, 현재 어디 위에 있는지)는 **view-local state**로 처리하고, 최종 drop 시점에만 `moveTo` 커맨드로 store를 변경하는 게 깔끔합니다.

## 갭 질문

가장 이해도가 낮은 **제약** 요소를 채우기 위해 질문합니다.

마우스 DnD의 범위가 어디까지인가요?

A) **같은 리스트 안에서 순서만 바꾸기** — 가장 단순. sortable 수준
B) **같은 트리 안에서 부모 변경 포함** — moveIn/moveOut에 해당하는 크로스 레벨 이동
C) **서로 다른 컨테이너(다른 useAria 인스턴스) 간 이동** — 칸반 컬럼 간 카드 이동 같은 케이스

**제 판단: B.** 이유: 현재 dnd 플러그인이 이미 moveIn/moveOut/moveTo로 트리 내 이동을 모두 지원하고 있어서, 마우스 DnD도 같은 범위를 커버하는 게 자연스럽습니다. C는 multi-zone 설계가 필요해서 별도 논의 대상이 될 것 같습니다.
