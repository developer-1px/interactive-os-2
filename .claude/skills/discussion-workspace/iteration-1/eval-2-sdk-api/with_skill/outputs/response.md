프로젝트 메모리를 먼저 살펴봤다. 이 주제와 직접 관련된 기존 결정들이 있어서 정리부터 한다.

**선행 조사 결과:**
- interactive-os v1이 "범용 hook 하나"로 갔다가 **과도한 추상화로 실패**한 전례가 있다
- v2 전략은 "용도별 완성품 다수" — behavior 기준 분류 금지, 표준 UI 어휘 기준
- 현재 `useAria`는 이미 존재하고, 거기서 engine을 분리한 `useAriaZone`도 있다
- UI SDK 3원칙: (1) 같은 behavior라도 용도 다르면 별개 완성품 (2) 범용 SDK가 아닌 완성품 다수 (3) 분류 기준 = 실제 앱 표준 UI 어휘
- LLM/사용자는 Aria primitives를 직접 쓰지 않고, UI 컴포넌트(`<TreeView>`, `<DataGrid>`)만 쓴다는 원칙 확정

---

"useAria 하나면 다 된다"는 말씀에서 두 가지를 동시에 읽었다. (1) 사용자에게 복잡성을 숨기고 싶다는 의도, (2) 아직 그 "하나"가 구체적으로 어떤 시그니처인지는 모른다는 상태.

의도는 강하게 공감한다. 트리 구조를 다루려면 store 셋업, behavior 선택, 플러그인 조합, 포커스 관리까지 엮어야 하는데, 이걸 사용자에게 노출하면 진입 장벽이 심각하다.

다만 하나 짚어야 할 게 있다. v1에서 정확히 이 방향 — "useAria 하나로 모든 트리 구조를 다룬다" — 을 시도해서 실패한 이력이 있다. 범용적일수록 좋다는 가정이 "아무 용도에도 맞지 않는 추상화"로 귀결됐다. v2에서 "용도별 완성품 다수"로 전략을 바꾼 이유가 이것이다.

그래서 "hook 하나"의 의미를 좀 더 좁힐 필요가 있다. 두 가지 해석이 가능하다:

- **A) 진짜 단일 hook** — `useAria({ data, behavior: 'treegrid', plugins: [...] })` 하나로 모든 패턴을 커버. v1 방향. 장점: API 하나. 단점: 옵션 폭발, 패턴별 타입 안전성 약화, "아무 용도에도 안 맞는" 위험 반복.
- **B) 용도별 hook이지만 사용감이 "하나처럼"** — `useTreeView(data)`, `useDataGrid(data)` 등 용도별 hook이 각각 존재하되, 공통된 규칙(같은 data 형태, 같은 반환 구조)으로 "배우면 다 아는" 일관성을 제공. v2 방향. 장점: 용도에 맞는 디폴트, 타입 안전성, v1 실패 회피. 단점: hook이 여러 개.

**제 판단: B.** 이유: v1 실패 경험 + 확정된 UI SDK 3원칙과 정합. 사용자가 원하는 건 "hook 개수가 1개"가 아니라 "학습 비용이 0에 가까운 것"이고, 그건 일관된 패턴으로 달성할 수 있다.
