좋은 주제입니다. 플러그인 키맵 오버라이드의 안전성 문제 — 이미 프로젝트에서 실제로 겪은 문제와 맞닿아 있어서 기존 경험을 토대로 정리해보겠습니다.

먼저 말씀하신 내용을 구조화하면:

**키맵 오버라이드가 기존 동작을 깨뜨리는 문제의 본질은 "우선순위 해결 전략의 부재"입니다.** 하드코딩된 keyMap이 하나의 덩어리로 존재하면, 플러그인이 부분적으로 교체할 방법이 없어서 전체를 교체하게 되고, 그 과정에서 의도치 않은 키가 누락됩니다.

이 프로젝트에서 이미 확립된 해결 구조가 있습니다: **축(axis) 분해 + chain of responsibility**. 핵심 아이디어는:

1. **keyMap을 원자 축(axis)으로 분해** — navigation, selection, activation 등 각 행동을 독립 partial keyMap으로 분리
2. **pattern = 축의 우선순위 스택** — `composePattern(meta, ...axes)`로 축을 합성하고, 키 충돌은 스택 상위가 승리
3. **plugin keyMap은 최상위 레이어** — 합성 우선순위: `behavior.keyMap → plugin.keyMap → options.keyMap`. 플러그인이 특정 키만 오버라이드하면 나머지는 아래 레이어가 처리

이 구조에서 플러그인이 `Ctrl+C`를 오버라이드하면 navigation이나 activation 축에는 영향이 없습니다. 각 축은 `(ctx) => Command | void`를 반환하고, void면 다음 축으로 넘어가는 chain of responsibility 패턴입니다.

단, 한 가지 중요한 구분이 있습니다: **키맵이 플러그인 책임인 것 vs 소비자 책임인 것**. clipboard, history처럼 키바인딩이 보편적인 건 plugin이 default keyMap을 소유해야 합니다 (5곳 복붙 → 6번째 누락 버그가 실제 발생했습니다). 반면 dnd, crud처럼 뷰마다 다른 건 options.keyMap으로 유지합니다.

**제 판단: 이 SDK는 이 프로젝트(interactive-os)의 내부 SDK이고, 트리뷰/그리드/리스트가 각각 다른 pattern을 사용하는 구조일 것.** 이유: 외부 공개 SDK라면 하위호환 제약이 전혀 다르고, 3개 컴포넌트가 "같은 keyMap을 쓴다"는 게 전부 동일하다는 건지, 공통 부분만 겹치는 건지에 따라 축 분해 전략이 달라집니다.
