카테고리 + 태그 공존 시스템이군요. 첫 턴이니 말씀하신 내용에서 요소를 뽑아보겠습니다.

---

| 요소 | 내용 | 이해도 |
|------|------|--------|
| 목적 | 사용자들이 계층 구조로 콘텐츠를 조직하길 원함. 태그의 flat 구조만으로는 부족 | 🟡 60% |
| 배경 | 현재 태그 시스템만 운영 중. 사용자 요청으로 계층 구조 필요성 대두 | 🟡 55% |
| 이상적 결과 | 카테고리(계층)와 태그(flat)가 공존하며, 검색이 두 시스템 모두를 커버 | 🟡 50% |
| 현실 | 태그 시스템만 존재. 계층 구조 없음 | 🟡 50% |
| 문제 | — | 🔴 |

---

**Steel-manning으로 정리하면:** 사용자 요청의 핵심은 단순히 "폴더를 달라"가 아니라, **분류 체계의 깊이가 부족하다**는 거죠. 태그는 다대다(N:N) 관계라 교차 탐색에 강하지만, "이 항목이 어디에 속하는지"라는 소속감(belongingness)은 표현 못 합니다. 카테고리는 그 반대고요 -- 트리 구조로 소속은 명확하지만 교차 분류가 약합니다. 두 개가 공존해야 한다는 건, 결국 **소속(where) + 성격(what) 두 축의 분류**가 필요하다는 뜻입니다.

여기서 몇 가지 갭이 보입니다:

1. **"계층 구조"의 깊이**: 카테고리가 1단(대분류)인지, N단(대>중>소) 재귀 트리인지에 따라 설계가 완전히 달라집니다. 단일 깊이면 enum에 가깝고, 다단이면 트리 + 경로(breadcrumb) 탐색이 필요합니다.

2. **카테고리와 태그의 관계**: 한 항목이 카테고리 1개 + 태그 N개인지(1:N + N:N), 카테고리도 여러 개 달 수 있는지(N:N + N:N)에 따라 데이터 모델이 다릅니다.

3. **검색 통합의 의미**: "두 시스템 모두 지원"이 (A) 카테고리 필터 + 태그 필터를 각각 제공하는 건지, (B) 하나의 검색창에서 두 시스템을 동시에 쿼리하는 건지.

---

**제 판단: 카테고리는 단일 소속(1:N), 태그는 다중 라벨(N:N)로 분리하는 게 가장 자연스럽습니다.** 이유: 카테고리에 복수 소속을 허용하면 태그와 역할이 겹치고, 사용자가 "이건 카테고리에 넣고 저건 태그로 달고" 판단하는 인지 부하가 커집니다. Gmail의 라벨(태그) vs 폴더(카테고리) 공존이 좋은 선례인데, Gmail도 결국 라벨에 수렴했고 폴더는 1개 소속입니다.

A) 카테고리 = 단일 소속 트리, 태그 = 다중 라벨 -- 역할 명확, 검색 UX 단순
B) 카테고리도 다중 소속 허용 -- 유연하지만 태그와 경계 모호
C) 다른 구조를 생각 중이시면

**갭 질문:**

1. 사용자들이 원하는 "계층"은 몇 단계 깊이인가요? 예를 들어 "기술 > 프론트엔드 > React" 같은 3단 이상인지, "기술 / 디자인 / 운영" 같은 1단 분류인지?

2. 지금 태그 시스템의 규모가 어느 정도인가요? (태그 수, 항목당 평균 태그 수) -- 기존 태그 중 일부를 카테고리로 승격시키는 마이그레이션 경로가 필요한지 판단하려고 묻습니다.
