카테고리 + 태그 공존, 좋은 주제입니다. 정리해보겠습니다.

## 이해도 테이블

| 요소 | 내용 | 이해도 |
|------|------|--------|
| 목적 | 사용자들이 콘텐츠를 계층적으로 분류하길 원함. 태그의 flat 구조로는 탐색이 어려워진 상황을 해소 | 🟡 70% |
| 배경 | 현재 태그 시스템이 존재하고 운영 중. 사용자 피드백으로 계층 구조 요구가 들어옴 | 🟡 60% |
| 이상적 결과 | 카테고리(계층)와 태그(flat)가 공존하며, 검색이 두 시스템 모두를 커버 | 🔴 40% |

## 응답

말씀하신 구조를 steel-manning 해보면 이렇습니다:

**태그 = 횡단 관심사(cross-cutting), 카테고리 = 종단 분류(hierarchical)**. 태그는 "긴급", "리뷰필요" 같은 성격이고, 카테고리는 "개발 > 프론트엔드 > React" 같은 위치입니다. 이 둘은 서로 다른 축이라 공존이 자연스럽습니다 -- 실제로 WordPress, Notion, GitHub Issues 등 대부분의 콘텐츠 시스템이 이 이중 분류를 채택하고 있습니다.

다만 "공존"이라고 해도 구현 복잡도는 꽤 다릅니다:

- **A) 카테고리는 단일 선택(배타적 소속)** -- 한 아이템은 하나의 카테고리에만 속함. Gmail 라벨 vs 폴더의 "폴더" 쪽. 구조가 단순하고 검색도 직관적.
- **B) 카테고리도 다중 선택(다중 소속)** -- 한 아이템이 여러 카테고리에 속할 수 있음. 유연하지만 "이건 태그와 뭐가 다른가?" 경계가 흐려짐.
- **C) 카테고리 단일 + 보조 카테고리 다중** -- 주 카테고리는 하나, 관련 카테고리에 링크 가능. 중간 지점.

**제 판단: A (단일 선택).** 이유: 태그가 이미 다중 소속을 커버하고 있기 때문에, 카테고리까지 다중 선택으로 가면 두 시스템의 역할이 겹칩니다. 카테고리는 "이 아이템의 집"을 정해주는 것이고, 태그는 "이 아이템의 성격"을 표현하는 것으로 역할을 분리하는 게 사용자 멘탈 모델이 깔끔합니다.

## 갭 질문

이상적 결과(3번)가 아직 구체적이지 않습니다. 두 가지를 알아야 합니다:

1. **"검색이 두 시스템 모두 지원"의 구체적인 모습은 뭔가요?** 예를 들어:
   - 카테고리 트리에서 선택하면 그 하위까지 포함해서 필터링?
   - 검색창에서 "카테고리:프론트엔드 태그:긴급" 같은 복합 쿼리?
   - 아니면 검색 결과에 카테고리 경로가 표시되는 정도?

2. **카테고리 깊이에 상한이 있나요?** 무제한 깊이는 UI 복잡도와 검색 쿼리 비용을 급격히 올립니다. 보통 2~3 depth면 충분한 경우가 많은데, 사용자들이 원하는 계층이 얼마나 깊은 건지 감이 있으신가요?
