좋은 질문이야. 플러그인이 keyMap을 오버라이드할 때 기존 동작이 깨지는 문제 — 이건 우리 프로젝트에서 이미 한 번 부딪힌 적 있는 문제와 구조적으로 같아. 정리해보자.

## 이해도 테이블

| 요소 | 내용 | 이해도 |
|------|------|--------|
| 목적 | 하드코딩된 keyMap을 플러그인으로 확장 가능하게 만들되, 기존 동작 안전성 보장 | 🟡 70% |

## 응답

핵심 긴장은 **"확장성 vs 안전성"**인데, 이건 사실 해결된 패턴이 있어.

**제 판단: Chain of Responsibility 기반 축(axis) 합성이 가장 적합하다.** 이유는 세 가지:

### 1. keyMap을 통째로 오버라이드하면 깨지는 건 당연하다

"오버라이드"라는 단어가 이미 위험 신호야. 플러그인이 `ArrowDown` 동작을 통째로 교체하면, 그 키에 걸려있던 navigation, selection 같은 기본 동작이 전부 날아가거든. 문제는 "오버라이드를 어떻게 안전하게 할까"가 아니라 **"오버라이드가 아닌 합성 구조가 필요하다"**는 거야.

### 2. 레이어 분리: axis -> pattern -> plugin keyMap

이미 검증된 3계층 모델이 있어:

1. **축(axis)** = 원자적 ARIA 행동. `navigate`, `select`, `activate` 같은 순수 단위
2. **패턴(pattern)** = 축의 우선순위 스택 조합. `composePattern(meta, ...axes)` → TreeView면 nav+depth+select를 조합
3. **플러그인 keyMap** = 컴포넌트 레벨 오버라이드. clipboard, history 같은 CRUD 계열

합성 우선순위: `behavior.keyMap(축) → plugin.keyMap → options.keyMap(사용자)`

핵심은 **chain of responsibility** — 각 레이어의 핸들러가 `Command | void`를 반환하고, 위에서부터 순회해서 첫 번째 non-void가 승리해. 플러그인이 특정 키를 처리하지 않으면(void 반환) 자연스럽게 아래 레이어(기본 behavior)로 넘어가.

### 3. 플러그인은 keyMap까지 소유해야 한다

commands만 제공하고 "keyMap은 알아서 연결해라"는 구조는 복붙 버그를 만들어. clipboard keyMap 3줄이 5곳에 복붙되다가 6번째에서 누락되는 식으로. 플러그인을 등록했으면 키바인딩이 자동으로 따라와야 해. 단, dnd/crud/rename처럼 뷰마다 다른 건 `options.keyMap`으로 남기고.

---

하지만 몇 가지 아직 불명확한 게 있어:

**"이상적 결과"가 구체적이지 않아서 방향이 갈릴 수 있어.** 지금 말한 3개 컴포넌트(트리뷰, 그리드, 리스트) — 이걸 하나의 엔진 위에 올리는 공통 SDK인지, 아니면 각 컴포넌트가 독립적이고 플러그인 인터페이스만 공유하는 건지에 따라 구조가 달라져.

A) **공통 엔진 + 패턴 분기** — 하나의 behavior engine이 있고, TreeView/Grid/List는 다른 패턴(축 조합)일 뿐. 플러그인은 엔진 레벨에 한 번만 등록.
B) **독립 컴포넌트 + 공유 인터페이스** — 각각 독립 구현이지만, 플러그인 인터페이스(keyMap 합성 프로토콜)가 동일.
C) 다른 구조를 생각하고 있다면.

**제 판단: A.** 이유: 3개 컴포넌트가 이미 같은 keyMap을 공유하고 있다는 건, 본질적으로 같은 엔진 위의 다른 프레젠테이션이라는 뜻이야. 분리하면 키바인딩 동기화 문제가 3배로 늘어나.

트리뷰/그리드/리스트가 지금 어떤 수준으로 keyMap을 공유하고 있는지 — 완전 동일한 객체인지, 아니면 비슷하지만 컴포넌트마다 변형이 있는지?
