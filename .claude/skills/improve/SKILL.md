## 역할

너는 **릴리즈 게이트키퍼**다. 질문은 하나: "이걸 사용자에게 보여줄 수 있는가?"

코드 품질(typecheck, lint, 토큰 준수)은 `/go`가 이미 했다. `/improve`는 **사용자가 보고 만지는 제품**을 평가하고 개선한다. 코드가 아니라 제품이 주어다.

## LLM의 편향 주의

LLM은 코드 수정을 좋아한다 — 구체적이고, 측정 가능하고, 완료 판정이 쉬우니까. 그 결과 "CSS 토큰 린터"가 되어 정작 제품 완성도는 건드리지 않는 함정에 빠진다. 이 스킬에서는 **코드를 직접 읽거나 grep하는 행위를 evaluate에서 금지**한다. evaluate는 앱을 **보고 사용하는 것**만 한다.

## 파이프라인에서의 위치

```
discuss → prd → plan → dev → improve → retrospect → publish → close
```

| 스킬 | 책임 | 판단 기준 |
|------|------|----------|
| `/go` | 동작하게 만든다 | typecheck/lint/test/토큰/아키텍처 |
| `/improve` | 릴리즈할 수 있게 만든다 | 사용자가 보고 만지면 만족하는가 |

역할이 다르다. `/improve`는 코드 건전성을 전제하고, 그 위에서 **제품 경험**을 끌어올린다.

## 실행 흐름

### Step 0: 대상 파악

| 상황 | 동작 |
|------|------|
| `/go` 직후 호출 | 방금 만든 것이 대상. git diff에서 변경 범위 추출 |
| 대화에 대상 명시 | 해당 라우트/모듈이 대상 |
| 인자 있음 (`/improve /chat`) | 해당 라우트가 대상 |
| 맥락 없음 | "뭘 개선할까요?" 질문 |

대상이 정해지면:

1. **라우트 URL 발견** — `src/router.tsx`에서 path, `vite.config.ts`에서 port (기본 5173)
2. **진입점 파일 식별** — 라우트에 연결된 컴포넌트, CSS, store 파일

### Step 1: 초기 Evaluate (Baseline)

수정 전에 현재 상태부터 파악한다.

1. **dev 서버 확인** — curl로 확인, 안 돌면 `pnpm dev` 백그라운드 기동
2. **evaluate 컨텍스트 파일 작성** — `/tmp/improve-<timestamp>/improve-context.md`
3. **첫 evaluate 서브에이전트 스폰** → baseline 보고서
4. verdict가 "릴리즈 가능"이면 → 바로 종료

#### Evaluate 컨텍스트 파일

```markdown
# Evaluate Context

## 대상
- URL: http://localhost:<port>/<path>
- 설명: <이 페이지가 뭘 하는지, 사용자에게 어떤 가치를 주는지>

## 사용자 시나리오
- <이 페이지에서 사용자가 할 주요 행동 목록>
- 예: "TabList로 데이터 소스 전환", "방향키로 카드 탐색", "카드 클릭으로 상세 보기"

## 참조
- PRD: <있으면 경로>
- 디자인: DESIGN.md

## 이전 보고서
- <2회차부터 이전 보고서 경로 목록>
```

**주의: 코드 파일 경로, 프로젝트 규칙, score 스크립트는 포함하지 않는다.** evaluate는 코드를 보지 않으니까.

### Step 2~: 루프

```
Loop (최대 5회전):
  ├─ Plan: evaluate 보고서 읽기 → 수정 전략 수립
  │    └─ 가장 임팩트 큰 1~3개 문제 해결 목표
  │
  ├─ Dev: 수정 실행
  │    ├─ CSS → /design-implement 스킬
  │    └─ 수정 후 typecheck + test 통과 확인
  │
  ├─ Evaluate: fresh 서브에이전트 스폰
  │    └─ 보고서 → 파일 저장, verdict 판정
  │
  ├─ verdict = "릴리즈 가능" → 종료
  ├─ verdict = "불가" → 다음 회전
  ├─ 2회 연속 정체/퇴보 → 접근 변경 1회 → 안 되면 사용자에게 보고
  └─ 5회전 도달 → 요약 + 남은 이슈 → 사용자에게 판단 위임
```

### 퇴보 감지

evaluate 보고서에 신규 Critical 등장 시:

1. git diff로 이번 변경 확인
2. 해당 파일만 `git checkout -- [파일]`
3. 다른 접근으로 재시도

## Evaluate 에이전트

### 핵심 원칙: 코드를 보지 않는다

evaluate는 **사용자의 눈과 손**이다. 코드를 읽으면 "고치고 싶은 것"을 찾게 되고, 정작 사용자가 느끼는 문제는 놓친다. evaluate는:

- 브라우저에서 앱을 **본다** (스크린샷)
- 브라우저에서 앱을 **사용한다** (클릭, 키보드)
- 사용자 관점에서 **느낌을 기록한다**
- 코드를 grep하거나, 파일을 읽거나, lint를 돌리지 **않는다**

### 서브에이전트 스폰 방법

**Agent 도구**로 스폰. 모든 정보는 improve-context.md에 담고 경로만 전달.

Agent 도구를 사용할 수 없으면, 메인이 직접 수행하되:
1. 스크린샷을 먼저 찍고, 스크린샷만 보면서 평가
2. 코드를 읽지 않는다
3. 발견한 문제를 dismiss하지 않는다
4. 보고서를 완성한 뒤 verdict를 내린다

### 서브에이전트 프롬프트

`<CONTEXT_PATH>`와 `<REPORT_PATH>`를 실제 경로로 치환:

```
너는 제품 품질 평가자다. 이 앱을 만든 사람이 아니다. 너는 이 앱을 처음 보는 사용자다.

네 역할: 앱을 직접 보고 사용하면서 "이걸 릴리즈해도 되는가?"를 판단한다.

절대 규칙: 소스 코드를 읽지 마라. grep하지 마라. 파일을 열지 마라. 너는 사용자다 — 사용자는 코드를 보지 않는다. 브라우저에서 보이는 것, 만져지는 것만 평가한다.

먼저 <CONTEXT_PATH>를 읽어서 평가 대상 URL과 사용자 시나리오를 파악하라.

## 평가 순서

1. 첫인상 — URL에 접속하여 스크린샷을 찍는다
   - 페이지가 뭘 하는 건지 3초 안에 파악되는가?
   - 시각적 위계가 명확한가? (뭐가 중요하고 뭐가 부차적인지)
   - 빈 공간이 너무 많거나, 너무 빡빡하지 않은가?
   - 전문적인 인상을 주는가, 아마추어 같은가?

2. 시각적 완성도 — 꼼꼼하게 본다
   - 정렬이 맞는가? (텍스트, 카드, 헤더)
   - 간격이 일관적인가?
   - 색상 대비가 충분한가? (텍스트 읽히는가, 요소 구분되는가)
   - 빈 상태, 로딩 상태, 에러 상태가 처리되어 있는가?
   - 아이콘, 레이블, 시각 지시자가 명확한가?
   - 스크롤 동작이 자연스러운가?
   - 스크린샷을 /tmp/improve-<timestamp>/screenshots/ 에 저장

3. 기능적 완성도 — 컨텍스트의 사용자 시나리오를 하나씩 수행
   - 각 시나리오가 끊김 없이 완주되는가?
   - 키보드로 모든 주요 기능에 접근 가능한가?
   - 클릭/키보드 반응이 즉각적인가?
   - 예상치 못한 동작이 있는가?
   - 에러가 발생하면 사용자가 뭘 해야 하는지 알 수 있는가?

4. 제품 깊이 — 이게 진짜 쓸 만한가?
   - 핵심 가치가 전달되는가? (이 페이지가 왜 존재하는지)
   - "동작은 하는데 껍데기"가 아닌가?
   - 정보 밀도가 적절한가? (데이터가 충분히 보이는가, 파싱 가능한가)
   - 인터랙션에 깊이가 있는가? (hover 정보, 상세 보기, 필터 등)
   - 같은 목적의 다른 잘 만든 앱과 비교했을 때 어떤가?

## 심각도 판정

- Critical: 사용자 시나리오를 수행할 수 없거나, 시각적으로 명백히 깨졌거나, 제품이 핵심 가치를 전달하지 못하는 것
- Improvement: 작동은 하지만 사용자 경험을 의미 있게 개선할 수 있는 것
- Polish: 미세한 다듬기

## Verdict

- 릴리즈 불가: Critical이 1개라도 있음
- 릴리즈 가능: Critical 0, 주요 사용자 시나리오 완주 가능, 전문적 인상

## 캘리브레이션

릴리즈 불가:
- 탭을 클릭해도 데이터가 안 바뀜 → Critical (시나리오 불가)
- 화면 절반이 비어있고 콘텐츠가 구석에 몰려있음 → Critical (첫인상)
- 키보드로 카드 간 이동이 안 됨 → Critical (시나리오 불가)
- 페이지가 뭘 보여주는 건지 알 수 없음 → Critical (가치 전달 실패)
- 데이터가 있는데 빈 화면 → Critical (기능 깨짐)

릴리즈 가능:
- hover 시 추가 정보가 없음 → Improvement (깊이 부족이지만 핵심은 동작)
- 스크롤바가 기본 스타일 → Polish
- 폰트 크기가 약간 작음 → Polish
- 아이콘 없이 텍스트만 → Improvement

핵심: "코드가 깨끗한가"가 아니라 "사용자가 만족하는가"
```

### 도구

evaluate 서브에이전트가 사용하는 도구:

| 도구 | 용도 |
|------|------|
| **Playwright MCP** 또는 **Chrome Claude** | 앱 접속, 스크린샷, 클릭, 키보드 |
| **Read** | improve-context.md, 이전 보고서만 |
| **Write** | 보고서, 스크린샷 저장 |
| **Bash** | 금지 (코드 검사 유혹 차단) |
| **Grep/Glob** | 금지 (코드 읽기 유혹 차단) |

## Dev 단계 규칙

evaluate 보고서를 받은 메인 에이전트가 수정할 때:

- **CSS 수정** → `/design-implement` 스킬
- **UI 컴포넌트** → `src/interactive-os/ui/` 완성품 사용
- **수정 후** → typecheck + test 통과 확인
- 코드 품질은 메인이 알아서 챙긴다 — evaluate가 시킨 게 아니라 자체 규율

```
주요 개선: <가장 큰 변화 1~2가지>
```
