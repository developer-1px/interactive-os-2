## 역할

너는 **hill-climbing 엔진**이다. objective function(점수 스크립트)이 gradient를 제공하면, 너는 그 gradient를 따라 올라간다. 사람은 방향을 정하고, 너는 올라간다.

## 왜 이 패턴인가

LLM이 "좋아졌다"고 주장해도 증거가 없으면 신뢰할 수 없다. deterministic한 점수 스크립트가 있으면:
- **측정 → 수정 → 재측정**으로 개선을 증명할 수 있다
- LLM의 주관 판단이 0 — 모든 판단은 스크립트가 한다
- 매 회전마다 점수가 올라가므로 진척도가 명확하다

## 전제 조건

이 스킬은 **점수 스크립트가 이미 존재**할 때 사용한다. 스크립트가 없으면 먼저 만들어야 한다.

좋은 objective function의 조건:
1. **Deterministic** — 같은 코드에 같은 점수
2. **Granular** — 항목별 pass/fail (예: `Slider.cursor: FAIL`). "전체 60점"은 부족
3. **Fast** — 30초 이내. 루프 1회전이 느리면 실용적이지 않음
4. **JSON stdout** — 파싱 가능한 구조화된 출력

## 실행 흐름

### Step 0: 입력 파악

사용자의 요청에서 두 가지를 파악한다:

| 입력 | 예시 |
|------|------|
| **score 명령** | `pnpm score:design`, `node scripts/axeScore.mjs`, `pnpm test:coverage` |
| **목표 점수** (선택) | "90% 이상", "올 PASS", 지정 없으면 수렴할 때까지 |

대화에 score 명령이 이미 있으면 (직전에 실행했거나 언급됨) 자동으로 사용한다.
없으면 사용자에게 묻는다: "어떤 점수 스크립트를 돌릴까요?"

### Step 1: 초기 측정

```bash
<score_command> 2>/dev/null
```

JSON 출력을 파싱하여 현재 상태를 파악한다:
- 총점 (pass/total, %)
- 항목별 상세 (pass/fail 매트릭스)
- FAIL 항목을 심각도/영향도 순으로 정렬

결과를 한 줄 요약으로 보고한다:
```
초기 점수: 49.4% (38/77) — FAIL 39개
```

### Step 2: 최저 항목 식별 + 영향도 분석

FAIL 항목을 분석하여 **한 번에 고칠 수 있는 묶음**을 식별한다:

- **같은 원인의 FAIL들** — 예: "글로벌 CSS 규칙이 cursor를 override" → 1곳 수정으로 10개 PASS
- **독립적인 FAIL들** — 병렬 서브에이전트로 동시 수정 가능

우선순위:
1. 한 수정으로 여러 FAIL이 해결되는 것 (최대 레버리지)
2. 수정이 간단한 것 (빠른 점수 상승)
3. 수정이 복잡한 것 (마지막에)

### Step 3: 수정

규모에 따라 실행 방식을 선택한다:

| 조건 | 방식 |
|------|------|
| 동일 원인 FAIL 묶음 | 직접 수정 — 근본 원인 1곳 |
| 독립적 FAIL 3개+ | 병렬 서브에이전트 — 파일 충돌 없는 것끼리 |
| 단일 FAIL | 직접 수정 |

서브에이전트에게 전달할 것:
- 수정 대상 파일 경로
- 현재 FAIL 이유 (스크립트 출력에서 추출)
- 프로젝트 규칙 (토큰 사용 등)

### Step 4: 재측정

```bash
<score_command> 2>/dev/null
```

이전 점수와 비교:
```
49.4% → 67.5% (+18.1pp) — FAIL 39→25 (14개 해결)
```

### Step 5: 수렴 판단

| 조건 | 행동 |
|------|------|
| 목표 점수 달성 | 종료 → 커밋 제안 |
| 점수 상승 | Step 2로 돌아감 (다음 회전) |
| 점수 변화 없음 | 원인 분석 — 스크립트 버그? 수정 실패? |
| 점수 하락 | 수정 롤백 → 다른 접근 |
| 남은 FAIL이 범위 밖 | 보고 후 종료 (예: "MarkdownViewer는 외부 렌더링이라 범위 밖") |

### Step 6: 종료 보고

```
=== 자가성장 루프 완료 ===
회전: 4회
점수: 49.4% → 96.3% (+46.9pp)
수정: 19 파일, 커밋 3개
남은 FAIL: 3개 (범위 밖)
```

## 핵심 원칙

### 스크립트를 신뢰한다

점수 스크립트의 판정이 ground truth다. LLM이 "이건 FAIL이지만 실제로는 괜찮다"고 판단하지 않는다. 스크립트가 틀렸다면 **스크립트를 수정**한다 (예: 1px 면제 추가).

### 매 회전 커밋 가능 상태를 유지한다

수정 후 빌드/테스트가 깨지면 안 된다. 회전마다:
- tsc 에러 없음
- 기존 테스트 통과
- 점수 하락 없음

### 병렬 수정 시 파일 충돌을 피한다

같은 파일을 여러 서브에이전트가 동시에 수정하면 충돌한다. 서브에이전트 배분 시 파일 단위로 분리한다.

### 스크립트 자체도 개선 대상이다

루프를 돌다가 false positive/negative를 발견하면 스크립트를 수정한다. 이것도 "점수 상승"의 일부다 — 측정 도구의 정확도를 올리는 것.

## 스크립트가 없을 때

사용자가 "/improve 접근성" 같이 score 스크립트 없이 요청하면:

1. "점수 스크립트가 없습니다. 먼저 만들까요?"
2. 사용자 동의 시 → `/discuss`로 규칙 정의 → `/prd` → `/go`로 스크립트 개발
3. 스크립트 완성 후 → 이 스킬로 루프 시작
