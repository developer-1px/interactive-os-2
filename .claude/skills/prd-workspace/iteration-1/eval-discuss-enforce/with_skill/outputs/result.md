# Eval: /prd discuss 강제 여부 분석

## 평가 질문

/prd를 /discuss 결과 없이 호출했을 때, 스킬이 discuss를 강제하는가?

## 분석

### Step 0 텍스트 (line 26-31)

```
## Step 0: 입력 확인 + FRT Pre-fill

1. Discussion 결과(10요소 이해도 테이블)가 대화에 있는지 확인
2. **없으면 /discuss를 강제한다**: "Discussion 결과가 필요합니다. /discuss를 먼저 해주세요." — PRD는 discuss 없이 시작할 수 없다
```

### 판정: 명시적으로 강제한다

Step 0의 2번 항목이 세 가지 수준에서 강제를 명시한다:

1. **행동 지시**: "없으면 /discuss를 강제한다" -- 조건부가 아니라 필수 행동으로 기술
2. **대사 템플릿**: "Discussion 결과가 필요합니다. /discuss를 먼저 해주세요." -- AI가 출력할 문장까지 제공
3. **불변 규칙 선언**: "PRD는 discuss 없이 시작할 수 없다" -- 예외 없는 제약으로 선언

### 추가 근거

- Step 1의 FRT 변환 규칙(line 33-44)이 discuss 10요소를 WHY 4요소로 매핑하는 구조이므로, discuss 결과가 없으면 FRT 변환 자체가 불가능하다. 입력이 없으면 변환할 것이 없다.
- 빈 표 템플릿의 "① 동기 > WHY (discuss FRT에서 이식)" 헤더(line 128)도 discuss 결과의 존재를 전제한다.
- 파이프라인 다이어그램(line 13-15)이 `Discussion(왜/뭘) -> /prd(어떻게)` 순서를 명시한다.

### 잠재적 약점

강제 메커니즘이 **텍스트 지시**에만 의존한다. 실제 LLM이 이 지시를 100% 따르는지는 별도의 런타임 eval이 필요하다. 하드코딩된 가드(예: discuss 결과 파싱 실패 시 자동 중단)는 존재하지 않으며, LLM의 지시 준수에 전적으로 의존한다. 이는 스킬 시스템의 구조적 한계이지 이 스킬의 결함은 아니다.

## 결론

**YES** -- /prd 스킬은 discuss가 먼저 실행되어야 함을 명시적으로 강제한다. Step 0에서 확인 -> 부재 시 차단 -> 대사 템플릿 제공 -> 불변 규칙 선언의 4중 구조로 설계되어 있다.
