interactive-os 아키텍처 접근 방식

3가지 접근 방식 비교 — 추천: Approach A

A

Command-Driven Plugin Engine (추천)

핵심: Command 패턴이 시스템의 심장. 모든 조작 = Command 객체.

// 레이어 구조
① Normalized Store — entities + relationships
② Command Engine — execute/undo, 미들웨어 파이프라인
③ Plugin System — 미들웨어로 Command 체인에 참여
④ ARIA Behavior — role별 키보드/포커스 매핑
⑤ Compound Components — render slot 기반 UI 바인딩

장점

  • undo/redo가 아키텍처에 내장
  • 플러그인 간 의존성 없이 Command로 소통
  • treegrid에서 검증 후 다른 role로 확장 용이
  • ProseMirror가 증명한 패턴

리스크

  • Command 설계가 초기에 잘 잡혀야 함
  • 복합 Command (batch) 처리 복잡도
B

Signal-Based Reactive Core

핵심: 정규화 데이터를 Signal(fine-grained reactivity)로 감싸서 플러그인이 반응형으로 동작

// 핵심 차이점
Command 대신 Signal mutation을 추적
history()는 signal 변경을 자동 스냅샷
플러그인은 signal을 구독/반응

장점

  • 세밀한 리렌더링 최적화
  • 플러그인 코드가 더 선언적

리스크

  • React 19와 Signal 통합이 아직 불안정
  • undo/redo를 위한 별도 diff 로직 필요
  • 디버깅이 어려움 (반응형 체인 추적)
C

Reducer + Action 패턴 (Redux 스타일)

핵심: 상태 변경이 모두 Action → Reducer를 통과. 플러그인 = 미들웨어 or 커스텀 리듀서.

// 핵심 차이점
Action은 순수 데이터 (serialize 가능)
Reducer가 상태 변환 담당
미들웨어가 Action을 가로챔

장점

  • React 생태계에 익숙한 패턴
  • DevTools, time-travel 디버깅 용이
  • Action이 직렬화 가능 → 협업/로깅

리스크

  • undo를 위해 inverse action 또는 스냅샷 필요
  • Command 패턴 대비 undo/redo가 자연스럽지 않음
  • 보일러플레이트가 많아질 수 있음

왜 A를 추천하는가

이 프레임워크의 핵심 가치는 "키보드로 데이터를 조작"하는 것입니다. 조작의 기본 단위가 Command가 되면:

  • 키 입력 → Command 매핑이 직관적 (Ctrl+Z → history.undo() → command.undo())
  • 플러그인 = Command 생산자 + 미들웨어라는 단일 모델
  • B의 장점(반응성)은 렌더링 레이어에서 React 자체가 해결
  • C의 장점(직렬화)은 Command에 serialize()를 추가하면 확보 가능
